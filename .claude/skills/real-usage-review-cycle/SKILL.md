## 배경

`PRD.md` 세션 인계 노트: "코드만 보고 '됐다'고 하지 말고, 실제 텍스트(뉴스
기사, 블로그 글 등)로 `main.py correct`를 돌려서 검증한다." 이 방식으로 실제
버그를 여러 번 잡았다(`docs/IMPLEMENTATION_LOG.md`의 §19, §20, §21이 모두 이
절차를 반복해 나온 결과물 — 예: kornorms 오조회로 "집"이 "지브"로 잘못
치환된 사고, 조사·어미 결합 재구성 시 음절 중복 출력 사고 등, 코드만 읽어서는
발견하지 못했던 버그들).

## 언제 적용

새 기능을 구현한 뒤 검증할 때, 또는 정기적으로 실사용 감수를 하고 싶을 때.

## 작업 순서

1. **텍스트 준비**: 저작권 있는 실제 원문(뉴스 기사, 블로그 글 등)을
   스크래치패드에만 둔다 — **리포에는 절대 커밋하지 않는다.**
2. **실행**: `python main.py correct <파일>`로 돌려 자동 교정 결과와 플래그
   리포트를 얻는다.
3. **대조 검토**: 원문의 의도와 자동 교정/플래그 결과를 사람이 직접 대조한다.
   각 불일치를 아래 두 갈래로 분류한다:
   - **확실한 버그** (사전/규범 근거와 명백히 다르게 동작함, 또는
     `docs/KNOWN_LIMITATIONS.md`의 알려진 한계에 해당하지 않는 새로운
     오탐/미탐) → 원인을 코드에서 추적해 **즉시
     수정**한다. 임시 패치가 아니라 근본 원인을 고친다 — 문법 규칙 관련이면
     [[grammar-rule-verify-then-code]] 절차를 따른다.
   - **판단이 필요한 애매한 사례** (문맥에 따라 답이 갈림, 사전 근거 자체가
     없음, 고유명사처럼 원칙상 항상 사람 확인이 필요한 영역) → 코드를
     건드리지 않는다. 이미 올바르게 플래그되고 있는지만 확인한다.
4. **회귀 확인**: 수정 후 `examples/sample.srt`로 기존 동작이 깨지지
   않았는지 재확인한다.
5. **`docs/IMPLEMENTATION_LOG.md` 기록**: §19~§21과 같은 형식으로 새 섹션을
   추가한다 — "N차 실사용 감수: 오탐/미탐 M건 조사 및 확실한 버그 K건 수정".
   각 버그마다 발견 경위(어떤 원문 표현에서, 왜 틀렸는지, 원인이 뭐였는지)와
   수정 내용을 구체적으로 적는다. 판단 필요 사례로 남긴 것도 왜 자동화하지
   않았는지 간단히 남긴다.
   - **로그 크기 관리**: 섹션을 추가하기 전에 `docs/IMPLEMENTATION_LOG.md`가
     대략 300줄 또는 30KB를 넘는지 확인한다. 넘으면 가장 오래된 날짜
     섹션들(파일 앞쪽 절반 정도)을 `docs/log-archive/<반기>.md`(예:
     `2026-h2.md`, 없으면 새로 만든다)로 옮기고, `IMPLEMENTATION_LOG.md` 맨
     위에 "이전 이력은 `docs/log-archive/2026-h2.md` 참고"처럼 링크 한 줄만
     남긴 뒤 새 섹션을 추가한다. 이 작업 자체는 파일을 자르는 것뿐이니 별도로
     PRD.md를 갱신할 필요는 없다.
6. **세션 종료 관례**: `PRD.md`와 방금 수정한 `docs/*.md`(주로
   `docs/IMPLEMENTATION_LOG.md`, 필요하면 `docs/KNOWN_LIMITATIONS.md`), 그리고
   미러 파일 `C:\Users\user\Documents\자막_맞춤법교정_PRD.md`(PRD.md 본문만
   미러링하면 됨, docs/ 하위 파일은 저장소 안에 있으므로 미러링 대상 아님)를
   동기화한 뒤, 이 프로젝트는 완료 시 자동 커밋 관례이므로 git commit한다(단,

   — 예: `gananda-precedent-research`는 사람 리뷰 전 커밋 금지).
