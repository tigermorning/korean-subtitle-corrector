## 배경 (왜 이 절차가 필요한가)

`PRD.md` §5 "작업 원칙: 검증표 먼저, 코드는 나중"(2026-07-17 확정)에서 온
교훈이다. `correct_aux_verb_spacing()` 패턴 2("할만하다"류)에서 "규칙이 이럴
것이다"라는 가정으로 코드를 먼저 짜고 예외를 나중에 패치하는 방식으로
진행하다가, 사용자가 실제 우리말샘 용례로 지적한 오류("할만하다"를 "할 만
하다"로 잘못 고침)를 **두 번이나** 겪었다. 이 프로젝트의 핵심 원칙("확률적
추측이 아닌 권위 있는 규범 근거 기반, 애매하면 사람에게 확인")과 정면으로
충돌하는 진행 방식이었다.

## 언제 적용

새로운 맞춤법/띄어쓰기 문법 패턴을 자동 교정 대상으로 추가하려 할 때 —
특히 예외가 있을 수 있는 패턴(의존명사, 보조 용언, 조사·어미 결합 등)일 때
반드시 이 순서를 지킨다. 이미 확정 오류 패턴처럼 예외가 없는 단순 치환은
이 절차가 필요 없다.

## 작업 순서

1. **경우의 수 나열**: 해당 문법 패턴에 들어갈 수 있는 표현을 가능한 만큼
   전부 나열한다(활용형, 준말, 흔한 변형 포함).
2. **정답표 작성** — 각 표현을 아래 순서로 직접 조회해 "입력 → 정답"
   표를 만든다. 코드를 짜기 **전에** 표부터 완성한다.
   - 표준국어대사전 `search.do`/`view.do` (표제어, `pos`, `norm_info`)
   - 우리말샘 (표준국어대사전에 없을 때 보완, 단 `_opendict_item_is_standard()`
     류 판정 — "⇒규범 표기는 ~이다" 안내가 있으면 비표준으로 취급)
   - 한국어 어문 규범 오픈API(`kornorms`) — 외래어/표기 세칙 관련이면 확인
   - 위 셋으로도 애매하면 `subtitle_corrector/gananda_precedents.py`와
     `docs/GRAMMAR_PRECEDENTS_TABLE.md`(온라인가나다 판례 사전 검증표)를
     확인 — 이미 배치 조사로 쌓여있는 판례가 있는지 먼저 본다.
   - 그래도 답이 없거나 조건부(문맥에 따라 다름)면 그 표현은 정답표에
     "자동 교정 대상 아님, 항상 확인 플래그"로 명시한다 — 억지로 단일 정답을
     만들지 않는다.
3. **표에 맞춰 코드 작성**: 정답표에 정확히 일치하는 로직만 짠다. 코드가
   표를 따라가는 것이지, 표가 코드를 사후 검증하는 게 아니다.
4. **새 사례 발견 시**: 구현 후 표에 없던 사례가 나오면 코드를 임시로
   패치하지 않는다. 먼저 그 사례를 정답표에 추가(2번 절차로 재조회)한 뒤,
   갱신된 표에 맞춰 코드를 고친다.
5. **회귀 테스트**: `examples/sample.srt` 및 관련 케이스를
   `tests/test_engine.py`에 추가해 이후 실수로 되돌아가지 않게 고정한다.
6. **기록**: `docs/KNOWN_LIMITATIONS.md`(관련 기존 항목이 있으면 그 옆에) 또는
   `docs/IMPLEMENTATION_LOG.md`(새 번호 섹션으로)에 구현 완료 표시와 함께,
   조회 과정에서 발견한 함정·버그가 있으면 기존 항목들과 같은 스타일로 발견
   경위를 구체적으로 남긴다.
