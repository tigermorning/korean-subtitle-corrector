## 1. 背景與目的

使用既有的韓語自動校正工具「釜山大學拼寫檢查器」時，主要有四點令人不滿之處：**誤判過多（連正常的表達與專有名詞都被誤判為錯誤）；不提供任何依據就一味要求修改；那些修改建議本身毫無一致性、看起來近乎隨機，反而讓那些建議感覺像是新的錯誤；而且一次能檢查的字數有上限，長文件必須分成好幾次翻頁檢查。**

本專案直接因應這四個問題：即時查詢韓國國立國語院的《標準國語大辭典》、우리말샘與語文規範，明確留下判斷的依據（在標記事由中一律載明依據）；只自動修改依據確鑿的項目；對模糊或視上下文而答案不同的項目，絕不隨機修改而是交由人類確認；並且不設字數上限，將一份文件檔案整份上傳、一次處理完——其中 v1 聚焦於**斷詞（空格）與拼寫校正**。

**目標**：以權威的語文規範依據為基礎、而非機率性猜測，就自動修改的項目達成 0 件錯誤。依據不確定的項目絕不擅自修改，而是交給人類。

**目標使用者**：專業譯者。目的在於減輕他們反覆進行斷詞（空格）與拼寫校正工作的負擔。因此誤判（明明正常卻被標記為錯誤）與遺漏（漏掉真正的錯誤）兩者都違背此目的——誤判會讓譯者浪費不必要的審校時間，遺漏則讓原本想替他們省下的工夫原封不動地留著。正因如此，「只自動處理確定的部分，模糊的一定委由人類」這個原則才格外重要。

## 2. 範圍（v1）

- 功能：韓語文字的**斷詞（空格）校正＋拼寫校正**
- 對象：純文字（.txt）、MS Word（.docx）、字幕（.srt）等各種格式的韓語文件——字幕檔案只是支援對象之一，本工具並非只為字幕作業而生
- 對象語言：**僅限韓語**

## 3. 輸入／輸出

- **輸入**：字幕（.srt）、純文字（.txt）、MS Word（.docx）檔案
- **輸出**：
  - 校正後的結果檔案（維持與原檔相同的格式——.srt 連時間碼都原樣保留）
  - 標記報告檔案（因模糊而未自動修改的項目清單）

## 4. 使用形態與技術堆疊

- **CLI 工具**
- 開發語言：**Python**
- **CLI 框架：Typer**
  - 由於結構上需要至少 2 個以上的子指令（`correct`：執行文字校正；`apply-report`：把使用者填好的報告重新套用到原檔），子指令管理比 argparse 更容易
  - 以型別提示為基礎，與本專案處理如韓國國立國語院 API 回應等明確資料結構的需求相當契合
  - 以 Click 為基礎，既維持穩定性又減少樣板程式碼
  - 日後以 pip 發佈時可直接沿用其慣例

### 架構原則（因應日後擴充）

校正邏輯（辭典／規範查詢、判斷、標記產生）設計為**與 CLI 分離的純函式庫模組**。CLI（Typer）僅作為呼叫此函式庫的薄介面。如此一來，在 §9 的後端擴充時，就能在 API 伺服器上直接沿用同一個校正引擎（無須大改 CLI 程式碼）。v1 階段不直接實作後端——現在要做的說到底就是本機 CLI 工具。

## 5. 校正判斷引擎（核心設計）

遇到模糊的句子時，依下列順序判斷；若到最後仍無法確信，就**不自動修改、原樣保留並加以標記**（基於「與其產生錯誤的修改，不如維持未修改狀態」的原則）。

1. **第 1 次判斷——規範／辭典依據**
   - 《標準國語大辭典》、우리말샘（韓國國立國語院 Open API，`search.do`）——確認是否有該詞條
   - 以《標準國語大辭典》`search.do` 回應的 `word`/`pos` 欄位，區分複合詞（連字號標記、有詞性→一律連寫）與名詞片語（插入符號標記、`pos: "품사 없음"`→依斷詞原則、亦允許連寫）。針對 kiwi.space() 漏掉的複合詞（例如「노천 카페」→未修正），以辭典依據補強而自動校正（`compound_status()`、`correct_compound_spacing()`，已實作）
   - 《標準國語大辭典》辭典內容 API（`view.do`）——確認用例、部分詞條所附的韓語拼寫依據（`norm_info`）（尚未實作）
   - 韓語語文規範 Open API（`kornorms/exampleReqList.do`）——查詢韓國國立國語院已官方確定的外來語、羅馬字標記用例（韓文標記 ↔ 原文標記，人名／地名／一般用語）。需要新的 API 金鑰
   - 外來語標記法規定原文（韓國國立國語院語文規範網站，5 條標記基本原則＋各語言別標記細則）——沒有 API 的靜態文件。僅在處理連 kornorms API 也沒有、完全嶄新的外來語（尚無官方標記的新造詞）時才參考的最後手段。若連這也模糊，就交到第 3 次
2. **第 2 次判斷——Online Ganada 答覆封存**
   - Online Ganada 並非即時 API，而是由人親自答覆的討論版，因此不進行即時查詢
   - 改為僅以**搜尋／參考既有已公開的過去答覆封存**的方式運用（累積的判例在 `subtitle_corrector/gananda_precedents.py`，尚未移入程式碼的待辦文法在 `docs/GRAMMAR_PRECEDENTS_TABLE.md`）
3. **第 3 次——仍然模糊就委由人類**
   - 禁止自動修改，於報告中標記

### 核心原則：「即時辭典資料永遠優先」

以《標準國語大辭典》／우리말샘 API 即時查詢的結果，永遠優先於程式碼中硬寫的假設或過去學到的教育資料內容。由於規定與辭典會改版，不能因為「以前確認過所以沒錯」就把固定值寫死在程式碼裡，而應在每次需要判斷的當下重新查詢。這個原則在實戰中為何必要（硬寫的假設實際上過時而成為錯誤的案例），請參閱 `docs/IMPLEMENTATION_LOG.md` §17「以即時查詢取代硬寫的驗證假設」。

### 工作原則：「先做驗證表，程式碼後寫」（2026-07-17 確定）

在 `correct_aux_verb_spacing()` 模式 2（「할만하다」類）中，曾以先寫程式碼、再對隨時發現的例外逐一修補的方式進行，結果被使用者以實際的 우리말샘 用例指出錯誤（把「할만하다」誤改為「할 만 하다」）多達兩次。原因就出在順序本身——先以「規則應該是這樣」的假設寫好程式碼，之後才以辭典驗證。

**今後在新增規則（尤其是可能有例外的文法模式）時，務必遵守以下順序**（具體程序參閱 `.claude/skills/grammar-rule-verify-then-code/SKILL.md`）：
1. 先把所有可能的情形（能歸入該模式的全部表達，如上述 §5-1/2/3，直到《標準國語大辭典》、우리말샘、kornorms、Online Ganada 判例）直接以辭典 API 查詢，**做出正解表（哪個輸入 → 哪個正解）。**
2. 寫出與該正解表完全相符的程式碼——是程式碼跟隨表，而不是表事後驗證程式碼。
3. 即使實作之後發現表中沒有的新案例，也不要臨時修補程式碼，而是先把該案例加入表中（重新查詢辭典），再依表修改程式碼。

### 已知限制／維護需求

實際使用中發現的細部誤判、錯誤及其修正原則，以及尚未實作的限制的完整清單，已移至 `docs/KNOWN_LIMITATIONS.md`——加入新功能前務必先瀏覽。

## 6. 標記與重新套用工作流程（已實作——`correct` / `apply-report`）

1. 處理整份字幕檔案
2. 能夠確信的項目立即自動校正
3. 模糊的項目在執行途中不中斷，待全部處理結束後，**彙整為單一報告檔案輸出**（行號＋文字＋參考依據）。像人名、地名這類雖已自動套用但需要再次確認的項目，會連同已套用的結果文字，一併在報告中留下「改了什麼、為什麼改」的參考依據。
4. 使用者在報告檔案中直接輸入修改值
5. 再次執行程式後，使用者輸入的修改值會**自動反映到原始字幕檔案**

- **測試集來源**：自行蒐集／製作（蒐集實際的 YouTube 影片、STT 產出並標註正解）

## 11. Web API／儲存（已實作，2026-07-16）

作為課程作業（「為我的服務加上後端」），實際實作了 §9 的「儲存」項目。未新增校正邏輯，依 §4 的架構原則僅直接呼叫既有引擎。

- **組成**：`subtitle_corrector/api.py`（FastAPI）——`POST /api/correct`（`.srt` 上傳 → `parsers.parse_srt` → `engine.correct_entries` → 儲存結果）／`GET /api/reports/{id}`（重新查詢已儲存的結果）。`static/index.html` 兼作上傳畫面與結果畫面（以 `?id=` 讓重新整理、在其他裝置上也能保留結果）。
- **儲存庫**：Supabase（Postgres），`subtitle_corrector/store.py` 以 `requests` 直接呼叫 REST（PostgREST）。資料表結構在 `supabase_schema.sql`。
- **安全設計——本專案選擇的結構並非講義資料中「瀏覽器直接存取 Supabase」的模式**：瀏覽器只向這個 FastAPI 伺服器發出請求，絕不直接存取 Supabase。因此即使只把 `SUPABASE_SERVICE_KEY`（管理員金鑰，會忽略 RLS）作為伺服器端環境變數保管也很安全；而且 `reports` 資料表已開啟 RLS 且未建立任何政策，因此以 anon（公開）金鑰的任何請求都會被拒絕——伺服器是唯一的存取路徑。
- **祕密值**：`STDICT_API_KEY`/`OPENDICT_API_KEY`/`KORNORMS_API_KEY`（既有）＋`SUPABASE_URL`/`SUPABASE_SERVICE_KEY`（新增），全部僅存在於 `.env`（本機）／Render 環境變數（部署）中。參閱 `.env.example`。
- **部署**：嘗試將 FastAPI 原樣部署到 Render（免費 Web 服務）。建立帳號、儀表板設定須由使用者親自進行（AI 無法代為登入），故在 `DEPLOY.md` 中逐步整理。**2026-07-22 實際嘗試部署的結果，kiwipiepy 模型的記憶體用量（約 310MB）幾乎用盡 Render 免費方案 512MB 的上限而發生 502 錯誤——暫緩此次部署嘗試本身**（參閱 `docs/RETROSPECTIVE.md`）。本機測試（以 stub 儲存庫驗證引擎串接）及實際 Supabase 串接（2026-07-16，由使用者親自建立帳號）皆已完成。
