## 1. 背景與目的

使用現有的韓語自動校對工具「釜山大學拼寫檢查器」時，主要有四個不便之處：**誤報過多（將正常表達和專有名詞錯誤地標記為錯誤）、不提供任何依據就要求修改、修改建議本身看起來毫無規律以至更像是新的錯誤，以及一次能檢查的字數有限，長文件需要分多次翻頁檢查。**

本專案直接應對這四個問題：透過即時查詢國立國語院的標準國語大辭典·我們語泉·語文規範，明確保留判斷依據（在標記原因中始終註明依據），僅自動修改有確切依據的項目，對模糊或因上下文而異的項目絕不隨意修改而是交由人工確認，上傳整個文件檔案無字數限制一次處理——其中 1.0 版專注於**分寫·拼寫校正**。

**目標**：基於權威語文規範的依據（而非機率性猜測），在自動修改的項目中實現零錯誤。對依據不確切的項目絕不輕易修改，交給人工處理。

**目標用戶**：專業翻譯。旨在減輕他們重複性的分寫·拼寫校對工作負擔。因此誤報（正常內容被標記為錯誤）和遺漏（真正的錯誤被放過）都違背此目的——誤報讓翻譯浪費不必要的時間在校對上，遺漏則使本應減輕的辛苦原封不動地留下。因此「確定的自動處理，有疑問的務必委託人工」這一原則尤為重要。

## 2. 範圍（1.0 版）

- **功能**：韓語文本的**分寫校正 + 拼寫校正**
- **對象**：純文本（.txt）、MS Word（.docx）、字幕（.srt）等多種格式的韓語文件——字幕檔案僅是支援對象之一，本工具並非僅為字幕工作而設計
- **目標語言**：**僅限韓語**

### 推遲至 2.0 版的項目（超出範圍）
- 翻譯文本的自然性·準確性驗證（需原文對照·翻譯風格判斷，無法僅憑本工具的判斷依據「權威規範性依據」進行驗證——參見 `docs/IMPLEMENTATION_LOG.md` §13）

## 3. 輸入 / 輸出

- **輸入**：字幕（.srt）、純文本（.txt）、MS Word（.docx）檔案
- **輸出**：
  - 校正結果檔案（保持與原檔案相同格式——.srt 連時間碼都原樣保留）
  - 標記報告檔案（因模糊而未自動修改的項目清單）

## 4. 使用形式與技術架構

- **CLI 工具**
- **開發語言**：**Python**
- **CLI 框架：Typer**
  - 需要至少 2 個以上的子命令（`correct`：執行文本校正、`apply-report`：將用戶填寫的報告重新應用到原檔案），因此比 argparse 更易於子命令管理
  - 基於型別提示，與本專案處理國立國語院 API 回應等清晰資料結構非常匹配
  - 基於 Click，保持穩定性的同時減少樣板程式碼
  - 遵循未來 pip 分發的慣例

### 架構原則（為將來擴展考慮）

校對邏輯（詞典/規範查詢、判斷、標記生成）設計為**與 CLI 分離的純函式庫模組**。CLI（Typer）僅作為呼叫該函式庫的薄介面。這樣在 §9 的後端擴展時，可以在 API 伺服器上直接複用相同的校對引擎（無需重寫 CLI 程式碼）。1.0 版階段不直接實現後端——現在創建的嚴格來說只是本地 CLI 工具。

## 5. 校對判斷引擎（核心設計）

遇到模糊的句子時，按以下順序判斷，到最後仍無法確定時**不自動修改而是標記**（錯誤修改不如未修改狀態好的原則）。

1. **第一次判斷——規範/詞典依據**
   - 標準國語大辭典、我們語泉（國立國語院 Open API，`search.do`）——確認詞頭存在與否
   - 用標準國語大辭典 `search.do` 回應的 `word`/`pos` 欄位區分合成詞（連字元標記，有詞性→必定連寫）和名詞短語（插入符標記，`pos: "無詞性"`→分寫原則·允許連寫）。用詞典依據補充 kiwi.space() 遺漏的合成詞（如"노천 카페"→不修改）進行自動校正（`compound_status()`、`correct_compound_spacing()`，已實現）
   - 標準國語大辭典內容 API（`view.do`）——確認用例、部分單詞附帶的韓語拼寫法依據（`norm_info`）（尚未實現）
   - 韓語語文規範 Open API（`kornorms/exampleReqList.do`）——查詢國立國語院已正式確定的外來語·羅馬字標記用例（韓語標記↔原文標記，人名/地名/一般用語）。需要新 API 金鑰
   - 外來語標記法規定原文（國立國語院語文規範網站，標記基本原則 5 條+各語言標記細則）——無 API 的靜態文件。僅在處理 kornorms API 中也沒有的全新外來語（尚無官方標記的新造詞）時作為最後手段參考。仍然模糊則轉入第三次

2. **第二次判斷——Online Gananda 回答檔案**
   - Online Gananda 不是即時 API 而是人工直接回答的論壇，不進行即時查詢
   - 改為僅透過**搜尋/參考已公開的過往回答檔案**方式利用（積累的判例在 `subtitle_corrector/gananda_precedents.py`，尚未轉為代碼的積壓語法在 `docs/GRAMMAR_PRECEDENTS_TABLE.md`）

3. **第三次——仍然模糊則委託人工**
   - 禁止自動修改，在報告中標記

### 核心原則：「即時詞典資料始終優先」

標準國語大辭典/我們語泉 API 的即時查詢結果，始終優先於代碼中的硬編碼假設或過去學習的教育資料內容。規定·詞典會修訂，因此不能將「以前確認過所以正確」的固定值硬編碼到代碼中，每次需要判斷時都要重新查詢。此原則在實際中為何必要（硬編碼假設實際上過時導致錯誤的案例），參見 `docs/IMPLEMENTATION_LOG.md` §17「將硬編碼驗證假設替換為即時查詢」。

### 工作原則：「驗證表優先，代碼在後」（2026-07-17 確定）

在 `correct_aux_verb_spacing()` 模式 2（"할만하다"類）中，採用了先寫代碼再在發現例外時逐個修補的方式，結果被用戶用實際我們語泉用例指出了兩次錯誤（"할만하다"被誤改為"할 만 하다"）。原因在於「規則應該是這樣的」這一假設先寫代碼再用詞典驗證的順序本身。

**今後添加新規則（特別是可能有例外的語法模式）時，必須遵守此順序**（具體步驟參見 `.claude/skills/grammar-rule-verify-then-code/SKILL.md`）：
1. 首先透過詞典 API 直接查詢所有相關情況（該模式中可能出現的所有表達，如上 §5-1/2/3 的標準國語大辭典·我們語泉·kornorms·Online Gananda 判例），創建**驗證表（什麼輸入→什麼正確答案）**。
2. 編寫完全匹配該表的代碼——代碼跟隨表，而非表事後驗證代碼。
3. 實現後發現表中沒有新的案例時，不要臨時修補代碼，先將該案例添加到表中（重新查詢詞典），然後按表修正代碼。

### 已知限制 / 維護需求

實際使用中發現的具體誤報·錯誤及其修正原則、尚未實現的限制的完整清單已移至 `docs/KNOWN_LIMITATIONS.md`——添加新功能前請務必先瀏覽。

## 6. 標記 & 再應用工作流程（已實現——`correct` / `apply-report`）

1. 處理整個字幕檔案
2. 有把握的項目立即自動校對
3. 模糊的項目在執行過程中不中斷，全部處理完成後**收集在一個報告檔案中輸出**（行號 + 文本 + 參考依據）。人名·地名等已自動應用但需再次確認的項目，報告中會保留已應用的結果文本以及為何更改的參考依據。
4. 用戶在報告檔案中直接輸入修正值
5. 再次執行程式時，用戶輸入的修正值會**自動應用到原字幕檔案**

## 9. 未來擴展路線圖（1.0 版之後）

現在創建的是本地 CLI 工具，但考慮到將來擴展為服務形式（Web/後端）的可能性，將以下項目作為路線圖保留。1.0 版設計僅反映 §4 的「架構原則」（校對引擎與介面分離），不包含以下功能本身。

- **登入/帳戶**：用於管理每個用戶的處理歷史。認證方式（自行實現 vs OAuth 等）尚未確定。尚未實現——§11 的儲存是在無需登入的「知道 ID 的任何人都可以查看結果」水準（短連結分享級別的安全性）。要區分管理多個用戶需先實現此項。
- **儲存——已實現（2026-07-16，參見 §11）**：將上傳文件的校對結果和標記報告儲存到 Supabase（Postgres）。儲存期限·個人資訊（文件內容）處理政策尚未確定（目前無限期儲存）——實際推出服務時必須決定。
- **付費**：付費方案/使用量計費。哪些功能需要付費化（如處理量、拼寫 API 呼叫次數）尚未確定。
- **領域專業術語詞典**：特定領域（醫學·法律等）的專業術語校正，如醫學術語詞典（KMLE）。由於不是國立國語院官方 Open API 而是民間網站，需先確認使用條款，且超出了 1.0 版的一般韓語文本範圍，推遲至 2.0 版之後。
- 此路線圖具體化時，將另設 PRD 處理（後端架構、認證、付費整合超出本文檔範圍）。

作為課堂作業（「給我的服務加後端」），實際實現了 §9 的「儲存」項目。未添加新的校對邏輯，僅按 §4 的架構原則直接呼叫現有引擎。

- **構成**：`subtitle_corrector/api.py`（FastAPI）——`POST /api/correct`（.srt 上傳 → `parsers.parse_srt` → `engine.correct_entries` → 儲存結果）/ `GET /api/reports/{id}`（重新獲取已儲存的結果）。`static/index.html` 兼作上傳畫面和結果顯示畫面（透過 `?id=` 重新整理·在其他裝置上也能保持結果）。
- **儲存**：Supabase（Postgres），`subtitle_corrector/store.py` 透過 `requests` 直接呼叫 REST（PostgREST）。表結構見 `supabase_schema.sql`。
- **安全設計——本專案選擇的結構不是教材中的「瀏覽器直接存取 Supabase」模式**：瀏覽器僅向本 FastAPI 伺服器傳送請求，絕不直接存取 Supabase。因此僅在伺服器端環境變數中持有 `SUPABASE_SERVICE_KEY`（管理員金鑰，忽略 RLS）是安全的，`reports` 表開啟 RLS 且不建立任何策略，因此 anon（公開）金鑰會拒絕所有請求——伺服器是唯一的存取路徑。
- **金鑰**：`STDICT_API_KEY`/`OPENDICT_API_KEY`/`KORNORMS_API_KEY`（現有）+ `SUPABASE_URL`/`SUPABASE_SERVICE_KEY`（新增），全部僅存在於 `.env`（本地）/Render 環境變數（部署）中。參見 `.env.example`。
- **部署**：嘗試將 FastAPI 直接部署到 Render（免費 Web 服務）。帳戶建立·儀表板設定需用戶自行操作（AI 無法代為登入），因此在 `DEPLOY.md` 中整理了逐步說明。**2026-07-22 實際部署嘗試結果：kiwipiepy 模型記憶體使用量（約 310MB）幾乎耗盡免費套餐 512MB 限額，導致 502 錯誤——該部署嘗試已暫停**（參見 `docs/RETROSPECTIVE.md`）。本地測試（用 stub 儲存驗證引擎整合）和實際 Supabase 整合（2026-07-16，用戶自行建立帳戶）已完成。
- **測試**：用 `examples/sample.srt` 驗證了 FastAPI 路由（200/404，檔案格式驗證）和引擎整合（將 `store.save_report`/`get_report` 替換為 stub 以在無實際 Supabase 帳戶的情況下驗證）。
