### [x] 1. 주어진 문제를 해결하는 완성된 코드가 제출되었나요?

CLI(`main.py`)와 웹 API(`subtitle_corrector/api.py`)가 동일한 엔진을 공유하는 구조로 구현되어 있고, 라우팅·유효성 검사·정적 페이지 서빙까지 실제 서버 기동으로 확인했습니다.

- 로컬에 임시 가상환경을 만들어 `uvicorn subtitle_corrector.api:app`을 직접 구동했습니다. `GET /` → 200(정적 페이지 정상 서빙), 잘못된 확장자(.pdf) 업로드 → `400 {".srt, .txt, .docx 파일만 지원합니다."}` (의도한 검증 그대로 동작).
- `POST /api/correct`에 `examples/sample.srt`를 실제로 업로드하면 요청이 `parse_srt → correct_entries → dictionary.word_exists → search_stdict`까지 정확히 코드가 의도한 순서로 도달하는 것을 서버 로그로 확인했습니다.
- **다만 이 프로젝트의 핵심 산출물(맞춤법·띄어쓰기 교정 자체의 정확성)은 국립국어원 API 키(`STDICT_API_KEY` 등)가 있어야 실제로 검증되는데, 이 키는 코더 개인 발급 키라 리뷰어 환경에 없습니다.** 그래서 위 요청은 `search_stdict()`가 `RuntimeError`를 던지는 지점에서 멈췄고(`500 Internal Server Error`), "교정이 실제로 정확하게 되는가"는 이번 리뷰에서 실시간으로 확인하지 못했습니다. 대신 `tests/test_integration.py`가 이 부분을 실제 API 응답 기준으로 검증하도록 설계되어 있으나, 같은 이유로 리뷰어 환경에서 `pytest`를 돌려 통과 여부를 확인하지는 못했습니다.
- **결론**: 코드 구조·배관(플러밍)은 완성되어 있고 실제로 동작함을 확인했지만, "기능이 정상적으로 작동하는지"의 가장 핵심인 교정 정확도는 **제출자가 API 키를 넣고 실행한 화면/로그를 근거로 직접 첨부해야 이 항목이 완전히 충족**됩니다. 현재 리포에는 그 근거(스크린샷, 실행 로그 등)가 없습니다.

**실제 동작하는 백엔드 — 실제로 확인한 것**
- 서버 기동, 정적 페이지 서빙(200), 잘못된 확장자 거부(400), 정상 확장자 요청이 실제 교정 엔진 내부(`dictionary.search_stdict`)까지 도달하는 것을 로그로 확인 — **배관 자체는 실제로 동작**합니다.
- 저장(Supabase) 실패와 교정 성공을 분리 처리하는 방어 코드(`api.py:82-90`)가 실제로 그 구조대로 짜여 있음을 코드로 확인했습니다(런타임으로 저장 실패 경로까지는 재현하지 않음).

**실제 동작하는 백엔드 — 문제/미흡한 점**
- API 키가 없을 때 앱이 부팅 시점에 이를 감지하지 못하고, 요청이 들어와서야 `correct_entries()` 깊숙한 곳에서 처음 던져지는 `RuntimeError`로 500이 발생합니다. 운영 관점에서는 배포 직후 설정 실수를 "첫 사용자의 첫 요청이 실패"하는 방식으로만 알아챌 수 있는 구조입니다 — 시작 시점에 필수 환경변수 존재 여부를 확인하는 fail-fast 체크가 없는 점은 명확한 개선 대상입니다.
- 위에서 언급했듯, **이 프로젝트가 약속하는 핵심 가치(정확한 맞춤법 교정)가 실제 사전 API 응답 기준으로 동작하는지는 이번 리뷰에서 직접 검증하지 못했습니다.** 리뷰어가 임의로 판단할 수 없는 영역이라, 코더가 실제 키로 구동한 결과(스크린샷, 로그, 또는 CI에서 시크릿을 넣고 돌린 pytest 결과)를 리포트에 첨부하는 것을 강하게 권합니다.

**프론트엔드-백엔드 연동 — 실제로 확인한 것**
- 브라우저로 페이지를 열어 파일을 선택하지 않고 "교정하기"를 눌렀을 때 `"파일을 먼저 선택하세요."`가 정확히 표시되는 것을 확인했습니다 — 클라이언트 유효성 검사가 소스 코드(`index.html:130`)와 실제 동작이 일치합니다.
- `fetch('/api/correct')`가 상대 경로를 쓰고, `StaticFiles`가 같은 FastAPI 앱에 마운트되어 있어(`api.py:110`) 프론트/백이 항상 같은 오리진에서 서빙되는 구조입니다 — 배포 시 CORS 이슈가 생길 여지가 없는 합리적인 설계입니다.
- 공유 링크 재조회 흐름(`?id=...` → `GET /api/reports/{id}`)에서 양쪽이 쓰는 필드명(`corrected_srt`, `applied_log`, `flags`, `id`)이 정확히 일치함을 코드 대조로 확인했습니다.
- 다만 실제 교정 결과가 화면에 렌더링되는 성공 경로 자체는(위와 같은 이유로 API 키가 없어) 눈으로 재현하지 못했고, 클라이언트 유효성 검사와 라우팅 배관만 확인했다는 점을 밝혀둡니다.
