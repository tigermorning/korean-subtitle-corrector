## PRT(Peer Review Template)

### [x] 1. 주어진 문제를 해결하는 완성된 코드가 제출되었나요?

CLI(`main.py`)와 웹 API(`subtitle_corrector/api.py`)가 동일한 엔진(`engine.correct_entries`)을 재사용하는 구조로 모두 구현되어 있고, `.srt`/`.txt`/`.docx` 세 가지 입력 형식을 실제로 처리합니다.

- 엔드 투 엔드 동작 근거: `examples/sample.srt`를 입력으로 넣었을 때 기대하는 출력이 `tests/test_integration.py`에 `EXPECTED_CORRECTED_TEXTS`로 명시되어 있고, "오늘은날씨가좋네요" → "오늘은 날씨가 좋네요"(띄어쓰기), "초코렛" → "초콜릿"(외래어), "스노우 기자"는 고유명사라 자동 반영 대신 플래그만(고유명사 안전장치)처럼 PRD가 약속한 판단 기준이 실제 파이프라인 출력과 일치함을 확인했습니다.
- 저장(Supabase) 실패와 교정 성공을 분리 처리하는 방어 코드도 있어 핵심 기능이 부가 기능 장애로 죽지 않게 설계되어 있습니다(`api.py:82-90`).
- 다만 이 환경에는 `.env`(국립국어원 API 키)가 없어 `pytest`를 직접 실행해 통과 여부를 검증하지는 못했습니다(README에도 "테스트는 실시간 사전 API 호출이 필요"라고 명시됨). 코드 정적 검토와 `py_compile`로 문법 오류가 없음은 확인했습니다. **리뷰이가 로컬에서 `pytest` 전체 통과 스크린샷을 리포트에 남기면 근거가 더 확실해질 것 같습니다.**

### [x] 2. 핵심적이거나 복잡한 부분의 설명을 보고 코드가 잘 이해되었나요?

이 프로젝트에서 가장 인상적인 부분입니다. 단순히 "무엇을 하는지"가 아니라 "왜 이렇게 짰는지"와 "이렇게 안 하면 어떤 사고가 나는지"를 각주처럼 남겨서, 코드만 봐서는 짐작하기 어려운 설계 의도가 명확하게 전달됩니다.

- 예: `_apply_replacements()`의 토큰 경계 매칭 이유 — "재판장님"(재판장+님) 안에 우연히 들어있는 "장님"을 "시각장애인"으로 잘못 치환하는 사고를 막기 위해 단순 문자열 매치가 아니라 kiwi 토큰 경계와 정확히 일치하는 위치만 교체한다는 설명(`engine.py:426-435`) — 그리고 이 설명이 실제로 `tests/test_engine.py:109` `test_jaepanjangnim_false_positive_fixed`로 뒷받침됩니다.
- 예: `_mechanical_respace()`가 토큰 표면형을 이어붙이지 않고 "간격"만 치환하는 이유 — '해'(하+어)처럼 형태소 위치가 겹치는 kiwi 특성 때문에 문자열을 재구성하면 음절이 중복 출력된다는 구체적 실패 사례까지 남겨둠(`engine.py:165-169`).
- `dictionary.py`의 `word_exists()`/`standard_term_replacement()` 문서화도 "우리말샘은 존재하면 곧 정답이 아니다"라는, API를 직접 조회해보지 않으면 알기 어려운 함정(예: "초코렛"도 표제어로 잡히지만 "규범 표기는 '초콜릿'" 안내가 붙어있음)을 근거와 함께 남겨서 이해에 크게 도움이 됩니다(`dictionary.py:75-93`).

### [x] 3. 에러를 디버깅한 기록, 또는 추가 시도/실험을 남겼나요?

`PRD.md`의 "알려진 한계 / 유지보수 요구사항" 절(§5)이 사실상 상세한 디버깅 일지 역할을 하고 있고, 각 사례가 실행 가능한 회귀 테스트로 그대로 옮겨져 있어 "기록만 남기고 검증은 안 함"이 아니라는 점이 좋습니다.

- 예: "집"(house)이 kornorms의 무관한 외래어 항목("jib"→"지브")과 우연히 겹쳐 잘못 치환되던 실사용 버그를 발견하고 `word_exists(token)` 선확인으로 수정한 기록(`PRD.md:101`) → `tests/test_engine.py:183` `test_registered_native_word_not_treated_as_loanword`로 회귀 테스트화.
- 예: "회전축에 목이"에서 조사로 분리된 "축"과 "목"이 합성어 후보 필터링 후 우연히 인접해 "회전축목이"로 잘못 합쳐지던 버그를 발견하고 간격 확인 로직(`gap_ok`)을 추가한 기록(`PRD.md:106`, `engine.py:335-336`).
- 커밋 로그도 "왜 고쳤는지"를 구체적으로 남기는 스타일을 일관되게 유지하고 있습니다(`git log` 확인: "고유명사(NNP) 외래어 표기 교정을 자동 반영 대신 항상 확인 플래그로 전환" 등).
- 추가 시도의 예로 "문맥 기반 자동판별을 시도했으나 채택하지 않은 이유"까지 남겨둔 점(`PRD.md:82`)이 특히 좋습니다 — 시도해서 실패한 접근도 기록해 다음 사람이 같은 길을 다시 가지 않게 합니다.

### [x] 5. 코드가 간결하고 효율적인가요?

전반적으로 관심사 분리가 잘 되어 있고(파싱/엔진/사전조회/리포트/저장/API가 각각 별 파일), 중복을 줄이려는 시도가 눈에 띕니다.

**잘된 점**
- `_apply_replacements()` 하나로 `correct_always_wrong`/`correct_nonstandard_terms`/`correct_discriminatory_terms` 세 기능이 공유하는 "치환값 사전(dict) → 적용" 로직을 통일했습니다(`engine.py:426-476`).
- API 호출에 `@lru_cache`를 붙여 같은 단어 반복 조회를 줄인 점, 그리고 그 트레이드오프(서버 재시작 시 캐시 초기화)까지 문서화한 점이 좋습니다(`dictionary.py:1-9`).
- 타입힌트(`str | None`, dataclass 등)를 일관되게 사용하고, PEP8 상 심각한 위반은 보이지 않습니다(`py_compile` 통과 확인, 줄 길이도 대체로 100~120자 이내).

**개선 여지가 있는 부분**
- `engine.py` 한 파일이 1,290줄로 이 프로젝트 코드량(~2,300줄)의 절반 이상을 차지합니다. 외래어/띄어쓰기/맞춤법/혼동어 관심사가 한 파일에 다 있어서, 예를 들어 `spacing.py`, `loanwords.py`처럼 기능 단위로 나누면 탐색성이 더 좋아질 것 같습니다.
- `_aux_verb_pattern_spans()`(`engine.py:531-562`)와 `correct_aux_verb_spacing()` 내부의 패턴 감지 로직(`engine.py:606-676`)이 "본용언+EC+VX", "ETM+의존명사+하다/싶다" 두 패턴의 태그 조건을 각자 다시 작성하고 있어 사실상 같은 판단 로직이 두 곳에 존재합니다. 용도가 미묘하게 달라(하나는 span 추출, 하나는 edit 계산) 완전히 합치기는 까다로워 보이지만, 조건 판정 부분만이라도 헬퍼로 뽑아내면 두 로직이 어긋날 위험(한쪽만 수정하고 다른 쪽을 놓치는 경우)을 줄일 수 있을 것 같습니다.
- `_protect_unfounded_respacing()`(`engine.py:1033-1139`)처럼 하나의 함수가 5~6가지 서로 다른 예외 케이스를 순차적으로 처리하는 긴 조건문이 몇 군데 있습니다. 각 케이스에 주석은 잘 달려있지만, 개별 조건을 이름 있는 함수로 뽑으면(`_is_bracket_meta_text()` 등) 함수 하나의 책임이 더 명확해질 것 같습니다.
