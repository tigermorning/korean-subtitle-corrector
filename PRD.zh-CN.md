## 1. 背景与目的

使用现有的韩语自动校对工具"釜山大学拼写检查器"时，主要有四个不便之处：**误报过多（将正常表达和专有名词错误地标记为错误）、不提供任何依据就要求修改、修改建议本身看起来毫无规律以至于更像是新的错误，以及一次能检查的字数有限，长文档需要分多次翻页检查。**

本项目直接应对这四个问题：通过实时查询国立国语院的标准国语大辞典·我们语泉·语文规范，明确保留判断依据（在标记原因中始终注明依据），仅自动修改有确切依据的项目，对模糊或因上下文而异的项目绝不随意修改而是交由人工确认，上传整个文档文件无字数限制一次处理——其中 v1 专注于**分写·拼写校正**。

**目标**：基于权威语文规范的依据（而非概率性猜测），在自动修改的项目中实现零错误。对依据不确切的项目绝不轻易修改，交给人工处理。

**目标用户**：专业翻译。旨在减轻他们重复性的分写·拼写校对工作负担。因此误报（正常内容被标记为错误）和遗漏（真正的错误被放过）都违背此目的——误报让翻译浪费不必要的时间在校对上，遗漏则使本应减轻的辛苦原封不动地留下。因此「确定的自动处理，有疑问的务必委托人工」这一原则尤为重要。

## 2. 范围（v1）

- **功能**：韩语文本的**分写校正 + 拼写校正**
- **对象**：纯文本（.txt）、MS Word（.docx）、字幕（.srt）等多种格式的韩语文档——字幕文件仅是支持对象之一，本工具并非仅为字幕工作而设计
- **目标语言**：**仅限韩语**

### 推迟至 v2 的项目（超出范围）
- 翻译文本的自然性·准确性验证（需原文对照·翻译风格判断，无法仅凭本工具的判断依据"权威规范性依据"进行验证——参见 `docs/IMPLEMENTATION_LOG.md` §13）

## 3. 输入 / 输出

- **输入**：字幕（.srt）、纯文本（.txt）、MS Word（.docx）文件
- **输出**：
  - 校正结果文件（保持与原文件相同格式——.srt 连时间码都原样保留）
  - 标记报告文件（因模糊而未自动修改的项目清单）

## 4. 使用形式与技术栈

- **CLI 工具**
- **开发语言**：**Python**
- **CLI 框架：Typer**
  - 需要至少 2 个以上的子命令（`correct`：执行文本校正、`apply-report`：将用户填写的报告重新应用到原文件），因此比 argparse 更易于子命令管理
  - 基于类型提示，与本项目处理国立国语院 API 响应等清晰数据结构非常匹配
  - 基于 Click，保持稳定性的同时减少样板代码
  - 遵循未来 pip 分发的惯例

### 架构原则（为将来扩展考虑）

校正逻辑（词典/规范查询、判断、标记生成）设计为**与 CLI 分离的纯库模块**。CLI（Typer）仅作为调用该库的薄接口。这样在 §9 的后端扩展时，可以在 API 服务器上直接复用相同的校正引擎（无需重写 CLI 代码）。v1 阶段不直接实现后端——现在创建的严格来说只是本地 CLI 工具。

## 5. 校正判断引擎（核心设计）

遇到模糊的句子时，按以下顺序判断，到最后仍无法确定时**不自动修改而是标记**（错误修改不如未修改状态好的原则）。

1. **第一次判断——规范/词典依据**
   - 标准国语大辞典、我们语泉（国立国语院 Open API，`search.do`）——确认词头存在与否
   - 用标准国语大辞典 `search.do` 响应的 `word`/`pos` 字段区分合成词（连字符标记，有词性→必定连写）和名词短语（插入符标记，`pos: "无词性"`→分写原则·允许连写）。用词典依据补充 kiwi.space() 遗漏的合成词（如"노천 카페"→不修改）进行自动校正（`compound_status()`、`correct_compound_spacing()`，已实现）
   - 标准国语大辞典内容 API（`view.do`）——确认用例、部分单词附带的韩语拼写法依据（`norm_info`）（尚未实现）
   - 韩语语文规范 Open API（`kornorms/exampleReqList.do`）——查询国立国语院已正式确定的外来语·罗马字标记用例（韩语标记↔原文标记，人名/地名/一般用语）。需要新 API 密钥
   - 外来语标记法规定原文（国立国语院语文规范网站，标记基本原则 5 条+各语言标记细则）——无 API 的静态文档。仅在处理 kornorms API 中也没有的全新外来语（尚无官方标记的新造词）时作为最后手段参考。仍然模糊则转入第三次

2. **第二次判断——Online Gananda 回答档案**
   - Online Gananda 不是实时 API 而是人工直接回答的论坛，不进行实时查询
   - 改为仅通过**搜索/参考已公开的过往回答档案**方式利用（积累的判例在 `subtitle_corrector/gananda_precedents.py`，尚未转为代码的积压语法在 `docs/GRAMMAR_PRECEDENTS_TABLE.md`）

3. **第三次——仍然模糊则委托人工**
   - 禁止自动修改，在报告中标记

### 核心原则：「实时词典数据始终优先」

标准国语大辞典/我们语泉 API 的实时查询结果，始终优先于代码中的硬编码假设或过去学习的教育资料内容。规定·词典会修订，因此不能将"以前确认过所以正确"的固定值硬编码到代码中，每次需要判断时都要重新查询。此原则在实际中为何必要（硬编码假设实际上过时导致错误的案例），参见 `docs/IMPLEMENTATION_LOG.md` §17"将硬编码验证假设替换为实时查询"。

### 工作原则：「验证表优先，代码在后」（2026-07-17 确定）

在 `correct_aux_verb_spacing()` 模式 2（"할만하다"类）中，采用了先写代码再在发现例外时逐个修补的方式，结果被用户用实际我们语泉用例指出了两次错误（"할만하다"被误改为"할 만 하다"）。原因在于"规则应该是这样的"这一假设先写代码再用词典验证的顺序本身。

**今后添加新规则（特别是可能有例外的语法模式）时，必须遵守此顺序**（具体步骤参见 `.claude/skills/grammar-rule-verify-then-code/SKILL.md`）：
1. 首先通过词典 API 直接查询所有相关情况（该模式中可能出现的所有表达，如上 §5-1/2/3 的标准国语大辞典·我们语泉·kornorms·Online Gananda 判例），创建**验证表（什么输入→什么正确答案）**。
2. 编写完全匹配该表的代码——代码跟随表，而非表事后验证代码。
3. 实现后发现表中没有的新案例时，不要临时修补代码，先将该案例添加到表中（重新查询词典），然后按表修正代码。

## 6. 标记 & 再应用工作流（已实现——`correct` / `apply-report`）

1. 处理整个字幕文件
2. 有把握的项目立即自动校正
3. 模糊的项目在执行过程中不中断，全部处理完成后**收集在一个报告文件中输出**（行号 + 文本 + 参考依据）。人名·地名等已自动应用但需再次确认的项目，报告中会保留已应用的结果文本以及为何更改的参考依据。
4. 用户在报告文件中直接输入修正值
5. 再次运行程序时，用户输入的修正值会**自动应用到原字幕文件**

## 11. Web API / 存储（已实现，2026-07-16）

作为课堂作业（"给我的服务加后端"），实际实现了 §9 的"存储"项目。未添加新的校正逻辑，仅按 §4 的架构原则直接调用现有引擎。

- **构成**：`subtitle_corrector/api.py`（FastAPI）——`POST /api/correct`（.srt 上传 → `parsers.parse_srt` → `engine.correct_entries` → 保存结果）/ `GET /api/reports/{id}`（重新获取已保存的结果）。`static/index.html` 兼作上传画面和结果显示画面（通过 `?id=` 刷新·在其他设备上也能保持结果）。
- **存储**：Supabase（Postgres），`subtitle_corrector/store.py` 通过 `requests` 直接调用 REST（PostgREST）。表结构见 `supabase_schema.sql`。
- **安全设计——本项目选择的结构不是教材中的"浏览器直接访问 Supabase"模式**：浏览器仅向本 FastAPI 服务器发送请求，绝不直接访问 Supabase。因此仅在服务器端环境变量中持有 `SUPABASE_SERVICE_KEY`（管理员密钥，忽略 RLS）是安全的，`reports` 表开启 RLS 且不创建任何策略，因此 anon（公开）密钥会拒绝所有请求——服务器是唯一的访问路径。
- **密钥**：`STDICT_API_KEY`/`OPENDICT_API_KEY`/`KORNORMS_API_KEY`（现有）+ `SUPABASE_URL`/`SUPABASE_SERVICE_KEY`（新增），全部仅存在于 `.env`（本地）/Render 环境变量（部署）中。参见 `.env.example`。
- **部署**：尝试将 FastAPI 直接部署到 Render（免费 Web 服务）。账户创建·仪表盘设置需用户自行操作（AI 无法代为登录），因此在 `DEPLOY.md` 中整理了逐步说明。**2026-07-22 实际部署尝试结果：kiwipiepy 模型内存使用量（约 310MB）几乎耗尽免费套餐 512MB 限额，导致 502 错误——该部署尝试已暂停**（参见 `docs/RETROSPECTIVE.md`）。本地测试（用 stub 存储验证引擎集成）和实际 Supabase 集成（2026-07-16，用户自行创建账户）已完成。
- **测试**：用 `examples/sample.srt` 验证了 FastAPI 路由（200/404，文件格式验证）和引擎集成（将 `store.save_report`/`get_report` 替换为 stub 以在无实际 Supabase 账户的情况下验证）。
- **Web API/存储相关实际使用错误 4 件、kiwi 未注册单词误报、安全/SQL 综合审查、存储失败隔离等**：详细经过参见 `docs/IMPLEMENTATION_LOG.md`（§11 相关项目、§13、§24、§25）。
