## 1. 背景及目的

在使用既有的韩语自动校正工具「釜山大学拼写检查器」时，令人不满之处大致有四点：**误报过多（连正常的表达·专有名词都被错误地当成错误挑出来），毫无依据地一味要求修改却不说为什么错，而且那些修改建议本身看起来随机到毫无一致性，反而让那些建议感觉像是新的错误，还有单次能检查的字数有限制，长文档不得不分多次翻页检查。**

本项目直接应对这四个问题：实时查询韩国国立国语院的《标准国语大词典》·우리말샘·语文规范，清楚地留下判断依据（在标记理由中始终写明依据），只自动修改依据确凿的项目，对含糊或因上下文而答案不一的项目绝不随机修改而是交由人确认，并且不设字数限制、一次整份上传文档文件来处理——其中 v1 集中于**空格·拼写校正**。

**目标**：以权威的语文规范依据而非概率性猜测为基础，仅就自动修改的项目达成 0 件错误。依据不确定的项目绝不随意修改，而是交给人。

**目标用户**：专业翻译。目的是减轻他们反复进行空格·拼写校正工作的负担。因此误报（本来正常却被标记为错）和遗漏（漏掉真正的错误）两者都有悖于这一目的——误报让翻译在不必要的校对上耗费时间，遗漏则把本想替其减轻的辛劳原封不动地留下。正因如此，「只自动处理确定的、含糊就务必委托给人」这一原则尤为重要。

## 2. 范围（v1）

- 功能：韩语文本的**空格校正 + 拼写校正**
- 对象：一般文本（.txt）、MS Word（.docx）、字幕（.srt）等各种格式的韩语文档——字幕文件也只是支持对象之一，本工具并非仅为字幕工作而设
- 目标语言：**仅韩语**

### 推迟到 v2 的项目（Out of scope）
- 翻译文本的自然度·准确性验证（需要与原文对照·判断翻译风格，仅凭本工具的判断标准「权威的规范依据」无法验证——参考 `docs/IMPLEMENTATION_LOG.md` §13）

## 3. 输入 / 输出

- **输入**：字幕（.srt）、一般文本（.txt）、MS Word（.docx）文件
- **输出**：
  - 校正后的结果文件（保持与原文相同的格式——.srt 连时间码都原样保留）
  - 标记报告文件（因含糊而未自动修改的项目列表）

## 4. 使用形态及技术栈

- **CLI 工具**
- 开发语言：**Python**
- **CLI 框架：Typer**
  - 由于是需要至少 2 个以上子命令（`correct`：执行文本校正，`apply-report`：将用户填好的报告重新应用到原文）的结构，子命令管理比 argparse 更容易
  - 基于类型提示，与本项目处理国立国语院 API 响应等明确数据结构的场景很契合
  - 基于 Click，在保持稳定性的同时样板代码较少
  - 便于日后 pip 发布时沿用惯例

### 架构原则（为日后扩展做准备）

校正逻辑（词典/规范查询、判断、标记生成）设计为**与 CLI 分离的纯库模块**。CLI（Typer）仅作为调用这个库的薄接口。这样在 §9 的后端扩展时，就能在 API 服务器上原样复用同一个校正引擎（无需重写 CLI 代码）。v1 阶段并不直接实现后端——现在做的终究只是本地 CLI 工具。

## 5. 校正判断引擎（核心设计）

遇到含糊的句子时，按下列顺序判断，直到最后仍无法确信时**不自动修改、原样保留并标记**（宁可保持未修改状态，也好过产生错误修改的原则）。

1. **第 1 层判断——规范/词典依据**
   - 《标准国语大词典》、우리말샘（国立国语院 Open API，`search.do`）——确认表条目是否存在
   - 用《标准国语大词典》`search.do` 响应的 `word`/`pos` 字段来区分合成词（连字符标记，有词性 → 一律连写）与名词短语（脱字符标记，`pos: "无词性"` → 空格原则·允许连写）。对 kiwi.space() 漏掉的合成词（例如「노천 카페」→ 不修改），以词典依据加以补充并自动校正（`compound_status()`、`correct_compound_spacing()`，实现完成）
   - 《标准国语大词典》词典内容 API（`view.do`）——确认用例、部分词条附带的韩语拼写依据（`norm_info`）（尚未实现）
   - 韩语语文规范开放 API（`kornorms/exampleReqList.do`）——查询国立国语院已官方确定的外来语·罗马字标记用例（韩语标记 ↔ 原语标记，人名/地名/一般术语）。需要新的 API 密钥
   - 外来语标记法规定原文（国立国语院语文规范网站，标记基本原则 5 项条款 + 各语言标记细则）——没有 API 的静态文档。仅在处理连 kornorms API 里也没有的全新外来语（尚无官方标记的新造词）时才参考的最后手段。若连这也含糊，则移交第 3 层
2. **第 2 层判断——Online Ganada 答复存档**
   - Online Ganada 不是实时 API，而是由人亲自答复的论坛，因此不做实时提问
   - 而是仅以**检索/参考此前已公开的过往答复存档**的方式加以利用（累积的判例在 `subtitle_corrector/gananda_precedents.py`，尚未转成代码的待办语法在 `docs/GRAMMAR_PRECEDENTS_TABLE.md`）
3. **第 3 层——若仍含糊则委托给人**
   - 禁止自动修改，在报告中标记

### 核心原则：「实时词典数据始终优先」

以《标准国语大词典》/우리말샘 API 实时查询的结果，始终优先于代码中硬编码的假设或过去学到的教学资料内容。规定·词典是会修订的，所以不能以「以前确认过所以正确」为由把固定值写死在代码里，而要在每次需要判断的时刻重新查询。这一原则在实战中为何有必要（硬编码的假设实际过时而变成 bug 的案例），参考 `docs/IMPLEMENTATION_LOG.md` §17「将硬编码的验证假设替换为实时查询」。

### 工作原则：「先做验证表，代码在后」（2026-07-17 确定）

在 `correct_aux_verb_spacing()` 模式 2（「할만하다」类）中，采用先写代码、随后遇到例外就临时打补丁的做法，结果被用户以实际 우리말샘 用例指出错误（把「할만하다」错误地改成「할 만 하다」）两次之久。原因就出在顺序本身——先以「规则大概是这样」的假设写代码，再用词典去验证。

**今后新增规则（尤其是可能有例外的语法模式）时，务必遵守以下顺序**（具体步骤参考 `.claude/skills/grammar-rule-verify-then-code/SKILL.md`）：
1. 先把所有相关的可能情况（能纳入该模式的全部表达，如上文 §5-1/2/3 那样，直到《标准国语大词典》·우리말샘·kornorms·Online Ganada 判例）都用词典 API 直接查询，**做成正确答案表（哪种输入 → 哪个正确答案）。**
2. 写出与那张正确答案表精确吻合的代码——是代码去跟随表，而不是表事后验证代码。
3. 即便实现之后，若发现表里没有的新案例，也不要临时给代码打补丁，而是先把该案例加入表中（重新查询词典），再按表修正代码。

## 6. 标记 & 重新应用工作流程（实现完成——`correct` / `apply-report`）

1. 处理整份字幕文件
2. 对可以确信的项目立即自动校正
3. 含糊的项目在执行中不中断，而是在全部处理结束后**汇总输出为一个报告文件**（行号 + 文本 + 参考依据）。对于人名·地名这类虽已自动应用但需要复核的项目，连同已应用的结果文本，在报告中留下什么被改了、为什么改的参考依据。
4. 用户直接在报告文件中填入修改值
5. 再次运行程序时，用户填入的修改值会**自动反映到原始字幕文件**

## 11. Web API / 存储（实现完成，2026-07-16）

作为课程作业（「给我的服务加上后端」），实际实现了 §9 的「存储」项目。没有新增校正逻辑，而是按 §4 的架构原则原样只调用既有引擎。

- **构成**：`subtitle_corrector/api.py`（FastAPI）——`POST /api/correct`（`.srt` 上传 → `parsers.parse_srt` → `engine.correct_entries` → 保存结果）/ `GET /api/reports/{id}`（重新查询已保存的结果）。`static/index.html` 既是上传界面又是结果界面（用 `?id=` 刷新·在其他设备上也保留结果）。
- **存储库**：Supabase（Postgres），`subtitle_corrector/store.py` 用 `requests` 直接调用 REST（PostgREST）。表结构在 `supabase_schema.sql`。
- **安全设计——本项目所选的结构不是讲义资料中「浏览器直接访问 Supabase」的模式**：浏览器只向这个 FastAPI 服务器请求，绝不直接访问 Supabase。因此即便只把 `SUPABASE_SERVICE_KEY`（管理员密钥，无视 RLS）作为服务器端环境变量持有也是安全的，而且 `reports` 表开着 RLS 却一条策略都不创建，用 anon（公开）密钥的任何请求都会被拒绝——服务器是唯一的访问路径。
