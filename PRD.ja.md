## 1. 背景および目的

既存の韓国語自動校正ツール「釜山大学正書法チェッカー」を使っていて不満だった点は大きく4つでした: **誤検出が過度に多く（正常な表現・固有名詞まで誤ってエラーと判定）、なぜ間違っているのか根拠なしにとにかく直せと言い、その修正提案自体がランダムに見えるほど一貫性がなく、むしろその提案が新たなエラーのように感じられ、一度に検査できる文字数に制限があって長い文書は何度も分けてページを送りながら検査しなければなりませんでした。**

このプロジェクトはこの4つの問題に直接対応します: 国立国語院の標準国語大辞典・ウリマルセム(開放型辞典)・語文規範をリアルタイムで照会して判断根拠を明確に残し（フラグ理由に根拠を常に明示）、根拠が確実な項目のみ自動的に直し、曖昧だったり文脈によって答えが分かれる項目は決してランダムに直さず人に確認を任せ、文字数制限なしに文書ファイル一つを丸ごとアップロードして一度に処理します — このうちv1は**分かち書き・正書法の校正**に集中します。

**目標**: 確率的な推測ではなく権威ある語文規範の根拠に基づき、自動修正した項目に限りエラー0件を達成します。根拠が不確実な項目は決してむやみに直さず人に渡します。

**対象ユーザー**: プロの翻訳者。彼らの反復的な分かち書き・正書法の校正作業の負担を減らすことが目的です。そのため誤検出（正常なのに間違いとフラグ）と見落とし（本物のエラーを逃す）の両方がこの目的に反します — 誤検出は翻訳者に不要な校閲時間を使わせ、見落としはそもそも軽減しようとした手間をそのまま残します。このため「確実なものだけ自動処理、曖昧なら必ず人に委任」という原則が特に重要です。

## 2. 範囲（v1）

- 機能: 韓国語テキストの**分かち書き校正 + 正書法校正**
- 対象: 一般テキスト(.txt)、MS Word(.docx)、字幕(.srt) など様々な形式の韓国語文書 — 字幕ファイルも対応対象の一つにすぎず、このツールが字幕作業だけのためのものではありません
- 対象言語: **韓国語専用**

### v2に先送りした項目（Out of scope）
- 翻訳テキストの自然さ・正確性の検証（原文対照・翻訳スタイルの判断が必要で、このツールの判断基準である「権威ある規範根拠」だけでは検証する方法がない — `docs/IMPLEMENTATION_LOG.md` §13 参照）

## 3. 入力 / 出力

- **入力**: 字幕(.srt)、一般テキスト(.txt)、MS Word(.docx) ファイル
- **出力**:
  - 校正された結果ファイル（原本と同じフォーマットを維持 — .srt はタイムコードまでそのまま保存）
  - フラグレポートファイル（曖昧で自動修正しなかった項目のリスト）

## 4. 使用形態および技術スタック

- **CLIツール**
- 開発言語: **Python**
- **CLIフレームワーク: Typer**
  - 最低2つ以上のサブコマンド（`correct`: テキスト校正の実行、`apply-report`: ユーザーが記入したレポートを原本に再適用）が必要な構造なので、argparseよりサブコマンド管理が容易
  - 型ヒントベースなので、国立国語院API応答など明確なデータ構造を扱うこのプロジェクトと相性が良い
  - Clickベースなので安定性は維持しつつボイラープレートが少ない
  - 今後のpip配布時の慣例をそのまま踏襲

### アーキテクチャ原則（今後の拡張に備えて）

校正ロジック（辞書/規範照会、判断、フラグ生成）は**CLIと分離した純粋なライブラリモジュール**として設計します。CLI（Typer）はこのライブラリを呼び出す薄いインターフェースとしてのみ置きます。こうすると§9のバックエンド拡張時に同じ校正エンジンをAPIサーバーでそのまま再利用できます（CLIコードを作り直す必要がない）。v1時点でバックエンドを直接実装はしません — 今作っているのはあくまでローカルCLIツールです。

## 5. 校正判断エンジン（中核設計）

曖昧な文に出会ったとき、以下の順序で判断し、最後まで確信できなければ**自動修正せずそのまま残したうえでフラグ**します（誤った修正が生じるより未修正状態のほうが良いという原則）。

1. **1次判断 — 規範/辞書根拠**
   - 標準国語大辞典、ウリマルセム(開放型辞典)（国立国語院 Open API、`search.do`）— 見出し語の存在有無を確認
   - 標準国語大辞典 `search.do` 応答の `word`/`pos` フィールドで、合成語（ハイフン表記、品詞あり → 無条件でくっつける）と名詞句（キャレット表記、`pos: "품사 없음"` → 分かち書き原則・くっつけ許容）を区別。kiwi.space()が見逃す合成語（例: 「노천 카페」→ 直さない）を辞書根拠で補完して自動校正（`compound_status()`、`correct_compound_spacing()`、実装完了）
   - 標準国語大辞典 辞書内容API（`view.do`）— 用例、一部の単語に付いた韓国語正書法の根拠（`norm_info`）を確認（まだ未実装）
   - 韓国語語文規範 オープンAPI（`kornorms/exampleReqList.do`）— 国立国語院がすでに公式確定した外来語・ローマ字表記の用例（ハングル表記 ↔ 原語表記、人名/地名/一般用語）を照会。新しいAPIキーが必要
   - 外来語表記法規定の原文（国立国語院 語文規範サイト、表記の基本原則5条項 + 言語別表記細則）— APIがない静的文書。kornorms APIにもない完全に新しい外来語（まだ公式表記がない新造語）を扱うときだけ参照する最終手段。これでも曖昧なら3次に回す

2. **2次判断 — Online Ganada回答アーカイブ**
   - Online GanadaはリアルタイムAPIではなく人が直接回答する掲示板なので、リアルタイム質疑はしない
   - 代わりに**既に公開された過去回答アーカイブを検索/参照する**方式でのみ活用（蓄積した判例は `subtitle_corrector/gananda_precedents.py`、まだコードに移していないバックログの文法は `docs/GRAMMAR_PRECEDENTS_TABLE.md`）

3. **3次 — それでも曖昧なら人に委任**
   - 自動修正禁止、レポートにフラグ

### 中核原則: 「リアルタイム辞書データが常に優先」

標準国語大辞典/ウリマルセムAPIでリアルタイム照会した結果が、コードにハードコードされた仮定や過去に学習した教育資料の内容より常に優先されます。規定・辞書は改定されるので、「以前確認したから正しい」とコードに固定値で埋め込んではならず、判断が必要な瞬間ごとに再度照会します。この原則が実戦でなぜ必要だったのか（ハードコードされた仮定が実際に古くなってバグになった事例）は `docs/IMPLEMENTATION_LOG.md` §17「ハードコードされた検証仮定をリアルタイム照会に置き換え」参照。

### 作業原則: 「検証表が先、コードは後」（2026-07-17 確定）

`correct_aux_verb_spacing()` パターン2（「할만하다」類）で、コードを先に書いて発見される例外をその都度パッチする方式で進めていたところ、ユーザーが実際のウリマルセム用例で指摘したエラー（「할만하다」を「할 만 하다」に誤って直した）を二度も経験しました。原因は、「規則はこうであろう」という仮定で先にコードを書いてから辞書で検証した順序そのものにありました。

**今後、規則（特に例外があり得る文法パターン）を新たに追加するときは、必ずこの順序を守ります**（具体的な手順は `.claude/skills/grammar-rule-verify-then-code/SKILL.md` 参照）:
1. 関連するすべての場合の数（該当パターンに入りうる表現の全部、上記§5-1/2/3のように標準国語大辞典・ウリマルセム・kornorms・Online Ganada判例まで）を先に辞書APIで直接照会して、**正答表（どの入力 → どの正答）を作る。**
2. その正答表に正確に合うコードを書く — コードが表に従うのであって、表がコードを事後検証するのではない。
3. 実装後にも表になかった新事例が発見されたら、コードを一時的にパッチせず、その事例をまず表に追加（辞書を再照会）したうえで、表に合わせてコードを直す。

### 既知の限界 / 保守要件

実使用中に発見した細かな誤検出・バグとその修正原則、まだ未実装の限界の全リストは `docs/KNOWN_LIMITATIONS.md` に移しました — 新機能を入れる前に必ずまず読み通すこと。

## 6. フラグ & 再適用ワークフロー（実装完了 — `correct` / `apply-report`）

1. 字幕ファイル全体を処理
2. 確信できる項目は即座に自動校正
3. 曖昧な項目は実行中に中断されず、全体処理が終わった後に**レポートファイル一つにまとめて出力**（行番号 + テキスト + 参考根拠）。人名・地名のように自動適用はされたがダブルチェックが必要な項目は、すでに適用された結果テキストと共に、何が・なぜ変わったのかの参考根拠をレポートに残します。
4. ユーザーがレポートファイルに直接修正値を入力
5. プログラムを再実行すると、ユーザーが入力した修正値が**原本の字幕ファイルに自動反映**

## 7. 完了基準 / 検証方法

  - 目標: プログラムが自動で（フラグなしで）修正した項目のうちエラー0件
- **テストセット出典**: 直接収集/制作（実際のYouTube動画・STT結果物を集めて正答ラベリング）

## 9. 今後の拡張ロードマップ（v1以降）

今作っているものはローカルCLIツールですが、後にサービス形態（Web/バックエンド）へ拡張する可能性を考慮して以下の項目をロードマップとして残しておきます。v1設計には§4の「アーキテクチャ原則」（校正エンジンとインターフェースの分離）だけを反映し、以下の機能自体は作りません。

- **ログイン/アカウント**: ユーザー別の処理履歴管理のために必要。認証方式（自前実装 vs OAuth など）は未定。まだ未実装 — §11の保存はログインなしで「idを知っている人なら誰でもその結果を見られる」という水準（短いリンク共有程度のセキュリティ）です。複数のユーザーを区別して管理するには、この項目が先行する必要があります。
- **保存 — 実装完了（2026-07-16、§11 参照）**: アップロードした文書の校正結果とフラグレポートをSupabase(Postgres)に保存。保存期間・個人情報（文書内容）の処理ポリシーはまだ決めていない（無期限保管中）— 実サービスに出す場合は必ず決定が必要。
- **決済**: 有料プラン/使用量課金の有無。どの機能を有料化するか（例: 処理量、正書法API呼び出し回数）は未定
- **ドメイン別専門用語辞書**: 医学用語辞書（KMLE）など特定分野（医学・法律など）の専門用語校正。国立国語院の公式オープンAPIではなく民間サイトなので利用規約の確認が先行する必要があり、一般韓国語テキストというv1の範囲を外れるのでv2以降に先送り
- このロードマップが具体化されたら別のPRDに分離して扱います（バックエンドアーキテクチャ、認証、決済連携はこの文書の範囲外）

## 11. Web API / 保存（実装完了、2026-07-16）

授業の課題（「自分のサービスにバックエンドを付ける」）として§9の「保存」項目を実際に実装しました。新しい校正ロジックは追加せず、§4のアーキテクチャ原則どおり既存のエンジンをそのまま呼び出すだけです。

- **構成**: `subtitle_corrector/api.py`（FastAPI）— `POST /api/correct`（`.srt` アップロード → `parsers.parse_srt` → `engine.correct_entries` → 結果保存）/ `GET /api/reports/{id}`（保存された結果の再照会）。`static/index.html` がアップロード画面兼結果画面（`?id=` でリロード・他の端末でも結果を維持）。
- **ストレージ**: Supabase(Postgres)、`subtitle_corrector/store.py` がREST(PostgREST)を `requests` で直接呼び出し。テーブルスキーマは `supabase_schema.sql`。
- **セキュリティ設計 — このプロジェクトが選択した構造は講義資料の「ブラウザがSupabaseに直接アクセス」パターンではありません**: ブラウザはこのFastAPIサーバーにのみリクエストし、Supabaseには決して直接アクセスしません。したがって `SUPABASE_SERVICE_KEY`（管理者キー、RLS無視）をサーバー側の環境変数としてのみ持っていても安全であり、`reports` テーブルはRLSをオンにしてポリシーを一つも作らないことで、anon（公開）キーではどんなリクエストも拒否されます — サーバーが唯一のアクセス経路です。
- **秘密値**: `STDICT_API_KEY`/`OPENDICT_API_KEY`/`KORNORMS_API_KEY`（既存）+ `SUPABASE_URL`/`SUPABASE_SERVICE_KEY`（新規）、すべて `.env`（ローカル）/Render環境変数（デプロイ）としてのみ存在。`.env.example` 参照。
- **デプロイ**: Render（無料Webサービス）にFastAPIをそのままデプロイ試行。アカウント作成・ダッシュボード設定はユーザーが自分で行う必要があるため（AIが代わりにログイン不可）`DEPLOY.md` に段階的に整理しておきました。**2026-07-22 実際のデプロイ試行の結果、kiwipiepyモデルのメモリ使用量（約310MB）がRender無料ティア512MB上限をほぼ使い切って502エラーが発生 — このデプロイ試行自体を保留しました**（`docs/RETROSPECTIVE.md` 参照）。ローカルテスト（スタブストレージでエンジン連携を検証）および実際のSupabase連携（2026-07-16、ユーザーが自分でアカウント作成）までは完了。
- **テスト**: `examples/sample.srt` でFastAPIルート（200/404、ファイル形式検証）とエンジン連携を確認しました（`store.save_report`/`get_report` をスタブに置き換えて実際のSupabaseアカウントなしで検証）。
