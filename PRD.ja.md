## 1. 背景と目的

既存の韓国語自動校正ツール「プサン国立大学 拼字検査器」使いで特に不便だった点は大きく4つあった：**誤検出が極めて多かった（正常な表現・固有名詞まで誤ってエラーとして検出）、なぜ間違っているか根拠なしに無条件に修正させ、その修正提案自体がランダムに見えるほど一貫性がなくむしろその提案が新しいエラーのように感じられ、一度に検査できる文字数に制限があるため長い文書は複数回に分けてページをめくって検査する必要があった。**

このプロジェクトはこの4つの問題に直接対応する：国立国語院の標準国語大辞典・ウリマルセム・語文規範をリアルタイムで照会して判断根拠を明確に残し（フラグ理由に根拠を常に明示）、根拠が確実な項目のみ自動で修正し、曖昧な場合や文脈によって答えが分かれる項目は決してランダムに修正せず人間に確認を委任し、文字数制限なしに文書ファイルを丸ごとアップロードして一度に処理する——このうちv1は**分かち書き・スペル校正**に集中する。

**目標**：確率的推測ではなく権威ある語文規範の根拠に基づき、自動修正した項目についてエラー0件を達成する。根拠が不確かな項目は決して安易に修正せず人間に渡す。

**対象ユーザー**：プロの翻訳者。彼らの繰り返し発生する分かち書き・スペル校正作業の負担を軽減することが目的。したがって誤検出（正常なのに誤りとしてフラグ）も漏れ（本当に誤りを見落とす）もこの目的に反する——誤検出は翻訳者が不要な校正時間を使うことになり、漏れは本来軽減しようとした手間をそのまま残す。このため「確実なものの自動自動処理、曖昧な場合は必ず人間に委任」という原則が特に重要。

## 2. 範囲（v1）

- **機能**：韓国語テキストの**分かち書き校正 + スペル校正**
- **対象**：プレーンテキスト（.txt）、MS Word（.docx）、字幕（.srt）など様々な形式の韓国語文書——字幕ファイルも対象の一つであり、このツールが字幕作業専用であるわけではない
- **対象言語**：**韓国語専用**

### v2 に先送りした項目（Out of scope）
- 翻訳テキストの自然さ・正確性検証（原文対照・翻訳スタイル判断が必要で、このツールの判断基準である「権威ある規範的根拠」だけでは検証方法がない——`docs/IMPLEMENTATION_LOG.md` §13 参照）

## 3. 入力 / 出力

- **入力**：字幕（.srt）、プレーンテキスト（.txt）、MS Word（.docx）ファイル
- **出力**：
  - 校正結果ファイル（元本と同じフォーマットを維持——.srt はタイムコードまでそのまま保存）
  - フラグレポートファイル（曖昧で自動修正しなかった項目のリスト）

## 4. 使用形態と技術スタック

- **CLI ツール**
- **開発言語**：**Python**
- **CLI フレームワーク：Typer**
  - 少なくとも 2 以上のサブコマンド（`correct`：テキスト校正実行、`apply-report`：ユーザーが入力したレポートを元本に再適用）が必要な構造であり、argparse よりサブコマンド管理が容易
  - タイプヒントベースであり、国立国語院 API 応答などの明確なデータ構造を扱う本プロジェクトと相性が良い
  - Click ベースであり、安定性を維持しながらボイラープレートを削減
  - 将来の pip 配布時の慣例に従う

### アーキテクチャ原則（将来の拡張に備えて）

校正ロジック（辞書/規範照会、判断、フラグ生成）は**CLI と分離された純粋なライブラリモジュール**として設計する。CLI（Typer）はこのライブラリを呼び出す薄いインターフェースのみとする。これにより §9 のバックエンド拡張時に同一の校正エンジンを API サーバーでそのまま再利用できる（CLI コードを書き直す必要がない）。v1 段階でバックエンドを直接実装はしない——今作成するのはあくまでローカル CLI ツール。

## 5. 校正判断エンジン（コア設計）

曖昧な文に遭遇した場合、以下の順序で判断し、最後まで確信が持てない場合は**自動修正せずそのままフラグ**する（誤った修正が生じるより未修正状態の方が良いという原則）。

1. **第一次判断——規範/辞書根拠**
   - 標準国語大辞典、ウリマルセム（国立国語院 Open API、`search.do`）——見出し語の存在確認
   - 標準国語大辞典 `search.do` 応答の `word`/`pos` フィールドで合成語（ハイフン表記、品詞あり→常に結合）vs 名詞句（キャレット表記、`pos: "品詞なし"`→分かち書き原則・結合許容）を区別。kiwi.space() が見落とす合成語（例：「노천 카페」→修正しない）を辞書根拠で補完して自動校正（`compound_status()`、`correct_compound_spacing()`、実装完了）
   - 標準国語大辞典辞書内容 API（`view.do`）——用例、一部の単語に付いた韓国語 拼字法根拠（`norm_info`）確認（まだ未実装）
   - 韓国語 語文規範 Open API（`kornorms/exampleReqList.do`）——国立国語院がすでに正式に確定した外来語・ローマ字表記用例（韓国語表記↔原語表記、人名/地名/一般用語）照会。新しい API キーが必要
   - 外来語表記法規定原文（国立国語院語文規範サイト、表記基本原則 5 条項+言語別表記細則）——API がない静的ドキュメント。kornorms API にもない完全に新しい外来語（まだ正式表記がない新造語）を扱う場合のみ参考にする最終手段。これでも曖昧なら第三次へ

2. **第二次判断——オンラインガナダ回答アーカイブ**
   - オンラインガナダはリアルタイム API ではなく人が直接回答する掲示板であり、リアルタイム質問はしない
   - 代わりに**既に公開された過去の回答アーカイブを検索/参照する方式のみ**で活用（蓄積された判例は `subtitle_corrector/gananda_precedents.py`、まだコードに移行していないバックログ文法は `docs/GRAMMAR_PRECEDENTS_TABLE.md`）

3. **第三次——それでも曖昧なら人間に委任**
   - 自動修正禁止、レポートにフラグ

### コア原則：「リアルタイム辞書データが常に優先」

標準国語大辞典/ウリマルセム API によるリアルタイム照会結果が、コードにハードコーディングされた前提や過去に学習した教育資料の内容より常に優先する。規定・辞書は改訂されるため、「以前確認したから正しい」とコードに固定値として詰め込むのは不可、判断が必要な都度再照会する。この原則が実際になぜ必要だったか（ハードコーディングされた前提が実際に古くなってバグになった事例）は `docs/IMPLEMENTATION_LOG.md` §17「ハードコーディングされた検証前提をリアルタイム照会に置換」参照。

### 作業原則：「検証表から先、コードは後」（2026-07-17 確定）

`correct_aux_verb_spacing()` パターン 2（「할만하다」系）でコードを先に書いて発見される例外をその都度パッチする方式で進行していたが、ユーザーが実際のウリマルセム用例で指摘した誤り（「할만하다」を「할 만 하다」に誤修正）を2度も経験した。原因は「規則はこうであろう」という前提で先にコードを書いてから辞書で検証した順序自体にあった。

**今後ルール（特に例外があるかもしれない文法パターン）を新たに追加する場合は必ずこの順序を守る**（具体的な手続きは `.claude/skills/grammar-rule-verify-then-code/SKILL.md` 参照）：
1. 関連するすべてのケース（そのパターンに入りうる表現すべて、上記 §5-1/2/3 のように標準国語大辞典・ウリマルセム・kornorms・オンラインガナダ判例まで）をまず辞書 API で直接照会して**検証表（ 어떤 입력 → 어떤 정답）** を作成する。
2. その検証表に正確に合ったコードを書く——コードが表に従うのであり、表がコードを事後検証するのではない。
3. 実装後にも表にない新しいケースが発見された場合、コードを一時的にパッチせず、まずそのケースを表に追加（辞書再照会）してから表に合わせてコードを修正する。

### 既知の限界 / メンテナンス要件

実使用中に発見された具体的な誤検出・バグとその修正原則、未実装の限界の完全なリストは `docs/KNOWN_LIMITATIONS.md` に移動済み——新機能を追加する前に必ずまず目を通すこと。

## 6. フラグ & 再適用ワークフロー（実装完了——`correct` / `apply-report`）

1. 字幕ファイル全体を処理
2. 確信できる項目は即座に自動校正
3. 曖昧な項目は実行中割り込みなし、全処理終了後に**レポートファイル1つにまとめて出力**（行番号 + テキスト + 参考根拠）。人名・地名のように自動適用されたが再チェックが必要な項目は、すでに適用された結果テキストとともに何がなぜ変更されたかの参考根拠をレポートに残す。
4. ユーザーがレポートファイルに直接修正値を入力
5. プログラムを再実行するとユーザーが入力した修正値が**元の字幕ファイルに自動反映**

## 7. 完了基準 / 検証方法

- **検証方法**：人が正解を付けたテストセットに対する精度測定
  - 目標：プログラムが自動で（フラグなしに）修正した項目の中でエラー0件
- **テストセット出처**：直接収集/作成（実際の YouTube 動画・STT 結果物を集めて正解ラベリング）

## 9. 将来拡張ロードマップ（v1 以降）

今作成するのはローカル CLI ツールだが、将来サービス形態（Web/バックエンド）への拡張可能性を考慮し、以下の項目をロードマップとして残す。v1 設計には §4 の「アーキテクチャ原則」（校正エンジンとインターフェースの分離）のみを反映し、以下の機能自体は作成しない。

- **ログイン/アカウント**：ユーザー別処理履歴管理に必要。認証方式（自前実装 vs OAuth 等）は未定。まだ未実装——§11 の保存はログインなしで「ID を知っている人は誰でもその結果を見られる」レベル（短いリンク共有程度のセキュリティ）。複数ユーザーを区別して管理する場合はこの項目が前提となる。
- **保存——実装完了（2026-07-16、§11 参照）**：アップロードした文書の校正結果とフラグレポートを Supabase（Postgres）に保存。保存期間・個人情報（文書内容）処理ポリシーはまだ未決定（無期限保存中）——実サービスに出す場合は必ず決定が必要。
- **決済**：有料プラン/使用量課金。何を有料化するか（例：処理量、スペル API 呼び出し回数）は未定。
- **ドメイン別専門用語辞書**：医学用語辞書（KMLE）など特定分野（医学・法律等）専門用語校正。国立国語院公式 Open API ではなく民間サイトなので利用規約確認が前提となり、v1 の範囲である一般韓国語テキストを逸脱するため v2 以降に先送り。
- このロードマップが具体化したら別途 PRD として分離して扱う（バックエンドアーキテクチャ、認証、決済連携はこの文書の範囲外）。

## 11. Web API / 保存（実装完了、2026-07-16）

授業課題（「自分のサービスにバックエンドを付ける」）として §9 の「保存」項目を実際に実装した。新しい校正ロジックは追加せず、§4 のアーキテクチャ原則通り既存エンジンをそのまま呼び出すのみ。

- **構成**：`subtitle_corrector/api.py`（FastAPI）——`POST /api/correct`（.srt アップロード → `parsers.parse_srt` → `engine.correct_entries` → 結果保存）/ `GET /api/reports/{id}`（保存された結果の再取得）。`static/index.html` がアップロード画面兼結果画面（`?id=` でリフレッシュ・他のデバイスでも結果維持）。
- **保存先**：Supabase（Postgres）、`subtitle_corrector/store.py` が REST（PostgREST）を `requests` で直接呼び出し。テーブルスキーマは `supabase_schema.sql`。
- **セキュリティ設計——このプロジェクトが選んだ構造は教材の「ブラウザが Supabase に直接アクセス」パターンではない**：ブラウザはこの FastAPI サーバーへのみリクエストし、Supabase には決して直接アクセスしない。したがって `SUPABASE_SERVICE_KEY`（管理者キー、RLS 無視）をサーバー側環境変数のみで持っていても安全であり、`reports` テーブルは RLS をオンにしてポリシーを一つも作らず anon（公開）キーではすべてのリクエストが拒否される——サーバーが唯一のアクセス経路。
- **シークレット**：`STDICT_API_KEY`/`OPENDICT_API_KEY`/`KORNORMS_API_KEY`（既存）+ `SUPABASE_URL`/`SUPABASE_SERVICE_KEY`（新規）、すべて `.env`（ローカル）/Render 環境変数（デプロイ）のみに存在。`.env.example` 参照。
- **デプロイ**：Render（無料 Web サービス）に FastAPI そのままデプロイを試行。アカウント作成/ダッシュボード設定はユーザーが手動で行う必要がある（AI が代行ログイン不可）ため `DEPLOY.md` にステップバイステップで整理。**2026-07-22 実際のデプロイ試行結果、kiwipiepy モデルメモリ使用量（約 310MB）が無料ティア 512MB 限界にほぼ達し 502 エラー発生——このデプロイ試行自体を保留**（`docs/RETROSPECTIVE.md` 参照）。ローカルテスト（スタブストレージによるエンジン統合検証）と実際の Supabase 統合（2026-07-16、ユーザーがアカウント作成）は完了済み。
- **テスト**：`examples/sample.srt` で FastAPI ルート（200/404、ファイル形式検証）とエンジン統合を確認（`store.save_report`/`get_report` をスタブに置換して実際の Supabase アカウントなしで検証）。
- **Web API/保存に関する実使用バグ 4 件、kiwi 未登録単語誤検出、セキュリティ/SQL 総合検証、保存失敗分離等**：詳細な経緯は `docs/IMPLEMENTATION_LOG.md`（§11 関連項目、§13、§24、§25）参照。
