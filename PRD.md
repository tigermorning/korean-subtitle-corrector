# 한국어 자막 띄어쓰기·맞춤법 자동 교정 CLI 도구 — PRD (초안)

## 1. 배경 및 목적

기존 자막 자동화 툴(유튜브 자동자막, Whisper 기반 툴 등)을 쓰면서 답답했던 오류는 크게 4가지였다: **인식 오류, 타이밍 싱크 오류, 문장 다듬기 오류, 무음/잡음 처리 오류.**

이 중 v1은 **문장 다듬기 오류**, 그 안에서도 **띄어쓰기·맞춤법 교정**에 집중한다. 나머지 세 영역(인식/타이밍/무음)은 확률적 판단(음성 인식 모델 정확도)에 의존해 "절대 오류 없음"을 보장하기 어렵지만, 띄어쓰기·맞춤법은 국립국어원의 공식 규범 문서를 근거로 삼을 수 있어 결정론적으로 검증 가능한 목표다.

**목표**: 확률적 추측이 아닌 권위 있는 어문 규범 근거 기반으로, 자동 수정한 항목에 한해 오류 0건을 달성한다. 근거가 불확실한 항목은 절대 함부로 고치지 않고 사람에게 넘긴다.

**대상 사용자**: 전문 번역가. 이들의 반복적인 띄어쓰기·맞춤법 교정 작업 부담을 줄여주는 것이 목적이다. 그래서 오탐(정상인데 틀렸다고 플래그)과 미탐(진짜 오류를 놓침) 둘 다 이 목적에 반한다 — 오탐은 번역가가 불필요하게 감수 시간을 쓰게 만들고, 미탐은 애초에 덜어주려던 수고를 그대로 남긴다. 이 때문에 "확실한 것만 자동 처리, 애매하면 반드시 사람에게 위임"이라는 원칙이 특히 중요하다.

## 2. 범위 (v1)

- 기능: 한국어 자막의 **띄어쓰기 교정 + 맞춤법 교정**
- 대상: **동일 언어 받아쓰기** 자막 (번역 자막 아님)
- 대상 언어: **한국어 전용**

### v2로 미룬 항목 (Out of scope)
- 인식(STT) 오류 개선
- 타이밍 싱크 오류 개선
- 무음/잡음 처리 오류 개선
- 다국어/번역 자막 지원

## 3. 입력 / 출력

- **입력**: 다음 두 가지를 모두 지원
  - 이미 만들어진 SRT/VTT 자막 파일
  - STT 엔진(Whisper 등)이 뱉은 raw 데이터 (타임스탬프 + 원시 텍스트)
- **출력**:
  - 교정된 자막 파일 (원본과 동일한 포맷·타임스탬프 유지)
  - 플래그 리포트 파일 (모호해서 자동 수정하지 않은 항목 목록)

## 4. 사용 형태 및 기술 스택

- **CLI 툴**
- 개발 언어: **Python**
- **CLI 프레임워크: Typer**
  - 최소 2개 이상의 서브커맨드(`correct`: 자막 교정 실행, `apply-report`: 사용자가 채운 리포트를 원본에 재적용)가 필요한 구조라 argparse보다 서브커맨드 관리가 쉬움
  - 타입 힌트 기반이라 국립국어원 API 응답 등 명확한 데이터 구조를 다루는 이 프로젝트와 궁합이 좋음
  - Click 기반이라 안정성은 유지하면서 보일러플레이트가 적음
  - 향후 pip 배포 시 관례를 그대로 따름

### 아키텍처 원칙 (향후 확장 대비)

교정 로직(사전/규범 조회, 판단, 플래그 생성)은 **CLI와 분리된 순수 라이브러리 모듈**로 설계한다. CLI(Typer)는 이 라이브러리를 호출하는 얇은 인터페이스로만 둔다. 이렇게 하면 §9의 백엔드 확장 시 동일한 교정 엔진을 API 서버에서 그대로 재사용할 수 있다 (CLI 코드를 뜯어고칠 필요 없음). v1 시점에 백엔드를 직접 구현하지는 않는다 — 지금 만드는 건 어디까지나 로컬 CLI 도구다.

## 5. 교정 판단 엔진 (핵심 설계)

모호한 문장을 만났을 때 아래 순서로 판단하며, 마지막까지 확신할 수 없으면 **자동 수정하지 않고 그대로 둔 채 플래그**한다 (잘못된 수정이 생기는 것보다 미수정 상태가 낫다는 원칙).

1. **1차 판단 — 규범/사전 근거**
   - 표준국어대사전, 우리말샘 (국립국어원 Open API, `search.do`) — 표제어 존재 여부 확인
   - 표준국어대사전 `search.do` 응답의 `word`/`pos` 필드로 합성어(하이픈 표기, 품사 있음 → 무조건 붙여씀) vs 명사구(캐럿 표기, `pos: "품사 없음"` → 띄어쓰기 원칙·붙임 허용) 구분. kiwi.space()가 놓치는 합성어(예: "노천 카페" → 안 고침)를 사전 근거로 보완해 자동 교정 (`compound_status()`, `correct_compound_spacing()`, 구현 완료)
   - 표준국어대사전 사전 내용 API(`view.do`) — 용례, 일부 단어에 붙은 한글 맞춤법 근거(`norm_info`) 확인 (아직 미구현)
   - 한국어 어문 규범 오픈API(`kornorms/exampleReqList.do`) — 국립국어원이 이미 공식 확정한 외래어·로마자 표기 용례(한글표기 ↔ 원어표기, 인명/지명/일반용어) 조회. 새 API 키 필요
   - 외래어 표기법 규정 원문(국립국어원 어문규범 누리집, 표기 기본 원칙 5개 조항 + 언어별 표기 세칙) — API가 없는 정적 문서. kornorms API에도 없는 완전히 새로운 외래어(아직 공식 표기가 없는 신조어)를 다룰 때만 참고하는 최후 수단. 이마저 애매하면 3차로 넘김
2. **2차 판단 — 온라인가나다 답변 아카이브**
   - 온라인가나다는 실시간 API가 아니라 사람이 직접 답변하는 게시판이므로, 실시간 질의는 하지 않는다
   - 대신 **기존에 공개된 과거 답변 아카이브를 검색/참고**하는 방식으로만 활용
3. **3차 — 그래도 모호하면 사람에게 위임**
   - 자동 수정 금지, 리포트에 플래그

### 알려진 한계 / 유지보수 요구사항

- **인명·지명 자동 반영 여부의 진짜 기준은 "정답이 하나로 일치하느냐" (구현 완료, 이후 아래 항목에서 고유명사 부분만 재수정됨)**: 처음에는 "인명·지명이면 무조건 확인 필요"로 구분했으나 이는 잘못된 기준이었다. 예를 들어 "스노우"는 인명(Snow, Edgar Parks 등) 표기에서도 예외 없이 "스노"가 맞다 — 이건 문맥에 따라 갈리는 게 아니라 국립국어원이 이미 확정한 단일 정답(relate_mark_o에 "(X)"로 명시)이므로, 일반 용어(초코렛→초콜릿)와 똑같이 확인 없이 자동 반영해야 한다. 진짜 확인이 필요한 경우는 **같은 원어에 대해 kornorms에 등재된 정답 자체가 서로 다르게 갈리는 경우**뿐이다 (예: Jeremiah → 예레미야(성경식) vs 제러마이아(현대 인명) 처럼 실제로 다른 관례가 공존). 이 경우는 텍스트만으로는 어느 쪽인지 판단할 수 없고 실제 영상 속 발음을 들어야만 확정 가능하므로, 첫 번째 후보를 적용하되 항상 "확인 필요"로 플래그한다. `loanword_fix()`는 검색된 모든 일치 항목의 교정값이 일치하는지(`foreign_gubun`이 아니라)로 이를 판단한다. 구현 시 추가로 발견한 문제: kornorms의 `korean_mark`는 인명 항목에서 "성, 이름" 전체를 담고 있어, 토큰 하나("스노우")를 통째로 전체 이름("스노, 에드거 파크스")으로 바꿔버리는 오류가 있었다 — `_closest_segment()`로 토큰과 가장 유사한 조각만 추출해 적용하도록 수정함.
- **고유명사(NNP) 외래어 표기는 정답이 하나뿐이어도 절대 자동 반영하지 않고 항상 확인 플래그로 전환 (구현 완료, 2026-07-15)**: 위 원칙("정답이 하나면 자동 반영")은 인명 표기 자체(스노우 기자→스노 기자)에는 맞지만, 같은 표기가 **특정 작품·제품의 고정된 고유 제목**일 경우 위험하다는 것을 사용자가 지적해 발견함 — 예를 들어 실제로 우리말샘에 "스노우맨"(2017년 영화·소설 제목)이 "⇒규범 표기는 '스노맨'이다"라는 안내와 함께 등재돼 있는데, 이 영화의 한국 배급사가 실제로 "스노우맨"이라는 표기를 그대로 제목으로 썼다면(트레이드마크성 고유 표기), 표기 규칙을 강제 적용해 "스노맨"으로 바꾸는 순간 실제 작품명을 훼손하는 오류가 된다. 문제는 kiwi의 NNP 태그만으로는 "인명 표기(규칙을 따라야 함)"와 "고정된 작품 제목(규칙과 무관하게 그 자체가 정답)"을 구분할 신호가 전혀 없다는 것 — 텍스트 자체에 이 둘을 가를 근거가 없으므로, 고유명사 전체를 자동 교정 대상에서 완전히 제외하고 항상 사람이 최종 승인하도록 정책을 바꿨다(`correct_loanwords()`가 NNP 매치는 `proper_noun_suggestions`로 분리해 텍스트를 건드리지 않고 `suggested_fix`만 채운 플래그로 남김). 트레이드오프: 자동화 비율은 줄지만(예: "스노우 기자"도 이제 매번 확인이 필요), "절대 오류 없는" 원칙상 안전이 우선이라는 사용자 결정에 따른 것.
  - **문맥 기반 자동판별을 시도하지 않은 이유**: "주변에 '영화', '개봉' 같은 단서가 있으면 고유 제목으로 보고 자동 반영을 건너뛴다" 같은 문맥 휴리스틱도 검토했으나 채택하지 않았다. 그런 단서 없이도 작품을 언급하는 문장이 흔하고("우리는 그거 봤어"), 반대로 사람 얘기를 하면서 우연히 그런 단어가 근처에 나올 수도 있어 오탐/누락 위험이 여전하다. 더 근본적으로, 이 프로젝트의 원칙(§5, 확률적 추측이 아닌 권위 있는 규범 근거 기반) 자체가 "이 표기가 특정 작품의 고정 제목인지"를 판정해 줄 국립국어원 데이터 소스가 없다는 뜻이므로, 휴리스틱을 만들어도 그 자체가 또 다른 확률적 추측일 뿐이다. 따라서 사람 확인이 유일하게 정당한 해법이다.
- **규칙 기반 도출과 관용(예외)의 구분**: "제러마이아"처럼 표기 세칙(예: 어말 스와(/ə/) → '아')으로 체계적으로 도출되는 일반 용어 표기와, "머라이어"(Mariah Carey)처럼 규칙이 아니라 유명세로 굳어진 순수 관용 예외는 서로 다르다. kornorms 선례 검색은 이미 등재된 관용 예외(일반 용어)는 잘 잡지만, **아직 kornorms에 없는 새 일반 용어가 규칙으로는 도출 가능한 경우**를 처리하려면 별도의 표기 세칙 규칙 엔진(IPA 기반)이 필요하다 — 아직 미구현
- **표기 세칙 규칙 엔진의 범위**: 외래어 표기는 영어만이 아니라 전 세계 모든 언어의 발음을 한글로 표기할 수 있어야 한다 (국립국어원 표기 세칙도 영어·독일어·프랑스어·에스파냐어·이탈리아어·일본어·중국어 등 20개 언어를 다룬다). 따라서 이 규칙 엔진은 영어 하나만 다루는 것보다 훨씬 큰 작업이며, v1 범위를 넘어선다 — v2 이후 백로그로 명확히 분리한다
- **규정 개정에 따른 유지보수**: 한국어 어문 규정(특히 외래어 표기법)은 종종 개정된다(예: 1986년 최초 제정 → 2017년 개정). kornorms API(선례 데이터베이스)는 국립국어원이 직접 관리·갱신하므로 우리 쪽 유지보수 부담이 적지만, 만약 표기 세칙을 직접 규칙 엔진으로 구현한다면 개정 여부를 주기적으로 확인하고 코드를 갱신하는 책임이 생긴다. 이 때문에 규칙을 직접 구현하기보다 kornorms 같은 국립국어원 관리 API에 최대한 의존하는 쪽을 우선한다.
  - **운영 원칙(2026-07-15 추가)**: 이 프로젝트로 다시 작업을 시작할 때마다(세션 시작 시), 국립국어원 웹 검색으로 한글 맞춤법/외래어 표기법/표준어 규정에 최근 개정이 있었는지 가볍게 확인하는 것을 관례로 삼는다. AI 세션은 상시 백그라운드로 돌지 않으므로 "주기적 자동 감시"는 불가능하지만, 매 작업 세션 시작 시 확인하는 것으로 사실상의 주기적 점검을 대신한다. 2026-07-15 확인 결과: 웹 검색상 2025~2026년 사이의 한글 맞춤법·외래어 표기법 개정 공지는 발견되지 않음(외래어 표기법은 1986년 제정, 2017년 개정 이후 그대로로 보임) — 현재 구현이 최신 규정과 어긋난다는 증거 없음.
- **고유명사(인명)를 맞춤법 오류로 오탐하는 문제 (구현 완료)**: 기존 도구(부산대 맞춤법 검사기)가 사람 이름을 이상하게 바꾸거나 잘못 처리한다는 지적이 있어 우리 코드를 실제로 점검했더니, 같은 종류의 버그가 있었다 — "민준"처럼 정상적인 한국인 이름이 표준국어대사전 표제어가 아니라는 이유만으로 "사전에 없는 단어"로 오탐됐다. 원인은 맞춤법 검사(§5-2)에서 고유명사(NNP) 태그를 일반명사와 동일하게 사전 대조했기 때문. 고유명사를 검사 대상에서 제외해 수정함 (`engine.py`의 `_SPELLING_CHECK_TAGS`는 NNG/VV/VA만 포함, NNP 제외).
- **인명·지명 검증은 원어(로마자) 없이는 근본적 한계가 있음 (미구현, 백로그)**: 자막에 이미 한글로 옮겨진 고유명사만으로는, 그게 맞는 표기인지 원어 자체를 모르니 검증이 불가능한 경우가 많다. 해결 방향: 번역가가 고유명사를 원어(로마자) 그대로 남겨두면, 프로그램이 이를 자동 감지(한글과 다른 토큰 태그)해 kornorms에 원어로 직접 조회 후 공식 표기로 치환한다. 이미 한글로 쓰인 고유명사는 번역가의 의도된 결정으로 보고 건드리지 않는다. 사용자 안내문("고유명사는 원지어를 그대로 두면 정확한 결과를 얻을 수 있음")과 함께, 실제 동작은 텍스트 형태로 자동 판별되게 해 사용자가 안내를 놓쳐도 최소한 결과가 일관되게 한다.
- **합성어 자동 붙여쓰기 (구현 완료)**: 표준국어대사전 표제어는 합성어를 하이픈("노천-카페")으로, 명사구를 캐럿("예방^접종")으로 표기하며 `pos` 필드로도 구분된다(품사 있음=합성어/무조건 붙임, `pos: "품사 없음"`=명사구/붙임 허용). `kiwi.space()`는 이런 합성어를 놓치는 경우가 있어("노천 카페" 원문을 안 고침), 사전 조회로 보완해 자동 교정한다. 부작용으로 kiwi가 이미 확정된 합성어를 다시 갈라놓자고 제안하는 모순이 있었는데, `check_spacing()`에서 확정된 합성어 부분만 원상복구해 해결함.
- **국립국어원 "한국어 어문 규범"(한글 맞춤법 해설서, 96쪽) 정독 완료 (2026-07-15)**: kiwi는 보조 도구일 뿐이고 국립국어원 자료가 항상 우선한다는 원칙에 따라, 규정 원문+해설을 직접 확인함. 주요 발견:

  - **제30항(사이시옷)**: 합성어+앞말 모음+뒷말 된소리/ㄴ 첨가+고유어 최소 1개 포함(외래어 있으면 불가) 조건. 이미 구현한 `compound_status()`(사전 하이픈 판정)로 사실상 커버됨 — 사이시옷 여부는 사전 표제어 형태 자체에 이미 반영되어 있으므로 별도 규칙 엔진 불필요
  - **제47항(보조 용언)**: 훈련 자료에서 배운 붙임 허용/항상 붙임/항상 띄움 분류표의 공식 근거 확인 + 세부 보강 — 2음절 예외는 본용언의 "활용형" 음절 수 기준(원형 음절 수가 아님), '예뻐하다'는 2023년에 표준국어대사전 표제어로 등재되어 항상 붙임, '도와드리다'는 '도와주다'가 사전 표제어인 것에 맞춰 항상 붙임
  - **제57항(동음이의어 혼동 쌍)**: 가름/갈음, 거름/걸음, 반드시/반듯이, 시키다/식히다, 붙이다/부치다, 목거리/목걸이 등 국립국어원이 직접 지정한 "자주 헷갈리는 말" 공식 목록 확보 — ②(확정 오류 패턴) 작업의 유력 후보. 다만 이들은 의미가 완전히 다른 별개의 단어라 어느 쪽이 맞는지는 문맥 의미 판단이 필요함 (예: "그리고 나서→그러고 나서"처럼 예외 없이 틀린 경우와는 다름) — 자동 교정 대상이 아니라 "이 단어 자주 헷갈림, 확인 필요" 플래그 후보로 적합
  - 문장 부호 규정(마침표~쌍점 등)도 확인했으나, 이는 일반 문어체 글쓰기 기준이라 자막 고유의 문장부호 관례(작업 기본 원칙 문서, 마침표 미사용 등)와는 별개로 봐야 함
- **보조 용언 붙임 허용 오탐 제거 (③, 구현 완료)**: 제47항에 따르면 대부분의 본용언+보조용언 띄어쓰기는 "붙임 허용"(붙여도, 띄어도 다 맞음)인데, `kiwi.space()`는 그중 하나의 형태만 강제로 제안해서("할만하다"→"할 만하다") 이미 올바른 표기까지 "확인 필요"로 오탐하고 있었다. `_normalize_aux_verb_spacing()`으로 두 가지 붙임 허용 패턴(본용언-아/어+보조용언 VX; 관형사형+의존명사(만/듯/척/체/법/양/성/직)+하다/싶다)을 감지해 kiwi 제안을 원문 형태로 되돌림. 3음절 이상 사전 등재 합성어 본용언 뒤(예: 덤벼들어보아라)는 항상 띄움 예외라 건드리지 않음 — kiwi가 이미 정확히 처리하고 있었음.
  - 구현 중 발견한 함정: kiwi 토큰의 `.form`은 표면형과 다를 수 있다(예: '낸다' → 내(VX)+ᆫ다(EF), 받침이 다음 형태소로 넘어감). 원문 슬라이싱은 반드시 `token.start`/`.len` 기반 표면 텍스트를 써야 하고 `.form`을 문자열 재구성에 쓰면 안 됨.

- **확정 오류 패턴 + 제57항 동음이의어 플래그 (②, 구현 완료)**: `common_errors.py`에 두 정적 목록을 정리함 — (1) `ALWAYS_WRONG`: "그리고 나서→그러고 나서", "그리고는→그러고는"처럼 문맥과 무관하게 항상 틀린 관용적 오용 사례, 초코렛→초콜릿과 같은 성격이라 자동 교정. (2) `CONFUSABLE_PAIRS`: 한글 맞춤법 제57항 동음이의어 혼동 쌍(가름/갈음, 반드시/반듯이, 시키다/식히다 등) — 의미가 완전히 다른 별개 단어라 절대 자동 교정하지 않고 등장 시 항상 확인 플래그.
  - 알려진 트레이드오프: `시키다`처럼 아주 흔한 단어가 포함된 쌍은 거의 항상 올바르게 쓰인 경우에도 매번 플래그될 수 있음(노이즈 위험). 일단 제57항 원문 그대로 구현해뒀고, 실제 사용 피드백을 보고 필요하면 흔한 단어는 목록에서 빼는 식으로 조정 예정.
  - **실전 테스트 중 발견한 심각한 버그(수정 완료)**: `correct_loanwords()`가 사전에 이미 등재된 정상 단어까지 구분 없이 kornorms에 조회하고 있어서, 흔한 고유어 "집"(house)이 kornorms의 무관한 외래어 항목("jib"→"지브")과 우연히 겹쳐 "지브"로 잘못 치환되는 사고가 실제로 발생함. `word_exists(token)`이 참이면(이미 정식 사전 표제어) 애초에 외래어 오표기 후보가 아니므로 loanword_fix 검사 자체를 건너뛰도록 수정. 이런 사고를 실제 테스트로 잡아낸 것 자체가, 계속 실전 사례로 검증하는 방식의 가치를 보여줌.
- **차별적 표현 자동 교정 + 일반 순화어 플래그 (④, 구현 완료)**: 차별적·비하적 표현(간질→뇌전증, 벙어리장갑→손모아장갑, 장님→시각장애인 등)은 관례를 따질 문제가 아니라 항상 바꿔야 하므로 `correct_discriminatory_terms()`로 자동 교정한다. 반면 일반 순화어(반팔→반소매, 유모차→유아차)는 취지는 좋지만 관례적 표현(반팔, 유모차)이 실제로는 더 널리 쓰이는 경우가 있어 자동 교정하지 않고 `check_purified_terms()`로 항상 확인 플래그만 한다 — "차별적 표현=항상 자동 교정, 일반 순화어=항상 사람 확인"이라는 원칙을 그대로 코드에 반영.
  - `_apply_replacements()` 헬퍼를 만들어 긴 표현을 먼저 치환하도록 정렬(예: '벙어리장갑'을 '벙어리'보다 먼저 처리) — 그렇지 않으면 짧은 표현이 긴 표현 안에서 먼저 매치되어 '언어장애인장갑' 같은 이중 치환 사고가 생김. `correct_always_wrong()`도 이 헬퍼로 통일.
  - 지금 목록은 학습 자료에서 확인된 소수 예시로 시작한 것이라 규모가 작음 — 향후 국립국어원 순화어 자료를 더 조사해 목록을 확장할 필요 있음 (백로그).
- **바른번역미디어 번역가 교육자료 학습 완료 (2026-07-15)**: 맞춤법/외래어표기법/작업원칙 문서 6종을 정독함. "그리고 나서→그러고 나서" 같은 확정 오류 패턴, 보조 용언 붙여쓰기 분류표(본용언 음절 수 기준), 차별적 표현 순화어(간질→뇌전증 등)가 다음 자동화 후보로 확인됨. 시키다/받다/하다/되다/드리다의 동작성 판단, 잘하다/잘되다류의 의미별 분류는 여전히 자동화 위험 판정(사람 확인 영역 유지).
- **실제 뉴스 기사 텍스트로 검증 중 발견한 심각한 버그(수정 완료)**: `correct_compound_spacing()`이 원래 "명사+명사(NNG+NNG)" 인접 토큰만 필터링한 뒤 그 필터링된 리스트에서 연속 쌍을 잡았는데, 이 방식은 "회전축에 목이"(축-에-목, 조사 "에"로 분리된 서로 다른 단어)를 필터링 후에는 조사가 빠져 "축"+"목"이 연속으로 보이는 바람에 "회전축목이"로 잘못 합쳐버렸다. 두 토큰 사이 원문 간격이 정확히 빈 문자열이거나 공백 하나인지(`text[t1.end:t2.start] in ("", " ")`) 확인하는 로직을 추가해 수정 — 조사가 실제로 끼어 있으면 후보에서 제외됨.
- **실제 블로그 글 텍스트로 검증 중 발견한 오탐(수정 완료)**: 합성어 자동 붙여쓰기가 명사+명사(NNG+NNG) 패턴만 잡고 있어서, "그때"(관형사 '그'+명사 '때', 표준국어대사전에 "그-때" 합성어로 등재)와 "쓴맛"(형용사 어간+관형사형 어미+명사 '쓴'+'맛', "쓴-맛" 합성어로 등재)이 오히려 반대로 "이미 붙어 있는 게 맞는데 kiwi가 갈라놓자고 제안하는" 상황에서 걸러지지 못하고 "확인 필요"로 오탐되었다. `_compound_candidate_spans()`를 (관형사/명사/고유명사)+(명사/고유명사) 패턴과 (동사·형용사 어간+관형사형 어미)+(명사) 패턴까지 포괄하도록 일반화하고, `check_spacing()`의 kiwi 제안 원상복구 로직도 같은 헬퍼를 쓰도록 통일해 해결. 남은 실제 한계(오탐 아님, 사전 자체의 공백): "포스팅"/"조회수"처럼 표준국어대사전·우리말샘 어디에도 등재되지 않은 신조어는 여전히 플래그됨 — 사전 커버리지 자체의 한계이며 코드 버그가 아님. "몇만/몇백만"처럼 수 표기 띄어쓰기(제44항)가 플래그되는 것도 문맥에 따라 갈릴 수 있어 현재로선 정상적인 동작으로 판단.
- **`word_exists()`/`compound_status()`에 우리말샘 병행 확인 추가 (구현 완료, 2026-07-15)**: 표준국어대사전을 1순위로 먼저 확인하고, 없으면 우리말샘(개방형 사전)도 확인해 정답 근거로 함께 쓰도록 확장했다. 구현 중 두 가지 중요한 함정을 발견해 반드시 막아야 했다:
  - **함정 1 — 우리말샘 검색은 "포함" 매치**: `channel.total > 0`만 보고 판단하면 안 된다. 예를 들어 "스노우"로 검색하면 "스노우 체인", "스노우맨"처럼 그 글자를 포함한 여러 단어(구)가 함께 나와 총 건수가 0보다 커진다. 그래서 실제로는 "스노우" 자체가 표제어가 아닌데도 존재하는 것으로 오판될 뻔했다 — 표제어(하이픈/캐럿 제거 후)가 검색어와 **정확히 일치**하는 항목만 인정하도록 수정.
  - **함정 2 (더 심각) — 우리말샘은 "이미 알려진 틀린 표기"도 정식 표제어로 등재해 둠**: "초코렛"을 우리말샘에서 조회하면 정확히 일치하는 표제어가 나오는데, 그 뜻풀이 끝에 "⇒규범 표기는 '초콜릿'이다"라는 안내가 붙어 있다. 즉 이 항목의 존재 자체가 "맞는 표기"라는 뜻이 아니라 "흔히 쓰이지만 틀린 표기, 정답은 따로 있음"이라는 뜻이다. 표제어 일치만으로 `word_exists()`를 True로 반환했다면 "초코렛"이 더 이상 자동 교정되지 않는 심각한 회귀가 생겼을 것 — 정규 테스트(`examples/sample.srt`)의 자동교정 건수가 5건에서 3건으로 줄어드는 것으로 실제로 잡아냈다. `_opendict_item_is_standard()`로 뜻풀이에 "규범 표기는"이 없는 뜻이 하나라도 있어야 표준 표기로 인정하도록 수정해 해결.
  - 이 두 함정 모두 "우리말샘은 국립국어원 공식 자료니 존재하면 곧 정답"이라는 순진한 가정이 틀렸다는 것을 보여준다 — 우리말샘은 표준국어대사전보다 훨씬 포괄적인(비표준 표기·방언·구까지 다 담는) 사전이라, 존재 여부 확인에는 반드시 "정확히 일치" + "비표준 안내가 없음" 두 조건을 함께 걸어야 한다.
- **우리말샘 용례를 헷갈리는 표현/순화어 플래그에 표시 (구현 완료, 2026-07-15)**: 우리말샘 상세보기(view) API(`method=target_code`)가 돌려주는 `example_info`(실제 용례 문장)를 가져오는 `usage_examples()`를 추가하고, `check_confusable_words()`(제57항 동음이의어)와 `check_purified_terms()`(순화어)의 플래그 사유에 관련 단어들의 실제 용례를 함께 붙이도록 했다. 번역가가 리포트만 보고도 각 단어가 실제 문장에서 어떻게 쓰이는지 바로 비교해 판단할 수 있어, 매번 사전을 따로 찾아봐야 하는 수고를 던다. 용례 조회가 실패해도 플래그 자체(사유·제안값)에는 영향이 없도록 예외를 안에서 흡수해 안전하게 구현.
- **조사·어미 결합 띄어쓰기 자동 교정 (③, 구현 완료, 2026-07-15)**: 기존에는 kiwi 제안이 원문과 다르면 성격과 무관하게 무조건 "확인 필요"로 플래그했는데, 이 중 "오늘은날씨가좋네요"처럼 조사(J*)·어미(E*)·서술격조사(VCP)를 앞말에 붙이는 지점은 한글 맞춤법 제41항에 따라 문맥과 무관하게 항상 정답이 하나뿐이라는 걸 사용자가 지적해 발견함. `correct_particle_spacing()`(내부 `_mechanical_respace()`)을 추가해 이 지점만 안전하게 자동 교정하고, 내용어끼리 이어지는 지점(합성어/의존명사/보조용언 후보)은 기존과 동일하게 손대지 않고 플래그만 남긴다.
  - **구현 중 발견한 심각한 버그(수정 완료)**: 처음에는 토큰 표면형을 순서대로 이어붙여 문자열 전체를 재구성했는데, '해'(하+어)처럼 두 형태소가 같은 음절 하나를 공유해 `start` 위치가 겹치는 경우(§5 다른 항목에서도 이미 확인된 kiwi의 특성) 그 음절이 중복 출력되는 사고가 실제로 발생함("한번 해보자" -> "한번 해해 보자"). 토큰을 이어붙이는 대신, 토큰 사이의 "간격"만 원문에서 찾아 필요할 때만 교체하는 방식(위치 기반 부분 치환)으로 수정.
  - **구현 중 발견한 설계 함정 2건(수정 완료)**: (1) 어미(EC) 뒤에 보조용언(VX)이 오는 경우(예: "해보자"의 EC "어"+VX "보")까지 "어미 뒤는 무조건 공백"으로 처리하면 제47항 붙임 허용 형태("해보자")를 "해 보자"로 잘못 갈라놓게 됨 — VX·의존명사(NNB)는 공백을 강제하는 쪽(뒤에 오는 토큰)에서 제외. (2) 관형사형 어미(ETM)는 그 자체로는 항상 앞말에 붙지만, ETM 뒤에 오는 명사와는 합성어를 이룰 수도 있어(예: "쓴"+"맛"="쓴맛") "ETM 뒤는 무조건 공백"이 아니다 — 공백을 강제하는 토큰(앞쪽) 집합에서 ETM을 제외하고, 조사·EF·EC·VCP만 남김. 두 함정 모두 실제 회귀 테스트(할만하다, 쓴맛, 그때 등)로 잡아냄.
- **보조 용언 붙임 허용 -> "원칙(띄어쓰기) 자동 통일"로 정책 변경 (구현 완료, 2026-07-15)**: 기존에는 붙임 허용 구간(제47항)을 "이미 쓰인 형태를 그대로 인정"(둘 다 정답, 손대지 않음)으로 처리했는데, 사용자가 "보조 용언은 띄어씀을 원칙으로 하고, 별도 지시가 없으면 띄어야 한다"고 명확히 함 — 이는 제47항 규정 원문 자체("띄어 씀을 원칙으로 하되... 붙여 씀도 허용한다")와도 일치한다. `correct_aux_verb_spacing()`을 새로 추가해 두 붙임 허용 패턴(본용언-아/어+보조용언, 관형사형+의존명사+하다/싶다)에서 붙여 쓴 형태를 띄어 쓴 형태로 자동 교정하는 것으로 기본값을 바꿨다. 아직 "사용자 지시로 붙임을 선택"하는 설정(CLI 옵션 등)은 없음 — 필요해지면 별도로 추가.
  - **구현 중 발견한 버그 3건(모두 회귀 테스트로 잡아 수정 완료)**: (1) 관형사형+의존명사 패턴에서 의존명사 뒤쪽 간격만 띄우고 앞쪽(관형사형-의존명사 사이)은 그대로 뒀더니("아는척 한다"), kiwi가 그 절반만 보고 "하다"를 XSV 대신 VV로 재태깅해 오히려 새 오탐 플래그가 생김 — 앞뒤 두 간격을 모두 원칙대로 띄우도록 수정("아는 척 한다"). (2) `check_spacing()`이 이미 확정된 형태와 kiwi의 재제안을 대조할 때, kiwi가 항상 같은 조합으로 띄어 쓰지 않는다는 걸 놓쳐서("할만하다"에 대해 "할 만하다"라는 제3의 조합을 제안) 자동교정 직후 재플래그되는 사고가 생김 — 미리 계산한 "붙임/뗌" 두 후보 중 하나로 단정하는 대신, 확정된 텍스트와 kiwi 제안 양쪽에서 패턴을 각각 독립적으로 찾아 등장 순서로 짝지어 맞바꾸는 방식(`_aux_verb_pattern_spans()`)으로 교체. (3) 가장 심각한 버그: "-아/어지다"(피동/사동)·"-아/어하다"는 제47항의 "붙임 허용" 대상이 아니라 완전히 별도로 "항상 붙임"이 원칙인 규정인데, 형태가 패턴 1(본용언-아/어+보조용언)과 똑같이 생겨서(예: "전해지다"의 "지"도 kiwi가 VX로 태깅) 같은 패턴으로 오인되어 "전해졌다"가 "전해 졌다"로 잘못 갈라질 뻔함 — 실제 뉴스 텍스트 회귀 테스트로 발견. `nxt.lemma`가 "지다"/"하다"인 경우를 패턴 1에서 제외해 해결.

## 6. 플래그 & 재적용 워크플로우 (구현 완료 — `correct` / `apply-report`)

1. 자막 파일 전체를 처리
2. 확신 가능한 항목은 즉시 자동 교정
3. 모호한 항목은 실행 중 끊기지 않고, 전체 처리가 끝난 뒤 **리포트 파일 하나로 모아서 출력** (라인 번호 + 텍스트 + 참고 근거). 인명·지명처럼 자동 적용은 됐지만 더블체크가 필요한 항목은, 이미 적용된 결과 텍스트와 함께 무엇이 왜 바뀌었는지 참고 근거를 리포트에 남긴다.
4. 사용자가 리포트 파일에 직접 수정값을 입력
5. 프로그램을 다시 실행하면 사용자가 입력한 수정값이 **원본 자막 파일에 자동 반영**

## 9. 향후 확장 로드맵 (v1 이후 — 지금은 구현하지 않음)

지금 만드는 것은 로컬 CLI 도구이지만, 나중에 서비스 형태(웹/백엔드)로 확장할 가능성을 고려해 아래 항목을 로드맵으로 남겨둔다. v1 설계에는 §4의 "아키텍처 원칙"(교정 엔진과 인터페이스 분리)만 반영하고, 아래 기능 자체는 만들지 않는다.

- **로그인/계정**: 사용자별 자막 처리 이력 관리를 위해 필요. 인증 방식(자체 구현 vs OAuth 등)은 미정
- **저장**: 업로드한 자막 파일, 교정 결과, 리포트를 서버에 보관. 저장 기간, 개인정보(영상 내용) 처리 정책 등 별도 검토 필요
- **결제**: 유료 플랜/사용량 과금 여부. 어떤 기능을 유료화할지(예: 처리량, 맞춤법 API 호출 횟수)는 미정
- **도메인별 전문용어 사전**: 의학 용어 사전(KMLE) 등 특정 분야(의학 드라마·다큐멘터리 등) 전문용어 교정. 국립국어원 공식 오픈API가 아닌 민간 사이트라 이용약관 확인이 선행되어야 하고, 일반 한국어 자막이라는 v1 범위를 벗어나므로 v2 이후로 미룸
- 이 로드맵이 구체화되면 별도 PRD로 분리해서 다룬다 (백엔드 아키텍처, 인증, 결제 연동은 이 문서의 범위를 벗어남)
