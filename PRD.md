## 1. 배경 및 목적

기존 한국어 자동 교정 툴 '부산대 맞춤법 검사기'를 쓰면서 답답했던 점은 크게 네 가지였다: **오탐이 지나치게 많았고(정상적인 표현·고유명사까지 오류로 잘못 잡아냄), 왜 틀렸는지 근거 없이 무조건 고치라고 했으며, 그 수정 제안 자체가 무작위로 보일 만큼 일관성이 없어 오히려 그 제안이 새로운 오류처럼 느껴졌고, 한 번에 검사할 수 있는 글자 수에 제한이 있어 긴 문서는 여러 번 나눠 페이지를 넘겨가며 검사해야 했다.**

이 프로젝트는 이 네 가지 문제에 직접 대응한다: 국립국어원의 표준국어대사전·우리말샘·어문 규범을 실시간으로 조회해 판단 근거를 명확히 남기고(플래그 사유에 근거를 항상 명시), 근거가 확실한 항목만 자동으로 고치며, 애매하거나 문맥에 따라 답이 갈리는 항목은 절대 무작위로 고치지 않고 사람에게 확인을 맡기고, 글자 수 제한 없이 문서 파일 하나를 통째로 업로드해 한 번에 처리한다 — 이 중 v1은 **띄어쓰기·맞춤법 교정**에 집중한다.

**목표**: 확률적 추측이 아닌 권위 있는 어문 규범 근거 기반으로, 자동 수정한 항목에 한해 오류 0건을 달성한다. 근거가 불확실한 항목은 절대 함부로 고치지 않고 사람에게 넘긴다.

**대상 사용자**: 전문 번역가. 이들의 반복적인 띄어쓰기·맞춤법 교정 작업 부담을 줄여주는 것이 목적이다. 그래서 오탐(정상인데 틀렸다고 플래그)과 미탐(진짜 오류를 놓침) 둘 다 이 목적에 반한다 — 오탐은 번역가가 불필요하게 감수 시간을 쓰게 만들고, 미탐은 애초에 덜어주려던 수고를 그대로 남긴다. 이 때문에 "확실한 것만 자동 처리, 애매하면 반드시 사람에게 위임"이라는 원칙이 특히 중요하다.

## 2. 범위 (v1)

- 기능: 한국어 텍스트의 **띄어쓰기 교정 + 맞춤법 교정**
- 대상: 일반 텍스트(.txt), MS Word(.docx), 자막(.srt) 등 다양한 형식의 한국어 문서 — 자막 파일도 지원 대상 중 하나일 뿐, 이 도구가 자막 작업만을 위한 것은 아니다
- 대상 언어: **한국어 전용**

### v2로 미룬 항목 (Out of scope)
- 번역 텍스트의 자연스러움·정확성 검증 (원문 대조·번역 스타일 판단이 필요해, 이 도구의 판단 기준인 "권위 있는 규범 근거"만으로는 검증할 방법이 없음 — §13 참고)

## 3. 입력 / 출력

- **입력**: 자막(.srt), 일반 텍스트(.txt), MS Word(.docx) 파일
- **출력**:
  - 교정된 결과 파일 (원본과 동일한 포맷 유지 — .srt는 타임코드까지 그대로 보존)
  - 플래그 리포트 파일 (모호해서 자동 수정하지 않은 항목 목록)

## 4. 사용 형태 및 기술 스택

- **CLI 툴**
- 개발 언어: **Python**
- **CLI 프레임워크: Typer**
  - 최소 2개 이상의 서브커맨드(`correct`: 텍스트 교정 실행, `apply-report`: 사용자가 채운 리포트를 원본에 재적용)가 필요한 구조라 argparse보다 서브커맨드 관리가 쉬움

  - Click 기반이라 안정성은 유지하면서 보일러플레이트가 적음
  - 향후 pip 배포 시 관례를 그대로 따름

### 아키텍처 원칙 (향후 확장 대비)

교정 로직(사전/규범 조회, 판단, 플래그 생성)은 **CLI와 분리된 순수 라이브러리 모듈**로 설계한다. CLI(Typer)는 이 라이브러리를 호출하는 얇은 인터페이스로만 둔다. 이렇게 하면 §9의 백엔드 확장 시 동일한 교정 엔진을 API 서버에서 그대로 재사용할 수 있다 (CLI 코드를 뜯어고칠 필요 없음). v1 시점에 백엔드를 직접 구현하지는 않는다 — 지금 만드는 건 어디까지나 로컬 CLI 도구다.

## 5. 교정 판단 엔진 (핵심 설계)

모호한 문장을 만났을 때 아래 순서로 판단하며, 마지막까지 확신할 수 없으면 **자동 수정하지 않고 그대로 둔 채 플래그**한다 (잘못된 수정이 생기는 것보다 미수정 상태가 낫다는 원칙).

1. **1차 판단 — 규범/사전 근거**
   - 표준국어대사전, 우리말샘 (국립국어원 Open API, `search.do`) — 표제어 존재 여부 확인
   - 표준국어대사전 `search.do` 응답의 `word`/`pos` 필드로 합성어(하이픈 표기, 품사 있음 → 무조건 붙여씀) vs 명사구(캐럿 표기, `pos: "품사 없음"` → 띄어쓰기 원칙·붙임 허용) 구분. kiwi.space()가 놓치는 합성어(예: "노천 카페" → 안 고침)를 사전 근거로 보완해 자동 교정 (`compound_status()`, `correct_compound_spacing()`, 구현 완료)
   - 표준국어대사전 사전 내용 API(`view.do`) — 용례, 일부 단어에 붙은 한글 맞춤법 근거(`norm_info`) 확인 (아직 미구현)
   - 한국어 어문 규범 오픈API(`kornorms/exampleReqList.do`) — 국립국어원이 이미 공식 확정한 외래어·로마자 표기 용례(한글표기 ↔ 원어표기, 인명/지명/일반용어) 조회. 새 API 키 필요
   - 외래어 표기법 규정 원문(국립국어원 어문규범 누리집, 표기 기본 원칙 5개 조항 + 언어별 표기 세칙) — API가 없는 정적 문서. kornorms API에도 없는 완전히 새로운 외래어(아직 공식 표기가 없는 신조어)를 다룰 때만 참고하는 최후 수단. 이마저 애매하면 3차로 넘김
2. **2차 판단 — 온라인가나다 답변 아카이브**
   - 온라인가나다는 실시간 API가 아니라 사람이 직접 답변하는 게시판이므로, 실시간 질의는 하지 않는다
   - 대신 **기존에 공개된 과거 답변 아카이브를 검색/참고**하는 방식으로만 활용
3. **3차 — 그래도 모호하면 사람에게 위임**
   - 자동 수정 금지, 리포트에 플래그

### 작업 원칙: "검증표 먼저, 코드는 나중" (2026-07-17 확정)

`correct_aux_verb_spacing()` 패턴 2("할만하다"류)에서 코드를 먼저 짜고 발견되는 예외를 그때그때 패치하는 방식으로 진행하다가, 사용자가 실제 우리말샘 용례로 지적한 오류("할만하다"를 "할 만 하다"로 잘못 고침)를 두 번이나 겪었다. 원인은 "규칙이 이럴 것이다"라는 가정으로 먼저 코드를 짜고 나서 사전으로 검증한 순서 자체에 있었다.

**앞으로 규칙(특히 예외가 있을 수 있는 문법 패턴)을 새로 추가할 때는 반드시 이 순서를 지킨다:**
1. 관련된 모든 경우의 수(해당 패턴에 들어갈 수 있는 표현 전부, 위 표 §5-1/2/3처럼 표준국어대사전·우리말샘·kornorms·[[온라인가나다 판례]]까지)를 먼저 사전 API로 직접 조회해 **정답표(어떤 입력 → 어떤 정답)를 만든다.**
2. 그 정답표에 정확히 맞는 코드를 짠다 — 코드가 표를 따라가는 것이지, 표가 코드를 사후 검증하는 게 아니다.
3. 구현 후에도 표에 없던 새 사례가 발견되면, 코드를 임시로 패치하지 말고 그 사례를 먼저 표에 추가(사전 재조회)한 뒤 표에 맞춰 코드를 고친다.

### 알려진 한계 / 유지보수 요구사항

- **인명·지명 자동 반영 여부의 진짜 기준은 "정답이 하나로 일치하느냐" (구현 완료, 이후 아래 항목에서 고유명사 부분만 재수정됨)**: 처음에는 "인명·지명이면 무조건 확인 필요"로 구분했으나 이는 잘못된 기준이었다. 예를 들어 "스노우"는 인명(Snow, Edgar Parks 등) 표기에서도 예외 없이 "스노"가 맞다 — 이건 문맥에 따라 갈리는 게 아니라 국립국어원이 이미 확정한 단일 정답(relate_mark_o에 "(X)"로 명시)이므로, 일반 용어(초코렛→초콜릿)와 똑같이 확인 없이 자동 반영해야 한다. 진짜 확인이 필요한 경우는 **같은 원어에 대해 kornorms에 등재된 정답 자체가 서로 다르게 갈리는 경우**뿐이다 (예: Jeremiah → 예레미야(성경식) vs 제러마이아(현대 인명) 처럼 실제로 다른 관례가 공존). 이 경우는 텍스트만으로는 어느 쪽인지 판단할 수 없고 실제 영상 속 발음을 들어야만 확정 가능하므로, 첫 번째 후보를 적용하되 항상 "확인 필요"로 플래그한다. `loanword_fix()`는 검색된 모든 일치 항목의 교정값이 일치하는지(`foreign_gubun`이 아니라)로 이를 판단한다. 구현 시 추가로 발견한 문제: kornorms의 `korean_mark`는 인명 항목에서 "성, 이름" 전체를 담고 있어, 토큰 하나("스노우")를 통째로 전체 이름("스노, 에드거 파크스")으로 바꿔버리는 오류가 있었다 — `_closest_segment()`로 토큰과 가장 유사한 조각만 추출해 적용하도록 수정함.
- **고유명사(NNP) 외래어 표기는 정답이 하나뿐이어도 절대 자동 반영하지 않고 항상 확인 플래그로 전환 (구현 완료, 2026-07-15)**: 위 원칙("정답이 하나면 자동 반영")은 인명 표기 자체(스노우 기자→스노 기자)에는 맞지만, 같은 표기가 **특정 작품·제품의 고정된 고유 제목**일 경우 위험하다는 것을 사용자가 지적해 발견함 — 예를 들어 실제로 우리말샘에 "스노우맨"(2017년 영화·소설 제목)이 "⇒규범 표기는 '스노맨'이다"라는 안내와 함께 등재돼 있는데, 이 영화의 한국 배급사가 실제로 "스노우맨"이라는 표기를 그대로 제목으로 썼다면(트레이드마크성 고유 표기), 표기 규칙을 강제 적용해 "스노맨"으로 바꾸는 순간 실제 작품명을 훼손하는 오류가 된다. 문제는 kiwi의 NNP 태그만으로는 "인명 표기(규칙을 따라야 함)"와 "고정된 작품 제목(규칙과 무관하게 그 자체가 정답)"을 구분할 신호가 전혀 없다는 것 — 텍스트 자체에 이 둘을 가를 근거가 없으므로, 고유명사 전체를 자동 교정 대상에서 완전히 제외하고 항상 사람이 최종 승인하도록 정책을 바꿨다(`correct_loanwords()`가 NNP 매치는 `proper_noun_suggestions`로 분리해 텍스트를 건드리지 않고 `suggested_fix`만 채운 플래그로 남김). 트레이드오프: 자동화 비율은 줄지만(예: "스노우 기자"도 이제 매번 확인이 필요), "절대 오류 없는" 원칙상 안전이 우선이라는 사용자 결정에 따른 것.
  - **문맥 기반 자동판별을 시도하지 않은 이유**: "주변에 '영화', '개봉' 같은 단서가 있으면 고유 제목으로 보고 자동 반영을 건너뛴다" 같은 문맥 휴리스틱도 검토했으나 채택하지 않았다. 그런 단서 없이도 작품을 언급하는 문장이 흔하고("우리는 그거 봤어"), 반대로 사람 얘기를 하면서 우연히 그런 단어가 근처에 나올 수도 있어 오탐/누락 위험이 여전하다. 더 근본적으로, 이 프로젝트의 원칙(§5, 확률적 추측이 아닌 권위 있는 규범 근거 기반) 자체가 "이 표기가 특정 작품의 고정 제목인지"를 판정해 줄 국립국어원 데이터 소스가 없다는 뜻이므로, 휴리스틱을 만들어도 그 자체가 또 다른 확률적 추측일 뿐이다. 따라서 사람 확인이 유일하게 정당한 해법이다.
- **규칙 기반 도출과 관용(예외)의 구분**: "제러마이아"처럼 표기 세칙(예: 어말 스와(/ə/) → '아')으로 체계적으로 도출되는 일반 용어 표기와, "머라이어"(Mariah Carey)처럼 규칙이 아니라 유명세로 굳어진 순수 관용 예외는 서로 다르다. kornorms 선례 검색은 이미 등재된 관용 예외(일반 용어)는 잘 잡지만, **아직 kornorms에 없는 새 일반 용어가 규칙으로는 도출 가능한 경우**를 처리하려면 별도의 표기 세칙 규칙 엔진(IPA 기반)이 필요하다 — 아직 미구현
- **표기 세칙 규칙 엔진의 범위**: 외래어 표기는 영어만이 아니라 전 세계 모든 언어의 발음을 한글로 표기할 수 있어야 한다 (국립국어원 표기 세칙도 영어·독일어·프랑스어·에스파냐어·이탈리아어·일본어·중국어 등 20개 언어를 다룬다). 따라서 이 규칙 엔진은 영어 하나만 다루는 것보다 훨씬 큰 작업이며, v1 범위를 넘어선다 — v2 이후 백로그로 명확히 분리한다
- **규정 개정에 따른 유지보수**: 한국어 어문 규정(특히 외래어 표기법)은 종종 개정된다(예: 1986년 최초 제정 → 2017년 개정). kornorms API(선례 데이터베이스)는 국립국어원이 직접 관리·갱신하므로 우리 쪽 유지보수 부담이 적지만, 만약 표기 세칙을 직접 규칙 엔진으로 구현한다면 개정 여부를 주기적으로 확인하고 코드를 갱신하는 책임이 생긴다. 이 때문에 규칙을 직접 구현하기보다 kornorms 같은 국립국어원 관리 API에 최대한 의존하는 쪽을 우선한다.
  - **운영 원칙(2026-07-15 추가)**: 이 프로젝트로 다시 작업을 시작할 때마다(세션 시작 시), 국립국어원 웹 검색으로 한글 맞춤법/외래어 표기법/표준어 규정에 최근 개정이 있었는지 가볍게 확인하는 것을 관례로 삼는다. AI 세션은 상시 백그라운드로 돌지 않으므로 "주기적 자동 감시"는 불가능하지만, 매 작업 세션 시작 시 확인하는 것으로 사실상의 주기적 점검을 대신한다. 2026-07-15 확인 결과: 웹 검색상 2025~2026년 사이의 한글 맞춤법·외래어 표기법 개정 공지는 발견되지 않음(외래어 표기법은 1986년 제정, 2017년 개정 이후 그대로로 보임) — 현재 구현이 최신 규정과 어긋난다는 증거 없음.
- **고유명사(인명)를 맞춤법 오류로 오탐하는 문제 (구현 완료)**: 기존 도구(부산대 맞춤법 검사기)가 사람 이름을 이상하게 바꾸거나 잘못 처리한다는 지적이 있어 우리 코드를 실제로 점검했더니, 같은 종류의 버그가 있었다 — "민준"처럼 정상적인 한국인 이름이 표준국어대사전 표제어가 아니라는 이유만으로 "사전에 없는 단어"로 오탐됐다. 원인은 맞춤법 검사(§5-2)에서 고유명사(NNP) 태그를 일반명사와 동일하게 사전 대조했기 때문. 고유명사를 검사 대상에서 제외해 수정함 (`engine.py`의 `_SPELLING_CHECK_TAGS`는 NNG/VV/VA만 포함, NNP 제외).
- **인명·지명 검증은 원어(로마자) 없이는 근본적 한계가 있음 (미구현, 백로그)**: 자막에 이미 한글로 옮겨진 고유명사만으로는, 그게 맞는 표기인지 원어 자체를 모르니 검증이 불가능한 경우가 많다. 해결 방향: 번역가가 고유명사를 원어(로마자) 그대로 남겨두면, 프로그램이 이를 자동 감지(한글과 다른 토큰 태그)해 kornorms에 원어로 직접 조회 후 공식 표기로 치환한다. 이미 한글로 쓰인 고유명사는 번역가의 의도된 결정으로 보고 건드리지 않는다. 사용자 안내문("고유명사는 원지어를 그대로 두면 정확한 결과를 얻을 수 있음")과 함께, 실제 동작은 텍스트 형태로 자동 판별되게 해 사용자가 안내를 놓쳐도 최소한 결과가 일관되게 한다.
- **합성어 자동 붙여쓰기 (구현 완료)**: 표준국어대사전 표제어는 합성어를 하이픈("노천-카페")으로, 명사구를 캐럿("예방^접종")으로 표기하며 `pos` 필드로도 구분된다(품사 있음=합성어/무조건 붙임, `pos: "품사 없음"`=명사구/붙임 허용). `kiwi.space()`는 이런 합성어를 놓치는 경우가 있어("노천 카페" 원문을 안 고침), 사전 조회로 보완해 자동 교정한다. 부작용으로 kiwi가 이미 확정된 합성어를 다시 갈라놓자고 제안하는 모순이 있었는데, `check_spacing()`에서 확정된 합성어 부분만 원상복구해 해결함.
- **국립국어원 "한국어 어문 규범"(한글 맞춤법 해설서, 96쪽) 정독 완료 (2026-07-15)**: kiwi는 보조 도구일 뿐이고 국립국어원 자료가 항상 우선한다는 원칙에 따라, 규정 원문+해설을 직접 확인함. 주요 발견:
  - **제42항(의존명사)**: "들/뿐/대로/만큼/만/지/차(次)/판" 8개 단어가 조사·접미사(체언 뒤, 붙임)와 의존명사(용언 관형사형 뒤, 띄움)로 갈리는 공식 목록. "대로/만큼"은 `kiwi.space()`가 이미 정확히 처리함을 확인했고(§5 알려진 한계, 이전 세션), 나머지 6개(들/뿐/만/지/차/판)는 아직 검증 안 함 — 다음 검증 대상
  - **제30항(사이시옷)**: 합성어+앞말 모음+뒷말 된소리/ㄴ 첨가+고유어 최소 1개 포함(외래어 있으면 불가) 조건. 이미 구현한 `compound_status()`(사전 하이픈 판정)로 사실상 커버됨 — 사이시옷 여부는 사전 표제어 형태 자체에 이미 반영되어 있으므로 별도 규칙 엔진 불필요
  - **제47항(보조 용언)**: 훈련 자료에서 배운 붙임 허용/항상 붙임/항상 띄움 분류표의 공식 근거 확인 + 세부 보강 — 2음절 예외는 본용언의 "활용형" 음절 수 기준(원형 음절 수가 아님), '예뻐하다'는 2023년에 표준국어대사전 표제어로 등재되어 항상 붙임, '도와드리다'는 '도와주다'가 사전 표제어인 것에 맞춰 항상 붙임
  - **제57항(동음이의어 혼동 쌍)**: 가름/갈음, 거름/걸음, 반드시/반듯이, 시키다/식히다, 붙이다/부치다, 목거리/목걸이 등 국립국어원이 직접 지정한 "자주 헷갈리는 말" 공식 목록 확보 — ②(확정 오류 패턴) 작업의 유력 후보. 다만 이들은 의미가 완전히 다른 별개의 단어라 어느 쪽이 맞는지는 문맥 의미 판단이 필요함 (예: "그리고 나서→그러고 나서"처럼 예외 없이 틀린 경우와는 다름) — 자동 교정 대상이 아니라 "이 단어 자주 헷갈림, 확인 필요" 플래그 후보로 적합
  - 문장 부호 규정(마침표~쌍점 등)도 확인했으나, 이는 일반 문어체 글쓰기 기준이라 자막 고유의 문장부호 관례(작업 기본 원칙 문서, 마침표 미사용 등)와는 별개로 봐야 함
- **보조 용언 붙임 허용 오탐 제거 (③, 구현 완료)**: 제47항에 따르면 대부분의 본용언+보조용언 띄어쓰기는 "붙임 허용"(붙여도, 띄어도 다 맞음)인데, `kiwi.space()`는 그중 하나의 형태만 강제로 제안해서("할만하다"→"할 만하다") 이미 올바른 표기까지 "확인 필요"로 오탐하고 있었다. `_normalize_aux_verb_spacing()`으로 두 가지 붙임 허용 패턴(본용언-아/어+보조용언 VX; 관형사형+의존명사(만/듯/척/체/법/양/성/직)+하다/싶다)을 감지해 kiwi 제안을 원문 형태로 되돌림. 3음절 이상 사전 등재 합성어 본용언 뒤(예: 덤벼들어보아라)는 항상 띄움 예외라 건드리지 않음 — kiwi가 이미 정확히 처리하고 있었음.
  - 구현 중 발견한 함정: kiwi 토큰의 `.form`은 표면형과 다를 수 있다(예: '낸다' → 내(VX)+ᆫ다(EF), 받침이 다음 형태소로 넘어감). 원문 슬라이싱은 반드시 `token.start`/`.len` 기반 표면 텍스트를 써야 하고 `.form`을 문자열 재구성에 쓰면 안 됨.
  - 남은 한계: '도와드리다'(도와주다가 사전 표제어라 항상 붙임이어야 함)처럼 개별 단어 단위의 특정 예외는 아직 못 잡음 — 본용언(돕다)이 2음절이라 일반적으로는 붙임 허용 케이스로 분류되기 때문. 위험하지는 않음(그냥 불필요한 플래그 하나가 남을 뿐), 우선순위 낮은 백로그.
- **확정 오류 패턴 + 제57항 동음이의어 플래그 (②, 구현 완료)**: `common_errors.py`에 두 정적 목록을 정리함 — (1) `ALWAYS_WRONG`: "그리고 나서→그러고 나서", "그리고는→그러고는"처럼 문맥과 무관하게 항상 틀린 관용적 오용 사례, 초코렛→초콜릿과 같은 성격이라 자동 교정. (2) `CONFUSABLE_PAIRS`: 한글 맞춤법 제57항 동음이의어 혼동 쌍(가름/갈음, 반드시/반듯이, 시키다/식히다 등) — 의미가 완전히 다른 별개 단어라 절대 자동 교정하지 않고 등장 시 항상 확인 플래그.
  - 알려진 트레이드오프: `시키다`처럼 아주 흔한 단어가 포함된 쌍은 거의 항상 올바르게 쓰인 경우에도 매번 플래그될 수 있음(노이즈 위험). 일단 제57항 원문 그대로 구현해뒀고, 실제 사용 피드백을 보고 필요하면 흔한 단어는 목록에서 빼는 식으로 조정 예정.
  - **실전 테스트 중 발견한 심각한 버그(수정 완료)**: `correct_loanwords()`가 사전에 이미 등재된 정상 단어까지 구분 없이 kornorms에 조회하고 있어서, 흔한 고유어 "집"(house)이 kornorms의 무관한 외래어 항목("jib"→"지브")과 우연히 겹쳐 "지브"로 잘못 치환되는 사고가 실제로 발생함. `word_exists(token)`이 참이면(이미 정식 사전 표제어) 애초에 외래어 오표기 후보가 아니므로 loanword_fix 검사 자체를 건너뛰도록 수정. 이런 사고를 실제 테스트로 잡아낸 것 자체가, 계속 실전 사례로 검증하는 방식의 가치를 보여줌.
- **차별적 표현 자동 교정 + 일반 순화어 플래그 (④, 구현 완료)**: 차별적·비하적 표현(간질→뇌전증, 벙어리장갑→손모아장갑, 장님→시각장애인 등)은 관례를 따질 문제가 아니라 항상 바꿔야 하므로 `correct_discriminatory_terms()`로 자동 교정한다. 반면 일반 순화어(반팔→반소매, 유모차→유아차)는 취지는 좋지만 관례적 표현(반팔, 유모차)이 실제로는 더 널리 쓰이는 경우가 있어 자동 교정하지 않고 `check_purified_terms()`로 항상 확인 플래그만 한다 — "차별적 표현=항상 자동 교정, 일반 순화어=항상 사람 확인"이라는 원칙을 그대로 코드에 반영.
  - `_apply_replacements()` 헬퍼를 만들어 긴 표현을 먼저 치환하도록 정렬(예: '벙어리장갑'을 '벙어리'보다 먼저 처리) — 그렇지 않으면 짧은 표현이 긴 표현 안에서 먼저 매치되어 '언어장애인장갑' 같은 이중 치환 사고가 생김. `correct_always_wrong()`도 이 헬퍼로 통일.
  - 지금 목록은 학습 자료에서 확인된 소수 예시로 시작한 것이라 규모가 작음 — 향후 국립국어원 순화어 자료를 더 조사해 목록을 확장할 필요 있음 (백로그).
- **바른번역미디어 번역가 교육자료 학습 완료 (2026-07-15)**: 맞춤법/외래어표기법/작업원칙 문서 6종을 정독함. "그리고 나서→그러고 나서" 같은 확정 오류 패턴, 보조 용언 붙여쓰기 분류표(본용언 음절 수 기준), 차별적 표현 순화어(간질→뇌전증 등)가 다음 자동화 후보로 확인됨. 시키다/받다/하다/되다/드리다의 동작성 판단, 잘하다/잘되다류의 의미별 분류는 여전히 자동화 위험 판정(사람 확인 영역 유지).
- **실제 뉴스 기사 텍스트로 검증 중 발견한 심각한 버그(수정 완료)**: `correct_compound_spacing()`이 원래 "명사+명사(NNG+NNG)" 인접 토큰만 필터링한 뒤 그 필터링된 리스트에서 연속 쌍을 잡았는데, 이 방식은 "회전축에 목이"(축-에-목, 조사 "에"로 분리된 서로 다른 단어)를 필터링 후에는 조사가 빠져 "축"+"목"이 연속으로 보이는 바람에 "회전축목이"로 잘못 합쳐버렸다. 두 토큰 사이 원문 간격이 정확히 빈 문자열이거나 공백 하나인지(`text[t1.end:t2.start] in ("", " ")`) 확인하는 로직을 추가해 수정 — 조사가 실제로 끼어 있으면 후보에서 제외됨.
- **실제 블로그 글 텍스트로 검증 중 발견한 오탐(수정 완료)**: 합성어 자동 붙여쓰기가 명사+명사(NNG+NNG) 패턴만 잡고 있어서, "그때"(관형사 '그'+명사 '때', 표준국어대사전에 "그-때" 합성어로 등재)와 "쓴맛"(형용사 어간+관형사형 어미+명사 '쓴'+'맛', "쓴-맛" 합성어로 등재)이 오히려 반대로 "이미 붙어 있는 게 맞는데 kiwi가 갈라놓자고 제안하는" 상황에서 걸러지지 못하고 "확인 필요"로 오탐되었다. `_compound_candidate_spans()`를 (관형사/명사/고유명사)+(명사/고유명사) 패턴과 (동사·형용사 어간+관형사형 어미)+(명사) 패턴까지 포괄하도록 일반화하고, `check_spacing()`의 kiwi 제안 원상복구 로직도 같은 헬퍼를 쓰도록 통일해 해결. 남은 실제 한계(오탐 아님, 사전 자체의 공백): "포스팅"/"조회수"처럼 표준국어대사전·우리말샘 어디에도 등재되지 않은 신조어는 여전히 플래그됨 — 사전 커버리지 자체의 한계이며 코드 버그가 아님. "몇만/몇백만"처럼 수 표기 띄어쓰기(제44항)가 플래그되는 것도 문맥에 따라 갈릴 수 있어 현재로선 정상적인 동작으로 판단.
- **`word_exists()`/`compound_status()`에 우리말샘 병행 확인 추가 (구현 완료, 2026-07-15)**: 표준국어대사전을 1순위로 먼저 확인하고, 없으면 우리말샘(개방형 사전)도 확인해 정답 근거로 함께 쓰도록 확장했다. 구현 중 두 가지 중요한 함정을 발견해 반드시 막아야 했다:
  - **함정 1 — 우리말샘 검색은 "포함" 매치**: `channel.total > 0`만 보고 판단하면 안 된다. 예를 들어 "스노우"로 검색하면 "스노우 체인", "스노우맨"처럼 그 글자를 포함한 여러 단어(구)가 함께 나와 총 건수가 0보다 커진다. 그래서 실제로는 "스노우" 자체가 표제어가 아닌데도 존재하는 것으로 오판될 뻔했다 — 표제어(하이픈/캐럿 제거 후)가 검색어와 **정확히 일치**하는 항목만 인정하도록 수정.
  - **함정 2 (더 심각) — 우리말샘은 "이미 알려진 틀린 표기"도 정식 표제어로 등재해 둠**: "초코렛"을 우리말샘에서 조회하면 정확히 일치하는 표제어가 나오는데, 그 뜻풀이 끝에 "⇒규범 표기는 '초콜릿'이다"라는 안내가 붙어 있다. 즉 이 항목의 존재 자체가 "맞는 표기"라는 뜻이 아니라 "흔히 쓰이지만 틀린 표기, 정답은 따로 있음"이라는 뜻이다. 표제어 일치만으로 `word_exists()`를 True로 반환했다면 "초코렛"이 더 이상 자동 교정되지 않는 심각한 회귀가 생겼을 것 — 정규 테스트(`examples/sample.srt`)의 자동교정 건수가 5건에서 3건으로 줄어드는 것으로 실제로 잡아냈다. `_opendict_item_is_standard()`로 뜻풀이에 "규범 표기는"이 없는 뜻이 하나라도 있어야 표준 표기로 인정하도록 수정해 해결.
  - 이 두 함정 모두 "우리말샘은 국립국어원 공식 자료니 존재하면 곧 정답"이라는 순진한 가정이 틀렸다는 것을 보여준다 — 우리말샘은 표준국어대사전보다 훨씬 포괄적인(비표준 표기·방언·구까지 다 담는) 사전이라, 존재 여부 확인에는 반드시 "정확히 일치" + "비표준 안내가 없음" 두 조건을 함께 걸어야 한다.
- **우리말샘 용례를 헷갈리는 표현/순화어 플래그에 표시 (구현 완료, 2026-07-15)**: 우리말샘 상세보기(view) API(`method=target_code`)가 돌려주는 `example_info`(실제 용례 문장)를 가져오는 `usage_examples()`를 추가하고, `check_confusable_words()`(제57항 동음이의어)와 `check_purified_terms()`(순화어)의 플래그 사유에 관련 단어들의 실제 용례를 함께 붙이도록 했다. 번역가가 리포트만 보고도 각 단어가 실제 문장에서 어떻게 쓰이는지 바로 비교해 판단할 수 있어, 매번 사전을 따로 찾아봐야 하는 수고를 던다. 용례 조회가 실패해도 플래그 자체(사유·제안값)에는 영향이 없도록 예외를 안에서 흡수해 안전하게 구현.
- **조사·어미 결합 띄어쓰기 자동 교정 (③, 구현 완료, 2026-07-15)**: 기존에는 kiwi 제안이 원문과 다르면 성격과 무관하게 무조건 "확인 필요"로 플래그했는데, 이 중 "오늘은날씨가좋네요"처럼 조사(J*)·어미(E*)·서술격조사(VCP)를 앞말에 붙이는 지점은 한글 맞춤법 제41항에 따라 문맥과 무관하게 항상 정답이 하나뿐이라는 걸 사용자가 지적해 발견함. `correct_particle_spacing()`(내부 `_mechanical_respace()`)을 추가해 이 지점만 안전하게 자동 교정하고, 내용어끼리 이어지는 지점(합성어/의존명사/보조용언 후보)은 기존과 동일하게 손대지 않고 플래그만 남긴다.
  - **구현 중 발견한 심각한 버그(수정 완료)**: 처음에는 토큰 표면형을 순서대로 이어붙여 문자열 전체를 재구성했는데, '해'(하+어)처럼 두 형태소가 같은 음절 하나를 공유해 `start` 위치가 겹치는 경우(§5 다른 항목에서도 이미 확인된 kiwi의 특성) 그 음절이 중복 출력되는 사고가 실제로 발생함("한번 해보자" -> "한번 해해 보자"). 토큰을 이어붙이는 대신, 토큰 사이의 "간격"만 원문에서 찾아 필요할 때만 교체하는 방식(위치 기반 부분 치환)으로 수정.
  - **구현 중 발견한 설계 함정 2건(수정 완료)**: (1) 어미(EC) 뒤에 보조용언(VX)이 오는 경우(예: "해보자"의 EC "어"+VX "보")까지 "어미 뒤는 무조건 공백"으로 처리하면 제47항 붙임 허용 형태("해보자")를 "해 보자"로 잘못 갈라놓게 됨 — VX·의존명사(NNB)는 공백을 강제하는 쪽(뒤에 오는 토큰)에서 제외. (2) 관형사형 어미(ETM)는 그 자체로는 항상 앞말에 붙지만, ETM 뒤에 오는 명사와는 합성어를 이룰 수도 있어(예: "쓴"+"맛"="쓴맛") "ETM 뒤는 무조건 공백"이 아니다 — 공백을 강제하는 토큰(앞쪽) 집합에서 ETM을 제외하고, 조사·EF·EC·VCP만 남김. 두 함정 모두 실제 회귀 테스트(할만하다, 쓴맛, 그때 등)로 잡아냄.
- **보조 용언 붙임 허용 -> "원칙(띄어쓰기) 자동 통일"로 정책 변경 (구현 완료, 2026-07-15)**: 기존에는 붙임 허용 구간(제47항)을 "이미 쓰인 형태를 그대로 인정"(둘 다 정답, 손대지 않음)으로 처리했는데, 사용자가 "보조 용언은 띄어씀을 원칙으로 하고, 별도 지시가 없으면 띄어야 한다"고 명확히 함 — 이는 제47항 규정 원문 자체("띄어 씀을 원칙으로 하되... 붙여 씀도 허용한다")와도 일치한다. `correct_aux_verb_spacing()`을 새로 추가해 두 붙임 허용 패턴(본용언-아/어+보조용언, 관형사형+의존명사+하다/싶다)에서 붙여 쓴 형태를 띄어 쓴 형태로 자동 교정하는 것으로 기본값을 바꿨다. 아직 "사용자 지시로 붙임을 선택"하는 설정(CLI 옵션 등)은 없음 — 필요해지면 별도로 추가.
  - **구현 중 발견한 버그 3건(모두 회귀 테스트로 잡아 수정 완료)**: (1) 관형사형+의존명사 패턴에서 의존명사 뒤쪽 간격만 띄우고 앞쪽(관형사형-의존명사 사이)은 그대로 뒀더니("아는척 한다"), kiwi가 그 절반만 보고 "하다"를 XSV 대신 VV로 재태깅해 오히려 새 오탐 플래그가 생김 — 앞뒤 두 간격을 모두 원칙대로 띄우도록 수정("아는 척 한다"). (2) `check_spacing()`이 이미 확정된 형태와 kiwi의 재제안을 대조할 때, kiwi가 항상 같은 조합으로 띄어 쓰지 않는다는 걸 놓쳐서("할만하다"에 대해 "할 만하다"라는 제3의 조합을 제안) 자동교정 직후 재플래그되는 사고가 생김 — 미리 계산한 "붙임/뗌" 두 후보 중 하나로 단정하는 대신, 확정된 텍스트와 kiwi 제안 양쪽에서 패턴을 각각 독립적으로 찾아 등장 순서로 짝지어 맞바꾸는 방식(`_aux_verb_pattern_spans()`)으로 교체. (3) 가장 심각한 버그: "-아/어지다"(피동/사동)·"-아/어하다"는 제47항의 "붙임 허용" 대상이 아니라 완전히 별도로 "항상 붙임"이 원칙인 규정인데, 형태가 패턴 1(본용언-아/어+보조용언)과 똑같이 생겨서(예: "전해지다"의 "지"도 kiwi가 VX로 태깅) 같은 패턴으로 오인되어 "전해졌다"가 "전해 졌다"로 잘못 갈라질 뻔함 — 실제 뉴스 텍스트 회귀 테스트로 발견. `nxt.lemma`가 "지다"/"하다"인 경우를 패턴 1에서 제외해 해결.

## 9. 향후 확장 로드맵 (v1 이후)

지금 만드는 것은 로컬 CLI 도구이지만, 나중에 서비스 형태(웹/백엔드)로 확장할 가능성을 고려해 아래 항목을 로드맵으로 남겨둔다. v1 설계에는 §4의 "아키텍처 원칙"(교정 엔진과 인터페이스 분리)만 반영하고, 아래 기능 자체는 만들지 않는다.

- **로그인/계정**: 사용자별 처리 이력 관리를 위해 필요. 인증 방식(자체 구현 vs OAuth 등)은 미정. 아직 미구현 — §11의 저장은 로그인 없이 "id를 아는 사람은 누구나 그 결과를 볼 수 있다"는 수준(짧은 링크 공유 정도의 보안)이다. 여러 사용자를 구분해서 관리하려면 이 항목이 선행되어야 함.
- **저장 — 구현 완료 (2026-07-16, §11 참고)**: 업로드한 문서의 교정 결과와 플래그 리포트를 Supabase(Postgres)에 저장. 저장 기간·개인정보(문서 내용) 처리 정책은 아직 정하지 않음(무기한 보관 중) — 실 서비스로 나갈 경우 반드시 결정 필요.
- **결제**: 유료 플랜/사용량 과금 여부. 어떤 기능을 유료화할지(예: 처리량, 맞춤법 API 호출 횟수)는 미정
- **도메인별 전문용어 사전**: 의학 용어 사전(KMLE) 등 특정 분야(의학·법률 등) 전문용어 교정. 국립국어원 공식 오픈API가 아닌 민간 사이트라 이용약관 확인이 선행되어야 하고, 일반 한국어 텍스트라는 v1 범위를 벗어나므로 v2 이후로 미룸
- 이 로드맵이 구체화되면 별도 PRD로 분리해서 다룬다 (백엔드 아키텍처, 인증, 결제 연동은 이 문서의 범위를 벗어남)

## 11. 웹 API / 저장 (구현 완료, 2026-07-16)

수업 과제("내 서비스에 백엔드 붙이기")로 §9의 "저장" 항목을 실제로 구현했다. 새 교정 로직은 추가하지 않고, §4의 아키텍처 원칙대로 기존 엔진을 그대로 호출만 한다.

- **구성**: `subtitle_corrector/api.py`(FastAPI) — `POST /api/correct`(`.srt` 업로드 → `parsers.parse_srt` → `engine.correct_entries` → 결과 저장) / `GET /api/reports/{id}`(저장된 결과 재조회). `static/index.html`이 업로드 화면 겸 결과 화면(`?id=`로 새로고침·다른 기기에서도 결과 유지).
- **저장소**: Supabase(Postgres), `subtitle_corrector/store.py`가 REST(PostgREST)를 `requests`로 직접 호출. 테이블 스키마는 `supabase_schema.sql`.
- **보안 설계 — 이 프로젝트가 선택한 구조는 강의 자료의 "브라우저가 Supabase에 직접 접근" 패턴이 아니다**: 브라우저는 이 FastAPI 서버에만 요청하고, Supabase에는 절대 직접 접근하지 않는다. 따라서 `SUPABASE_SERVICE_KEY`(관리자 키, RLS 무시)를 서버 쪽 환경변수로만 갖고 있어도 안전하며, `reports` 테이블은 RLS를 켜두고 정책을 하나도 만들지 않아 anon(공개) 키로는 어떤 요청도 거부된다 — 서버가 유일한 접근 경로.
- **비밀값**: `STDICT_API_KEY`/`OPENDICT_API_KEY`/`KORNORMS_API_KEY`(기존)+`SUPABASE_URL`/`SUPABASE_SERVICE_KEY`(신규), 전부 `.env`(로컬)/Render 환경변수(배포)로만 존재. `.env.example` 참고.
- **배포**: Render(무료 웹서비스)에 FastAPI 그대로 배포. 계정 생성·대시보드 설정은 사용자가 직접 해야 해서(AI가 대신 로그인 불가) `DEPLOY.md`에 단계별로 정리해둠 — **이 세션에서는 로컬 테스트(스텁 저장소로 엔진 연동 검증)까지만 완료, 실제 Supabase 프로젝트 생성/Render 배포는 아직 안 됨.**
- **테스트**: `examples/sample.srt`로 FastAPI 라우트(200/404, 파일 형식 검증)와 엔진 연동을 확인함(`store.save_report`/`get_report`를 스텁으로 교체해 실제 Supabase 계정 없이 검증). 실제 Supabase 연동은 사용자가 계정을 만든 뒤 `DEPLOY.md` 2단계(로컬 테스트)로 재확인 필요.
- **실제 Supabase 연동 확인 완료 (2026-07-16)**: 사용자가 Supabase 프로젝트를 만들고 `.env`에 키를 넣은 뒤, 로컬 uvicorn 서버로 실제 저장/재조회까지 확인함. Render 배포는 아직 안 함.
- **실제 긴 SDH 자막(336줄)으로 검증하며 발견한 버그 4건, 전부 수정 완료 (2026-07-16)**: `examples/sample.srt`(전부 한 줄짜리 대사)로는 안 걸리던 버그들이 줄바꿈+방언+구어체가 섞인 실제 드라마 자막에서 한꺼번에 드러남.
  1. `/api/correct`가 `async def`라 교정 중(사전 API 순차 호출)에는 이벤트 루프 전체가 막혀 다른 요청도 응답 못 받았음 -> `sync def`로 변경, FastAPI가 자동으로 스레드풀 실행.
  2. 우리말샘 서버가 느릴 때(`requests.exceptions.ReadTimeout`) 처리되지 않은 예외로 요청 전체가 500으로 죽었음 -> `search_stdict`/`search_opendict`/`search_kornorms`에서 `requests.RequestException`을 "찾지 못함"과 동일하게 처리(`usage_examples()`의 기존 원칙과 동일). 같은 단어 반복 조회를 없애는 `@lru_cache`도 추가.
  3. **`_mechanical_respace()`가 두 줄 자막을 한 줄로 합쳐버림** — "어절 완결 지점은 항상 공백"이라는 규칙을 적용할 때 원래 간격이 줄바꿈이어도 그냥 공백으로 덮어씀. 336줄 중 54곳에서 발생 확인, 간격에 `"\n"`이 있으면 건드리지 않도록 수정 후 재검증 -> 0건.
  4. `.txt` 일반 텍스트 지원 추가(`parsers.parse_plain_text`/`write_plain_text`) — 교정 엔진 자체는 자막 전용이 아니라 한국어 텍스트 한 줄을 다루는 범용 엔진이므로, `.srt`(타임코드 보존)와 `.txt`(줄 구성만 보존) 둘 다 같은 엔진을 재사용. CLI(`main.py`)/웹 API 모두 적용.
  - **이 세션에서 발견했지만 아직 안 고친 것 (다음 작업 후보)**: (a) 고유명사(예: "연실")를 kiwi가 몰라서 "연 실"로 잘못 쪼개자고 `check_spacing()`이 계속 제안함 — 맞춤법 검사에서는 이미 NNP를 제외했지만 띄어쓰기 검사에서는 아직 제외 안 함. (b) `[이름/상황]`(SDH 브래킷 태그) 표기를 `kiwi.space()`가 일반 문장처럼 분석해 "/" 뒤에 공백을 넣자고 계속 제안함. (c) "작게"(작다의 활용형)처럼 문맥이 고립된 브래킷 안에서는 kiwi가 이상하게 태깅해 `check_spelling()`이 "사전에 없는 단어"로 오탐. (d) `_mechanical_respace()`/`correct_aux_verb_spacing()`이 "뭐라는"->"뭐라 는", "준다잖아"->"준다 잖아"처럼 일부 어미를 실제로 잘못 쪼개서 **자동 적용**함(플래그가 아니라 결과물 자체가 틀림 — 우선순위 높음). (a)~(c)는 비파괴적(리포트 노이즈)이라 우선순위 낮음, (d)는 파괴적이라 우선순위 높음.

## 13. kiwi 미등록 단어 오분석 대응 + .docx 지원 (구현 완료, 2026-07-16)

실제 대사(336줄 SDH 자막)로 검증하며 발견: 맞춤법/외래어/합성어 판단은 이미 사전(국립국어원)이 최종 근거인데, **띄어쓰기 제안(`check_spacing`)만 kiwi 자체 통계 모델(`kiwi.space()`)에 그대로 의존**하고 있어, kiwi가 모르는 단어(고유명사, 새 합성어)를 통계적으로 잘못 쪼갬 — "연실"→"연 실", "탄두리치킨"→"탄두 리 치킨", "한잔하다"→"한잔 하다가".

- **`_protect_unfounded_respacing()`(engine.py)**: kiwi.space()가 원문에 없던 공백을 새로 끼워 넣는 지점마다 (a) 고유명사(NNP) 경계, (b) 사전에 이미 붙여 쓴 형태로 등재됨(용언은 표면형 대신 lemma로 비교해야 함 — "한잔"+"하"로는 사전에 없지만 "한잔"+"하다"는 있음), (c) "/" 바로 뒤나 자막 `[이름/상황]` 브래킷 안(한글 맞춤법이 다루는 대상이 아님) 중 하나면 되돌린다.
- **`register_custom_words(words, tag)`(engine.py, `Kiwi.add_user_word` 활용)**: 번역가가 이 파일에 나오는 고유명사·요리/음료 이름을 미리 알려주면 kiwi가 이후 절대 잘못 안 쪼갬 — 실증 확인(연실/덱스/테일즈/탄두리치킨 전부 해결). CLI(`--names`/`--dishes` 옵션, 파일 경로)와 웹(업로드 화면 "고유명사"/"요리·음료 이름" 입력창, `names`/`dishes` form 필드) 둘 다 반영.
- **`check_spelling()` 플래그 문구 개선**: "사전에 없는 단어"만 말하던 걸, 외국어 음차·고유명사일 수 있다는 점과 사용자의 번역가 교육자료(구글 문서)가 권장하는 실제 검증 방법(국립국어원 용례, 발음기호 사전 Longman/Collins, 한글라이즈 hangulize.org)을 안내하도록 변경 — 그 문서가 "자동교정보다 수동 검증"을 강조하는 원칙을 그대로 반영.
- **`.docx` 지원(`parsers.parse_docx`)**: python-docx로 문단 텍스트를 읽어 다른 일반 텍스트와 동일하게 처리. 서식(볼드체 등) 보존은 이 도구의 범위 밖으로 명확히 둠 — 결과는 항상 순수 텍스트로 돌려준다(`write_plain_text` 재사용). 표 안의 텍스트는 다루지 않음(본문 문단만).
- **프론트엔드**: 더 이상 영상 자막에 국한되지 않는다는 점을 반영해 아이콘(🎬→📝)·제목 크기·설명 문구 정리("일반 텍스트, MS Word(.docx), 자막(.srt) 모두 그대로 업로드 가능"). 한글과컴퓨터(.hwp)는 아직 미구현이라 문구에 넣지 않음 — 넣게 되면 반드시 실제 구현 먼저.

**사용자의 번역가 교육자료(구글 문서) 전체 재검토 결과**: 문서는 29개 섹션(띄어쓰기, 동사/접사 판별, 보조용언, 숫자/단위, 문장부호, 외래어 IPA 세부 표기, 인명/지명/상표명 표기, 번역투/비문 판정 등)으로 이 도구의 v1 범위보다 훨씬 넓다. 재검토 후 판단:
- 대부분은 이미 §2/§5에서 "v1 범위 아님"으로 명시한 영역(외래어 표기법 세부 규칙 엔진은 §5에 "영어 하나만 다루는 것보다 훨씬 큰 작업, v2 이후 백로그"로 이미 기록됨) — 새로 발견된 게 아님.
- **문서 후반부 "번역투|오역|윤문" 섹션(비문 판정, 피동/사역 회피, 능동형 선호, 사물 주어 지양 등)은 아예 성격이 다른 카테고리** — "규정상 맞다/틀리다"가 아니라 번역 스타일 판단이라, 이 도구의 핵심 원칙("확률적 추측이 아닌 권위 있는 규범 근거 기반")으로 검증할 방법이 없다. 자동화 대상이 아님을 명확히 함.
- **북한어/방언 지양**: 이론적으로 사전 조회로 가능하지만, 이 프로젝트의 실제 입력(사투리 대사가 의도적으로 많은 드라마 대본 등)에서는 오히려 정상적인 캐릭터 화법을 전부 오탐하게 될 위험이 커서, 구현하지 않기로 함(백로그로도 낮은 우선순위).
- **작게 반영할 만한 것(아직 안 함)**: "한번(행위)/한 번(수효)", "한잔(행위)/한 잔(수효)" 플래그에 문서의 구체적 구분 기준을 문구로 추가하면 도움될 것.

## 15. 고유명사 등록 통합/자동감지 + 실사용 오탐 3건 수정 (구현 완료, 2026-07-16)

- **고유명사·요리/음료 입력창 통합**: 이름(NNP)과 요리/음료 이름(NNG)을 별도 태그로 등록했었는데, 이 프로젝트에서 이 태그는 "kiwi야 절대 쪼개지 마" 표시로만 쓰이고 다른 로직에 미치는 영향도 둘 다 원하는 방향과 같아 실질적 차이가 없었다. 웹 입력창 하나로 통합(CLI `--dishes` 옵션도 `--names`로 흡수).
- **`detect_recurring_unknown_words()`**: 문서 전체에서 사전에 없이 3번 이상 반복되는 명사를 자동 감지해 등록. 오타는 문서 전체에서 우연히 같은 형태로 반복될 가능성이 낮은 반면, 캐릭터 이름·요리명은 계속 같은 형태로 등장한다는 빈도 차이를 이용함. 번역가가 목록을 안 적어도 대부분의 고유명사 오분석이 해결됨 — 다만 kiwi가 애초에 여러 조각으로 쪼개는 단어(예: "탄두리치킨"->탄두+리+치킨)는 이 방식으로도 못 잡아 여전히 수동 등록이 필요.
- **`_token_containing()`**: kiwi.tokenize()가 이미 하나의 형태소로 본 토큰 **내부**에 kiwi.space()가 공백을 끼워 넣으려는 경우("연실"->"연 실")를 되돌림. tokenize()와 space()가 서로 다른 내부 모델이라 어긋난 것으로, 기존 `_straddling_tokens`(토큰 사이 경계만 확인)로는 못 잡던 사각지대였다.
- **실사용 문장에서 발견한 오탐 3건, 전부 함수 단위(패턴)로 수정**:
  1. `_apply_replacements()`(차별적 표현·확정 오류 치환의 공통 로직)가 kiwi 토큰 경계 확인 없이 문자열 치환만 하다가, "재판장님"(재판장+님) 안에 우연히 들어있는 "장님"을 "시각장애인"으로 잘못 바꿈. 이제 매치 위치가 토큰 시작/끝과 정확히 일치할 때만 치환 — 검증 결과 "간질"(뇌전증) vs "간질이다"(간지럽히다, 무관한 동사)에서도 정확히 구분됨을 확인. `common_errors.py`의 8개 항목 전체에 일괄 적용됨(개별 단어별 예외 처리가 아님).
  2. `correct_aux_verb_spacing()`이 "보조용언은 원칙적으로 띄어쓴다"를 사전 등재 여부 확인 없이 강제 적용해 "여쭤보다"(표준국어대사전 등재)를 "여쭤 보다"로 잘못 쪼갬. `correct_compound_spacing()`이 명사 합성어를 사전으로 확인하는 것과 같은 원칙으로, 본용언-어/아 실제 표면형(축약형 그대로) + 보조용언 사전 기본형을 합친 후보를 `word_exists()`로 확인하도록 수정.
  3. `_mechanical_respace()`가 존대 보조사 "요"(이거요, 빨리요 등 아무 말에나 붙어 존댓말을 만드는 조사)를 kiwi가 가끔 관형사(MM) 등으로 잘못 태깅하는 걸 그대로 믿고 띄어쓰기를 강제("때만요"->"때만 요"). 원문에서 이미 붙어 있던 "요" 한 글자는 kiwi 태그가 무엇이든 그대로 두도록 예외 추가 — 진짜 관형사 "요"(요 녀석)라면 애초에 띄어 쓰여 있었을 것이므로 안전함.
- **웹 UI 레이아웃**: 2단(그리드) 레이아웃을 시도했다가 사용자가 "제목→소개→하는 일→고유명사→업로드" 순서의 단일 컬럼을 요청해 재변경. 본문 폭 1500px로 확대(양옆 여백 축소), "이 교정기가 하는 일" 박스는 4개 항목을 2x2 그리드로 배치해 세로 공간 절약. 체크마크/구분선 스타일은 사용자가 명시적으로 거부해 뺌.

## 17. 온라인가나다 판례 축적 (구현 완료, 2026-07-17)

`correct_aux_verb_spacing()` 패턴 2("할만하다"류) 검증 과정에서, 규칙에는 표준국어대사전만으로는 알 수 없는 예외가 있고 이런 예외를 확실히 잡으려면 온라인가나다(국립국어원 상담 게시판)까지 확인해야 한다는 점이 드러났다. 온라인가나다는 공식 검색/크롤링 API가 없어 매 교정 요청마다 실시간 조회할 수는 없다.

- **`subtitle_corrector/gananda_precedents.py`(신규)**: `common_errors.py`와 같은 방식의 정적 참고 데이터 파일. `expression`/`correct`/`source`(질문·답변 요지)/`verified`(확인 날짜) 필드를 가진 판례 리스트, `check_precedent(expression)`이 조회 함수.
- **`dictionary.py`의 `word_exists()`에 연동**: 표준국어대사전·우리말샘 실시간 조회 결과가 둘 다 없을 때만 마지막으로 이 판례를 확인 — 실시간 사전 데이터가 항상 우선이라는 원칙(§5)은 그대로 유지하고, 판례는 사전에 답이 아예 없는 빈틈만 메운다.
- **채우는 방식**: (1) 실제 사용 중 애매한 사례를 만날 때마다 그 자리에서 검증해 수동으로 추가, (2) 예약 작업 `gananda-precedent-research`(매주 월요일 오전, `~/.claude/scheduled-tasks/`에 등록)가 PRD.md와 `_AUX_NNB_FORMS` 같은 코드를 스스로 읽어 아직 다루지 않은 후보를 골라 WebSearch로 온라인가나다 답변을 찾고, 국립국어원 공식 출처로 명확히 확인된 것만 추가한다. 답이 조건부이거나 출처가 불명확하면 억지로 채우지 않고 건너뛰도록 명시함 — 개수보다 정확성 우선.
- **아직 안 한 것**: 예약 작업이 실제로 몇 차례 돌면서 판례가 얼마나 쌓이는지, 그리고 축적된 판례가 실제 오탐을 얼마나 줄이는지는 검증 전. `PRECEDENTS` 리스트는 현재 비어 있음(0건) — 앞으로 실사용/예약 작업을 거치며 채워진다.

### 하드코딩된 검증 가정을 실시간 조회로 교체 (구현 완료, 2026-07-17)

사용자가 "-ㄹ 만하다"는 사실 표준국어대사전에 있는데(실제로 "만-하다"가 보조 형용사로 등재됨을 API로 확인) AI가 이걸 잘 못 찾는 것 같다고 지적함. 원인 분석 결과, 패턴 2의 (c) 처리("의존명사+하다/싶다 사이는 항상 붙임")가 실제로는 지난 세션에 9개 의존명사(뻔/만/법/듯/성/직/척/체/양)를 한 번 수동으로 확인한 뒤 **그 결과를 코드 주석에 "항상 참"이라고 하드코딩**한 것이었음 — 매 실행마다 다시 확인하는 게 아니었다. `word_exists("할만하다")`(특정 본용언이 붙은 전체 표면형)로 조회하면 0건이 나오는 게 맞지만(이 형태 자체는 개별 표제어가 아니므로), `word_exists("만하다")`(의존명사+하다 단독)로 조회하면 등재된 걸 확인할 수 있는데, 원래 코드는 이 후자 조회를 하지 않았다.

- **수정**: `correct_aux_verb_spacing()` 패턴 2의 (c) 분기를 하드코딩된 가정 대신 `word_exists(cur.form + nxt_citation)`(의존명사+하다/싶다 단독 조합) 실시간 조회로 교체 — §5 "실시간 사전 데이터가 항상 우선" 원칙을 이 지점에도 일관되게 적용. 만약 이 조회마저 실패하면(예: "만싶다"처럼 실제로 존재하지 않는 조합) 붙인다고 단정하지 않고 그대로 둔다(사람 확인 영역).
- **검증**: 할만하다/아는척하다/그럴법하다/날뻔했다/갈만하다 -> 정상 분리, 그럴듯하다/볼만하다(전체가 독립된 표제어로 등재된 경우) -> 그대로 둠, 전체 회귀(`examples/sample.srt`) 결과 동일(자동교정 6건/플래그 4건) — 하드코딩을 없앴지만 기존 동작은 그대로 유지됨을 확인.
- **이 사례가 보여주는 일반 원칙**: `word_exists()` 같은 검증 함수를 쓸 때, 후보 문자열 하나(특히 가장 복잡하게 재구성한 전체 표면형)만 조회하고 실패하면 "사전에 없다"고 단정하면 안 된다 — 사전 표제어 표기 관례(하이픈 결합형, 특정 본용언 없이 추상화된 형태 등)에 따라 더 단순한 형태로 조회하면 존재가 확인되는 경우가 있다. 앞으로 비슷한 검증 로직을 짤 때는 여러 조회 변형을 시도하는 걸 기본으로 한다.

### 자동화된 회귀 테스트 스위트 도입 (구현 완료, 2026-07-17)

위 하드코딩 사례를 계기로, "정확도를 특정 단어 하나가 아니라 전체적으로 어떻게 높일 것인가"를 논의함. 그동안 이 프로젝트의 회귀 검증은 전부 수동이었다 — PRD.md에 "검증 완료"라고 기록해 두고, 다음에 코드를 고칠 때마다 사람(AI)이 그 사례들을 기억해서 다시 실행해보는 방식이었고, 실제로 이 문서에는 테스트 코드가 하나도 없었다. 이는 "검증표 먼저, 코드는 나중"(§5) 원칙을 문서(prose)로만 지키는 것이지, 기계적으로 강제하는 게 아니었다.

- **`tests/test_engine.py`**: 이 문서(§5, §13, §15, §17)에 기록된 실사용 버그 수정 사례를 pytest 테스트로 변환 — 보조용언 패턴 1/2(여쭤보다, 할만하다, 그럴듯하다, 볼만하다 등), 차별적 표현 토큰 경계(재판장님/간질/간질이다/간질간질/장님), 확정 오류(그리고 나서/그리고는), 조사 띄어쓰기(자막 줄바꿈 보존, 존대 "요" 보호), 합성어 붙여쓰기(노천카페/그때/쓴맛), 외래어 표기(초코렛/집) 등 23개 케이스.

- **의의**: 이제 새 규칙을 추가하거나 기존 로직을 고칠 때, 과거에 검증된 모든 사례가 `pytest` 한 번으로 자동 재검증된다 — 사람이 기억해서 수동으로 다시 돌려볼 필요가 없어져, 회귀가 조용히 재발할 위험이 크게 줄었다.

### 실사용 오탐 발견: kiwi가 근거 없이 공백을 지워버리는(단어를 붙여버리는) 경우 미방지 (구현 완료, 2026-07-17)

실제 자막으로 웹 화면을 테스트하던 사용자가 두 가지를 보고함:

1. **"한짓골"(동네 이름, "한지"+"골")**: kiwi가 "한"(관형사)+"짓"(명사)+"골"(명사) 세 조각으로 잘못 쪼개 띄어쓰기 제안이 "한 짓 골"로 엉망이 됨 — §13에 이미 문서화된 "kiwi가 처음부터 여러 조각으로 쪼개는 단어(탄두리치킨 등)는 자동 감지로 못 잡고 수동 등록이 필요하다"는 한계와 같은 종류. 코드 수정 대상 아님, 사용자가 고유명사 목록에 추가하면 해결됨.
2. **"그만하려고 합니다" -> "그만하려고합니다"(실제 버그)**: "-려고 하다"(의도를 나타내는 본동사 구성, 제47항 보조 용언 붙임 허용과 무관하게 항상 띄어 써야 함)를 `kiwi.space()`가 통계적으로 붙여 쓴 형태로 제안했는데, `check_spacing()`이 이 오류를 걸러내지 못했다.

원인: `_protect_unfounded_respacing()`은 "kiwi가 원문에 없던 공백을 근거 없이 새로 끼워 넣는 것"만 막고 있었고, **그 반대 방향("kiwi가 원문에 있던 공백을 근거 없이 지워버리는 것")은 애초에 아예 방지 로직 자체가 없었다** — "특정 단어 하나"가 아니라 이 방향 전체가 빠져 있었던 구조적 빈틈.

- **`_removed_space_points()`/`_protect_unfounded_joining()`(engine.py, 신규)**: `_inserted_space_ranges()`/`_protect_unfounded_respacing()`과 정확히 대칭되는 반대 방향 로직. kiwi가 지운 공백 지점마다, 그 앞뒤 두 토큰이 사전에 실제로 합쳐진 단어로 등재되어 있는지(`word_exists`) 확인하고, 아니면 공백을 되돌린다. `check_spacing()`의 보호 체인 마지막에 추가.
- **구현 중 발견한 함정**: 두 토큰 사이 공백이 정확히 하나였는지 확인할 때, 토큰의 표면형(`form`)을 이어붙여 원문과 비교하려 했더니 실패함 — "합니다"처럼 어간("하")과 다음 형태소("ᆸ니다")가 받침 하나를 공유해 표면형과 실제 글자가 어긋나는 kiwi 특성(제41항 관련 로직에서도 이미 확인된 문제) 때문. 표면형 재구성 비교 대신 토큰 위치(`start`/`len`) 기반으로 그 사이 간격만 직접 확인하도록 수정해 해결.
- **검증**: "그만하려고 합니다"/"나는 가려고 합니다"/"먹으려고 해요" 전부 더 이상 플래그되지 않음(일반화 확인 — 특정 동사 하나가 아니라 "-려고 하다" 패턴 전체에 적용됨). 기존 23개 테스트 전부 통과 + 회귀 테스트 2건 추가(`tests/test_engine.py::TestCheckSpacingJoiningProtection`). 실제 웹 화면(로컬 서버)에서도 최종 확인함.

## 18. 미구현 문법 규칙 사전 검증표 (선행 조사, 2026-07-17 설계)

`gananda_precedents.py`는 `word_exists()`가 실제로 조회하는 정확한 문자열에만 대응되므로, 아직 코드로 구현되지 않은 문법 규칙(예: §5 "제42항 의존명사 잔여 6개": 들/뿐/만/지/차/판)은 그 형식으로 조사할 수 없다 — 코드가 없으니 정확한 쿼리 문자열 자체가 아직 정해지지 않았기 때문이다. 그런데 이런 규칙들은 이미 이 문서(§5)에 "언젠가 구현할 백로그"로 명시되어 있으므로, 나중에 실제로 코드를 짤 때 온라인가나다를 처음부터 다시 뒤지는 이중 작업을 피하기 위해, `gananda-precedent-research` 배치 조사가 이 백로그 항목도 함께 조사해 아래 표에 미리 정답을 쌓아둔다 — "검증표 먼저, 코드는 나중"(§5) 원칙을 코드 작성 이전 단계까지 확장한 것.

**조사 범위는 이 문서(§5)에 이미 이름이 명시된 백로그 항목으로 한정한다.** 온라인가나다 전체를 무작정 훑는 게 아니라, 이 도구가 언젠가 실제로 쓸 것으로 이미 확정된 문법만 대상으로 삼아 범위가 무한정 커지지 않도록 한다.

## 19. 실제 다큐멘터리 자막 감수로 발견한 오탐 무더기 수정 (구현 완료, 2026-07-17)

사용자가 실제 다큐멘터리 한글 자막본(전쟁 다큐멘터리, 군사 용어·편제 번호가 많음)으로 프로그램을 돌려본 결과, 서로 다른 단어처럼 보이지만 근본 원인은 단 세 가지로 좁혀지는 오탐 8건을 한 번에 보고받음.

- **원인 1 — `_protect_unfounded_respacing()`의 사전 등재 확인 로직이 표면형 재구성 방식이라 축약형에서 깨짐**: "잘해야"(잘하다의 활용형, "해"="하"+"어"의 축약)를 "잘 해야"로 잘못 갈라놓음. `word_exists("잘하다")`는 True인데도, 코드가 `text[before.start:after.start+after.len] == before.form + after.form`(표면형을 그대로 이어붙여 비교)로 간격을 확인하다가 "잘"+"하"(재구성) != "잘해"(실제 글자)라 아예 검사를 건너뛰고 있었다 — §17에서 `_protect_unfounded_joining()`에 이미 적용했던 것과 정확히 같은 함정(어간+어미가 받침/음절을 공유하는 kiwi 특성)을 이 더 오래된 함수에는 아직 안 옮겨서 생김. `before.start + before.len == after.start`(위치 기반, 표면형 재구성 없음) 확인으로 교체.
- **원인 2 — 전문 용어·고유명사·편제 번호 복합어를 사전에 없다는 이유만으로 갈라놓음**: "제505공수보병연대원", "폭파병", "제82공수사단장", "제505공수보병연대 2대대", "E중대", "20만"(독일군 20만 명) 전부 같은 원인 — 이런 조합은 애초에 사전에 개별 표제어로 등재될 수 없을 만큼 무한히 다양한데(부대 번호 등), 한글 맞춤법 제49항(고유 명사)·제50항(전문 용어)이 "단어별로 띄어 씀을 원칙으로 하되, 붙여 쓸 수 있다"고 이미 명시적으로 허용하고, 제44항은 "만" 단위 이내 숫자는 붙여 쓴다고 규정한다. 즉 사전 등재 여부와 무관하게 이미 붙여 쓴 것 자체가 허용된 표기이므로, kiwi가 통계적으로 갈라놓자는 제안을 신뢰할 근거가 없다. `_TERM_COMPOUND_TAGS = {NNG, NNP, SN, SL, XPN, NR}` 신설 — 이 태그에 속하는 토큰끼리 원래 붙어 있었다면 사전 확인 없이 무조건 보호한다(NNP 단독 보호를 일반화한 것). 의존명사(NNB)는 제외 — 그건 제42항에 따라 실제로 항상 띄어 써야 하는 별개 규칙이라 포함하면 정당한 오류까지 숨기게 됨.
  - **트레이드오프(의도적)**: 이 일반화는 "완전히 다 붙여 쓴 일반 문장"(내용어 여러 개가 진짜 오타로 안 띄어 쓰인 경우)에 대한 띄어쓰기 제안도 함께 억제한다. 다만 이 도구의 실제 입력(번역가가 이미 정상적으로 띄어 쓴 자막)에서는 이런 경우가 드물고, 반대로 전문용어·고유명사 오탐이 훨씬 흔하다는 실사용 근거(이번 8건)에 따라 노이즈를 줄이는 쪽을 택함 — §5 원칙("잘못된 수정보다 미수정이 낫다")과도 일치.
- **원인 3 — 생산적 파생어(국가명+군/인/어)를 사전 등재 여부만으로 신조어 취급**: "영국군"이 "사전에 없는 단어"로 플래그됨. `word_exists("영국군")`은 False인데 `word_exists("미군")`/`word_exists("독일군")`은 True — 사전이 이 계열 단어를 들쭉날쭉하게만 등재해 둔 것뿐, "국가명+군(軍)"은 규칙적으로 항상 만들 수 있는 정상적인 조합이다. `_is_productive_demonym_compound()`(접미사 군/인/어를 떼고 남은 부분이 `word_exists()`로 확인되면 신조어 취급 안 함) 신설, `check_spelling()`에 연동.
- **검증**: 8건 전부 실제 웹 화면(로컬 서버)에서 재확인, 기존 회귀 테스트 전부 통과 + 신규 테스트 6건 추가.
- **"마치다/맞히다" 목적어 기반 판별 추가 (구현 완료)**: "아르덴 전투 지역에 전개를 마칩니다"에서 이 플래그가 의미 없다는 지적에 대해, 사용자가 "두 단어의 의미를 확실히 파악해 혼동 오류를 없애 달라"고 명확히 요청함. 표준국어대사전 뜻풀이를 확인한 결과 "마치다"(일·과정이 끝나다)와 "맞히다"(정답/자연현상/의료 행위를 맞게 하다)는 목적어(를/을 앞 명사) 종류만 봐도 대부분 의미가 명확히 갈린다 — "전개를"/"수업을"처럼 활동성 명사가 목적어면 "맞히다"일 수가 없고, "정답을"/"문제를"처럼 답·자연현상성 명사가 목적어면 "마치다"일 수가 없다. `_MACHIDA_OBJECTS`/`_MAJIDA_OBJECTS`(사전 뜻풀이·용례 기반 전형적 목적어 목록)와 `_machida_majida_resolved()`를 추가해, 목적어가 한쪽 부류에만 속하고 실제 쓰인 동사도 그 부류와 일치하면 플래그를 생략한다. 목적어가 반대쪽 부류인데 다른 동사가 쓰였거나(예: "정답을 마쳤다", 실제 오류일 가능성 높음) 목적어가 둘 다 아니면 여전히 플래그한다 — 다른 제57항 쌍처럼 자동 교정은 하지 않는다(목적어 목록 기반 추정이지 사전 문자열 일치가 아니므로 확신의 등급이 다름). 이 쌍에만 적용된 예외적 처리이며, 다른 20개 혼동 쌍에 이 방식을 일반화하려면 각 쌍마다 별도로 목적어/문맥 근거를 조사해야 한다.

## 20. 2차 실사용 감수: 확실한 버그 3건 수정 + 판단 필요한 사례 2건 발견 (2026-07-17)

같은 다큐멘터리 자막을 계속 감수하며 추가로 5건을 보고받음. 이번엔 "제대로 수정할 수 없는 오류는 추정하지 말고 반드시 확인해달라"는 사용자의 명확한 새 원칙에 따라, 확신 있는 것만 바로 고치고 나머지는 사용자에게 판단을 구했다.

- **확실한 버그로 확인되어 수정 완료**:
  1. **불규칙 활용 어간 태그 불일치**: kiwi가 불규칙 활용 어간(잇다의 "잇" 등)을 "VV"가 아니라 "VV-I"(하위분류 접미사)로 태깅하는데, `correct_aux_verb_spacing()`/`_aux_verb_pattern_spans()`의 패턴 1이 `prev.tag in ("VV", "VA")`(정확히 일치)로만 확인해 이런 동사가 낀 보조 용언 구성을 통째로 놓치고 있었다. "이어가다"(사전에 없음)가 이 버그 때문에 전용 로직을 타지 못하고 애매한 일반 플래그("문맥에 따라 정답이 다를 수 있음")로 새어나감 — `.startswith("VV")`/`.startswith("VA")`로 교체해 해결. 원래 의도대로 이제 사전 미등재 상태이므로 원칙(띄어쓰기)대로 "이어 가다"로 자동 교정됨.
  2. **"마치다/맞히다" 목적어 목록 보강**: "설치를 마쳤으며"가 목적어 목록에 "설치"가 없어 여전히 플래그되던 것 — "설치"/"공격"/"작전"을 `_MACHIDA_OBJECTS`에 추가.
  3. **"동음이의어" 용어 정정**: 사용자가 "마치다와 맞히다는 맞춤법(표기)이 다르기 때문에 동음이의어라고 볼 수 없다"고 지적 — 정확한 지적이다. "동음이의어"는 엄밀히는 표기·발음이 모두 같은 서로 다른 단어를 가리키는 용어라, 표기가 다른 이 혼동 쌍들에 쓰는 건 용어 오용이었다. 플래그 문구를 "한글 맞춤법 제57항: 소리가 비슷해 혼동하기 쉬운 말"로 정정.
- **사용자에게 확인 후 처리 완료**:
  1. **"안되다"/"안 되다" — 조건·전제 어미 기반 문맥 판별 추가 (구현 완료)**: 사용자가 "문맥에 의거해 의미로 구분하라"고 명확히 요청함. "안되다"(형용사)와 "안 되다"(부정 부사+동사)는 사전 등재만으로 구분이 안 되지만, "-면"/"-거든"/"-아서는·-어서는" 같은 조건·전제 어미 바로 뒤에 오는 "안 되다"는 사실상 예외 없이 금지·불가 구성이라는 확실한 문법적 근거가 있다. `_andoeda_forces_split()`(engine.py) 추가 — 이 신호가 있으면 사전 등재 여부와 무관하게 항상 띄어 쓰도록 `_protect_unfounded_respacing()`/`_protect_unfounded_joining()` 양쪽에 연동. 신호가 없는 나머지(예: "공부가 안된다")는 기존 사전 등재 판단(항상 붙임)을 그대로 따른다 — 확신 없는 나머지 경우까지 추정하지 않음.
     - **추가 검증 + 사각지대 보강 (2026-07-21)**: 사용자가 "그렇게 하시면 안됩니다"(='잘되다'의 반대 맥락, 붙임)와 "그렇게 하시면 안 됩니다"(='되다'의 부정, 띄어씀)라는 본인 구분을 제공. 표준국어대사전·우리말샘을 직접 재조회해 검증한 결과, "안-되다"(붙임) 표제어의 공식 뜻은 ①일·현상·물건이 좋게 이루어지지 않다 ②사람이 훌륭하게 되지 못하다 ③일정한 수준·정도에 이르지 못하다 ④섭섭하거나 가엾어 마음이 언짢다 ⑤근심·병으로 얼굴이 상하다, 이 다섯 개뿐이고 "허용되지 않음/금지"라는 뜻은 없음을 확인 — 사용자의 구분이 사전 근거와 정확히 일치한다. 온라인가나다 답변(qna_seq=324260, 326430, mcfaq_seq=8427 — WebFetch 요약 기반이라 인용 시 재확인 권장)도 "그런 짓을 하면 안 된다"류의 금지 구성은 띄어쓰기로 안내함. 이 검증 과정에서 기존 `_andoeda_forces_split()`이 조건 어미와 "안" 사이에 주어 등 다른 어절이 끼면 신호를 놓치는 사각지대를 실제 테스트로 발견함(예: "그렇게 급하게 진행하시면 결과가 안됩니다"가 플래그되지 않음) — `_conditional_marker_before()`를 추가해 체언·조사·부사만 있는 구간(최대 5토큰)은 건너뛰고 조건 어미를 찾도록 확장, 처음 만나는 어미(EC)가 조건형이 아니면 그 자리에서 탐색을 멈춰 다른 절의 조건 어미까지 잘못 끌어오지 않게 함. 회귀 테스트 2건 추가(`TestAndoedaContextDisambiguation`), 87개 전체 테스트 통과.
     - **미해결 캐뷰앳(백로그)**: 뜻풀이 ③("수준·정도 미달", 예: "나이가 서른도 안된 젊은이")은 온라인가나다의 일부 답변(qna_seq=324260, "만 원이 안 된다"/"600명이 안 된다")이 사전 표제어 뜻풀이와 다르게 띄어쓰기를 안내하고 있어, 국립국어원 스스로도 이 뜻풀이의 실제 적용에 결이 다른 답변이 존재하는 것으로 보인다. 현재 코드는 이 뜻풀이 ③ 케이스를 별도로 다루지 않고(조건 어미 신호가 없으면 기존 사전 등재 판단인 "붙임"을 그대로 따름) 있으므로, 실사용 중 "수량+안되다/안 되다" 오탐이 보고되면 이 캐뷰앳부터 재조사할 것.
  2. **`compound_status()` 우연한 동형이의어 충돌 — 관형사+명사는 허용 목록 방식으로 전환 (구현 완료)**: "두 강"→"두강"(杜康=술의 별칭, 무관), "그 다리"→"그다리"("다리쇠"의 방언, 무관) 등 사전이 "합성어"라고 확인해 줘도 원문 의도와 무관한 우연한 충돌이 실제로 발견됨. 사용자는 "그때/그것/이것처럼 이미 사전에 등재된 조합만 허용하고, 나머지는 한 단어로 취급하지 않는다"는 방향을 확인 — 관형사(그/이/저/두/세 등)가 이끄는 조합만 `_MM_NOUN_COMPOUND_ALLOWLIST`(그때/그날/이날/그곳/이곳/저곳, 검증된 것만)로 제한하고, 그 외는 사전이 "합성어"라고 해도 자동으로 붙이지 않도록 `_compound_candidate_spans()` 수정. 명사+명사(NNG+NNG) 조합은 이번 결정 범위에 포함되지 않아 기존 방식(사전 등재 시 자동 붙임) 그대로 유지 — 즉 "강 다리"→"강다리"(장작 단위/받침나무, 무관) 같은 명사+명사 충돌은 아직 남아 있는 채로 의도적으로 남겨둠.
  3. **"이어가다"/"동음이의어" 용어/목적어 목록 등 나머지 3건**: 확신 있는 버그로 판단해 사용자 확인 없이 바로 수정(§20 상단 참고).

## 21. 3차 실사용 감수: 오탐 10건 조사 및 확실한 버그 8건 수정 (2026-07-17)

같은 다큐멘터리 자막을 계속 감수하며 10건을 추가로 보고받음. 실제 사전/토큰 분석으로 각각 검증한 뒤, 확신 있는 버그는 바로 고치고 나머지는 "판단이 필요한 사례"로 분리해 사용자에게 확인을 구했다.

- **`_opendict_item_is_standard()`의 "미확정" 오판 (구현 완료)**: "쉴더병"이 우리말샘에 있는데도 "사전에 없는 단어"로 플래그됨. 뜻풀이에 "⇒규범 표기는 미확정이다"라는 문구가 있었는데, 기존 코드는 "규범 표기는"이라는 문자열만 보고 "다른 대안 표기가 있다"(초코렛→초콜릿 같은 경우)는 뜻으로 오판했다. "미확정"(아직 정하지 못함, 대안이 없음)과 "다른 표기가 있음"은 다른 뜻이므로 구분하도록 수정.
- **3형태소 복합동사(어간+연결어미+동사)의 사전 확인 실패 (구현 완료)**: "기어다니다"("기다"+"어"+"다니다"), "데려다주다"("데리"+"어다"+"주다")가 잘못 갈라짐. `_protect_unfounded_respacing()`의 사전 등재 확인이 공백 삽입 지점 바로 양옆 토큰 두 개만 보고 후보를 만들다가, 연결어미(EC) 하나만으로는 사전 표제어와 비교가 안 되는 걸 놓치고 있었다("어다"+"주다"="어다주다"라는 없는 말이 후보가 됨). 연결어미 앞의 어간까지 찾아 실제 표면형("데려다")으로 후보를 만들도록 수정 — 어간과 어미가 위치를 살짝 겹치는 경우("데리"+"어다"="데려다")까지 포함.
- **"과"(조사 "~와/과" vs 접두사 "과-[過]") 동형이의어 (구현 완료)**: "좌심실 과증식 상태"에서 "과"가 조사로 오판되어 앞말("좌심실")에 강제로 붙어버림("좌심실과 증식"). "과"+뒷말이 사전에 등재된 단어("과증식")이고 이미 붙어 있으면 접두사로 보고 앞말에 붙이지 않도록 `_mechanical_respace()` 수정.
- **종결어미(EF) 보호 일반화 (구현 완료)**: "말했잖아요"→"말했잖아 요", "그냥요"→"그냥 요", "사람들은요"→"사람들은 요", "모범생 같잖아요"→"모범생 같잖 아요"— 전부 kiwi의 tokenize()/space()가 축약형("잖아요"="지"+"않"+"아요")을 서로 다르게 분석해 생긴 문제. 기존에 "요" 한 글자만 보호하던 걸, 종결어미(EF) 태그 전체로 일반화(_mechanical_respace()의 기존 원칙 — EF는 항상 앞말에 붙음 — 을 check_spacing()의 보호 로직에도 동일 적용).
- **숫자+기호("80%") 보호 (구현 완료)**: 숫자(SN)+기호(SW, %/$/# 등)는 사전 등재 여부와 무관하게 항상 붙여 쓰는 순수 표기 관례라 별도로 보호 추가.
- **`check_spelling()`에 kornorms(외래어 표기 용례) 보강 (구현 완료)**: "뉴암스테르담"이 표준국어대사전/우리말샘엔 없지만 국립국어원 외래어 용례(kornorms)엔 있다는 지적 — 실제로 재현되진 않았으나(kiwi가 NNP로 태깅해 이미 검사 대상에서 제외됨), 유사한 경우를 대비해 `check_spelling()`이 단어 하나를 최종적으로 "사전에 없음"으로 판정하기 전에 kornorms도 확인하도록 방어적으로 보강.
- **"시간이 없다잖나" 해결 (구현 완료, 2026-07-17)**: 당시엔 "잖" 한 글자에 "고 하지 않"까지 압축해서 분석하는 더 깊은 수준의 자기 불일치라 EF 일반화로도 못 잡고 미해결로 남겼던 사례 — §23 "kiwi는 참고일 뿐, 사전 표제어가 기준" 재설계로 해결. 상세 내용은 §23 참고.
- **사용자에게 확인 후 처리 완료**:
  1. **"7년 전 일" -> "전일" 충돌 — 명사+명사 허용 목록이 아니라 문맥(선행 시간 표현) 기반으로 해결 (구현 완료)**: 사용자가 "이건 명사+명사로 설명할 수 없는 문제"라고 정확히 지적함 — "7년 전 일"(7년 전+일, 띄어야 함)과 "전일"(全日/前日, 붙여야 함)의 차이는 품사 조합이 아니라 "전" 앞에 시간 표현이 있느냐로 갈린다. §20에서 도입한 "명사+명사는 그대로 유지"라는 결정과는 별개로, "숫자+시간단위(년/월/일 등) 뒤에 오는 '전'은 뒤 명사와 절대 합성어가 될 수 없다"는 문법적으로 확실한 신호를 찾아 `_DURATION_UNIT_NNB` 기반 예외로 `_compound_candidate_spans()`에 추가.
  2. **"느리다/늘이다/늘리다", "주리다/줄이다"를 CONFUSABLE_PAIRS에서 제외 (구현 완료)**: 사용자가 "우리는 글을 감수하는 것이므로 발음이 비슷한 것은 관계가 없다. 맞춤법이 다르고 의미도 완전히 다르므로 혼동해서는 안 된다"고 명확히 판단 — 듣고 받아쓰는 게 아니라 이미 쓰인 글을 감수하는 이 도구의 실제 용도상, 이 두 쌍은 노이즈만 크고 실제 혼동 위험은 낮다고 판단해 목록에서 제외(`common_errors.py`). 나머지 제57항 쌍(가름/갈음, 반드시/반듯이, 마치다/맞히다 등)은 그대로 유지 — 이 두 쌍만 특정해서 제외해 달라는 요청이었음.
  3. **"호출받다"(명사+받다 생산적 파생) — 판단 보류**: 사용자가 "PDF 참고"라고 답함. §5에 요약만 남아있고("받다의 동작성 판단은... 사람 확인 영역") 원본 자료의 세부 판단 기준은 이 세션에 없어, 문서를 다시 공유받아야 처리 가능 — 다음 세션으로 이월.
- **검증**: 확신 있는 버그 8건 + 사용자 확인 후 처리 2건 전부 수정 확인, 회귀 테스트 11건 추가, 전체 53개 테스트 통과.

## 22. "받다" 동작성 판단 — 교육자료 재확인 후 처리 (2026-07-17)

§5/§21에서 보류해 둔 "명사+받다"(예: 호출받다) 판단을 위해, 사용자가 원본 번역가 교육자료(PDF, "동사/접사 구분법" 단원)를 다시 공유해 확인함.

- **핵심 규칙(자료 내용을 정리한 것)**: "받다"가 구체적인 사물 명사(상, 만점 등 — 손으로 건네받을 수 있는 것) 뒤에 오면 "받다"는 독립된 동사이므로 띄어 쓴다. 반대로 동작성 명사(호출, 사랑, 상처 등 — 어떤 행위·작용을 나타내는 명사) 뒤에 오면 "받다"는 접사로 붙는다. 이건 앞서 이미 다룬 시키다/당하다/하다/되다와 같은 계열의 판단 기준이다.
- **구현**: "동작성 명사+받다" 조합(호출받다, 사랑받다, 상처받다 등)은 그 조합 자체가 개별 표제어로 사전에 등재되어 있지 않은 경우가 많아(교육자료 예시 단어들도 확인해보니 실제로 그랬음) 사전 등재 여부로는 판단할 수 없다. 대신 "명사+하다"가 사전에 등재되어 있는지로 동작성을 판단하는 `_is_action_noun()`을 추가 — 동작성 명사는 거의 항상 "하다"를 붙여 그 행위 자체를 가리키는 동사를 만들 수 있기 때문("호출하다", "사랑하다" 등은 있지만 "상"은 "하다"를 붙일 수 없음). `_protect_unfounded_respacing()`에 연동.
- **검증**: "호출받다"/"사랑받다"/"상처받다"는 더 이상 플래그되지 않고, "상을 받다"/"만점을 받다"(구체적 사물 명사)는 원래대로 영향 없음을 확인. 회귀 테스트 2건 추가, 전체 55개 테스트 통과.
- **"스트레스받다" 추가 규칙 확인 후 해결 (구현 완료)**: 위에서 남겨뒀던 빈칸 조건을, 사용자가 관련 구글 문서를 공유해 확인함 — **동작성이 없어도 피동 의미(누군가·무언가로부터 그 상태를 겪게 됨)가 있으면 접사로 붙인다**는 추가 규칙. "스트레스"는 "스트레스하다"라는 말 자체가 없어 `_is_action_noun()`으로는 못 걸러내지만, 피동 의미가 있는 대표 사례라 `_PASSIVE_ONLY_BATDA_NOUNS`(현재 "스트레스" 1건)라는 별도 목록으로 관리한다 — "피동 의미가 있다"는 사전 API로 기계적으로 확인할 방법이 없어, common_errors.py의 다른 목록들처럼 검증된 사례를 하나씩 추가하는 방식을 택함. 회귀 테스트 1건 추가, 전체 56개 테스트 통과.

  - **정정(같은 날)**: 사용자가 "우리말샘에 '표준 용어는 아이오딘이다'라고 명기되어 있다"고 재지적 — 실제로 표준국어대사전(stdict)엔 이 안내가 없지만 우리말샘(opendict)에는 있었는데, 제 확인 스크립트가 stdict만 보고 opendict은 제대로 안 봐서 놓쳤음. 이 재확인으로 **두 가지 실제 코드 버그**를 발견해 수정함:
    1. **`_opendict_item_is_standard()`가 "규범 표기는"만 확인하고 "표준 용어는"(화학·의학 등 전문 용어에 쓰는 다른 문구)은 놓치고 있었음** — 두 문구 모두 확인하도록 수정.
    2. **`word_exists()`가 표준국어대사전에 표제어가 있으면 곧바로 True를 반환해, 우리말샘에만 추가된 최신 비표준 안내를 아예 확인하지 않고 있었음** — 표준국어대사전 등재 여부와 무관하게 우리말샘의 비표준 안내도 항상 확인하도록 수정.
  - **`correct_nonstandard_terms()`(engine.py, 신규)**: 위 수정으로 확인 가능해진 "우리말샘이 명시한 비표준 표기"를 실시간으로 조회해 자동 교정하는 세 번째 교정 경로 추가(`correct_always_wrong`의 정적 목록, `correct_loanwords`의 kornorms에 이은 것) — "요오드"는 kornorms엔 오히려 정답으로 등재되어 있어 correct_loanwords로는 못 잡고, 우리말샘 자체의 표준화 안내에서만 확인되는 사례라 새 경로가 필요했음. `dictionary.py`의 `standard_term_replacement()`가 "규범 표기는/표준 용어는 'X'이다" 문구에서 정규식으로 X를 추출.
  - **구현 중 발견한 동형이의어 함정(수정 완료)**: "집"(거처, 표준)과 "집"(즙의 옛 표기, 비표준)이 같은 표제어를 공유하는데, 초기 구현이 첫 번째로 검색된 동형이의어만 보고 "집" 전체를 비표준으로 오판해 "그리고 나서 집에 갔다"가 "그러고 나서 즙에 갔다"로 잘못 고쳐지는 사고가 남(§20의 "두강"/"그다리" 동형이의어 충돌과 같은 종류의 함정). `word_exists()`/`standard_term_replacement()` 둘 다, 같은 표제어를 공유하는 동형이의어 중 **하나라도 표준이면 전체를 비표준으로 단정하지 않도록** 수정.
  - **또 다른 실제 버그 발견(수정 완료)**: "요오드가"→"아이오딘가"처럼, 단어를 치환한 뒤 받침 유무가 달라지는데 뒤따르는 조사(이/가, 은/는, 을/를, 과/와)를 안 바꿔 비문이 생기는 문제를 검증 중 발견 — 이미 배포되어 있던 `correct_discriminatory_terms()`에도 같은 버그가 있었음("벙어리가"→"언어장애인가"). `_apply_replacements()`(공통 헬퍼)에 한글 유니코드 받침 판별(`_has_batchim`)과 조사 짝 자동 교체(`_matching_particle_allomorph`) 추가 — "으로/로"는 ㄹ받침 예외가 있어 확신이 없어 제외.
  - **검증**: 요오드/아이오딘/초코렛/집/쉴더병/노천카페 전부 정상 확인, 회귀 테스트 4건 추가, 전체 60개 테스트 통과.
  - **최종 확정(같은 날)**: 커밋 직후 "화학 학술 용어 권고일 뿐, 다큐멘터리 등 일반 문맥에서도 강제로 바꾸는 게 맞느냐"는 자체 의문이 들어, "표준 용어는" 안내를 자동 교정에서 제외하고 참고 플래그로만 남기는 시도를 했었음. 그러나 사용자가 "아이오딘이 표준 맞춤법이다, 요오드가 나오면 앞으로 항상 아이오딘으로 바꿔야 한다"고 명확히 정정 — 화학 용어에 국한된 얘기가 아니라 국립국어원이 갱신한 표준 표기 자체로 봐야 한다는 뜻. 이 지적을 반영해 "표준 용어는"도 다시 "규범 표기는"과 동일하게 자동 교정 대상으로 되돌림(해당 시도 커밋을 revert). **한국어 맞춤법/외래어 표기법은 개정이 잦으므로, 이 판단은 코드에 고정된 값이 아니라 매번 우리말샘을 실시간으로 재조회한 결과라는 점이 핵심** — 국립국어원이 나중에 다시 입장을 바꿔도(예: "나트륨"에 언젠가 같은 안내가 붙거나, 반대로 "요오드" 안내가 사라지는 경우) 코드 수정 없이 자동으로 반영된다.

## 23. 어미 처리 재설계 — "kiwi는 참고일 뿐, 사전 표제어가 기준" (2026-07-17)

사용자가 "이 프로그램이 어미를 잘 처리하지 못하는 문제가 심각하다"고 지적하고, 개선 방향으로 "kiwi는 참고일 뿐, 사전의 표제어와 용례를 기준으로 해야 한다"는 원칙을 명확히 제시함.

- **기존 구조의 근본 문제**: `check_spacing()`은 kiwi.space()의 통계적 제안을 원문과 diff해서 그 차이를 사후에 하나씩 막는 방식이었다(존대 보조사 "요", 종결어미(EF) 태그, 조건 어미 등 지금까지 추가한 예외가 전부 이런 패치였음). 반대로 `_mechanical_respace()`(조사 처리)는 kiwi의 제안을 아예 참고하지 않고 형태소 태그만으로 직접 판단하는 방식이라 지금까지 버그가 거의 없었다. "시간이 없다잖나"("없다고 하지 않나"의 압축)가 계속 못 잡히던 근본 원인은 전자의 한계 — "잖"이라는 한 글자에 여러 형태소가 압축되면서 kiwi의 재구성이 원문과 어긋나기 때문이었다.
- **검증(구현 전)**: 표준국어대사전/우리말샘이 실제로 "-잖다", "-거든", "-ㄹ걸" 같은 압축·구어체 어미를 앞에 하이픈을 붙인 형태로 하나의 표제어로 등재해 두고 있음을 실시간 조회로 확인 — 이 하이픈 표기 관례(어미·조사·접사가 공통으로 씀, 앞말에 붙는다는 뜻)를 근거로 삼을 수 있다는 걸 먼저 검증한 뒤 구현에 들어감(§5 "검증표 먼저, 코드는 나중" 원칙).
- **구현 1 — `registered_ending()`(dictionary.py, 신규)**: candidate가 "-candidate" 형태로 사전에 등재되어 있으면 그 표제어를 돌려준다. `_protect_unfounded_respacing()`에서 기존에 "종결어미(EF)만" 보호하던 걸 "제41항 부착 태그 전체(`_ATTACH_TAGS`: EC 포함) 또는 사전에 등재된 어미·접사 표제어"로 확장 — kiwi의 태그 분류가 아니라 사전 표제어 자체를 최종 근거로 삼는다.
- **구현 2 — `_tokenization_unstable_near()`(engine.py, 신규)**: "없다잖나"는 kiwi가 "하다"를 통째로 생략해 길이 0인 유령 토큰까지 만들어내는 더 깊은 압축이라, 구현 1만으로는 못 잡았다(위치 기반 토큰 검색 자체가 엉뚱한 토큰을 짚어옴). 공백 삽입 지점 주변에 길이 0인 토큰이 있으면 — kiwi 스스로도 이 구간의 형태소 경계를 확신하지 못한다는 신호이므로 — kiwi의 공백 제안을 검증하지 않고 아예 원문을 보존하도록 추가.
- **회귀 발견 및 수정**: 처음엔 "길이 0" 대신 "위치 겹침(overlap)"까지 신호로 썼다가, "됩니다"(되+ㅂ니다)처럼 어간과 어미가 받침 하나를 공유해 위치가 겹치는 지극히 정상적인 활용까지 오탐해 "그러면 안됩니다"의 정당한 "안 됩니다" 분리 제안(§20)이 막혀버리는 회귀가 생김 — 위치 겹침 신호는 제거하고 길이 0(형태소가 완전히 생략됨)만 이례적인 신호로 남김.
- **검증**: "시간이 없다잖나"는 더 이상 잘못 갈라지지 않고, "그러면 안됩니다"→"안 됩니다" 분리 제안은 그대로 정상 동작함을 확인. 회귀 테스트 2건 추가, 전체 62개 테스트 통과.

## 24. 제출 전 점검 — 저장 실패와 교정 실패 분리 (2026-07-17)

과제 제출 루브릭(보안 점검/백엔드 동작 일치/서비스 직관성) 기준으로 프로젝트를 검토하던 중, 실제 백엔드 동작이 의도와 어긋나는 지점을 하나 발견함.

- **문제**: `api.py`의 `/api/correct`가 `store.save_report()`(Supabase 저장)를 예외 처리 없이 호출하고 있었다. 한국어 교정 자체(핵심 로직)는 이미 성공했는데도, 저장 단계에서 실패하면(Supabase 무료 티어 일시정지, 네트워크 지연 등 — DEPLOY.md에 이미 명시된 위험) 예외가 그대로 올라가 교정 결과 전체가 사라지고 사용자는 "서버 오류"만 보게 됨. "의도한 동작"(교정하고 저장한다)과 실제 동작(저장 실패 시 교정도 실패한 것처럼 보임)이 어긋나는 지점.
- **조사**: Supabase의 실제 비활성 정책을 웹 검색으로 확인 — 활동 판단 기준은 실제 DB 쿼리이고, 일단 일시정지되면 요청이 온다고 자동으로 재개되지 않으며 대시보드에서 사람이 직접 "Resume project"를 눌러야 함(90일 지나면 복구 불가). 사용자가 매일 직접 사용하기로 해서 "비활성으로 인한 정지"라는 원인 자체는 사실상 해결됐지만, 채점 시점을 사용자가 통제할 수 없다는 점과 네트워크 지연 등 다른 일시적 실패 원인은 남아 있어, 코드 쪽 완화가 여전히 필요하다고 판단해 진행.
- **구현**: `store.save_report()` 호출을 `try/except (RuntimeError, requests.RequestException)`로 감싸 실패 시 `report_id = None`으로 처리 — 교정 결과(`corrected_srt`/`flags`/`applied_log`)는 그대로 응답한다. 프론트(`static/index.html`)도 `data.id`가 없으면 공유 링크 대신 "저장에 실패해 공유 링크를 만들지 못했다"는 안내만 보여주고, 교정 결과 자체는 정상적으로 렌더링하도록 수정.
- **검증**: `store.save_report`를 `RuntimeError`(환경변수 누락)와 `requests.exceptions.Timeout`(네트워크 지연) 양쪽으로 모의 실패시켜, 두 경우 모두 `id=None`이면서도 `corrected_srt`/`flags`(4건)/`applied_log`(6건)는 정상 반환됨을 직접 확인. 정상 경로(저장 성공)도 회귀 없이 그대로 동작함을 재확인.

## 25. 네이티브 의존성 / 보안 / SQL 종합 검토 (2026-07-17 기록, 2026-07-21 실제 코드 재적용)

## 26. 한글 맞춤법 제57항(소리 혼동 쌍) 검사 전체 제거 (2026-07-17 기록, 2026-07-21 실제 코드 재적용)

사용자가 "마치다/맞히다" 플래그를 다시 보고 재지적: "이 교정기는 말하기와 무관한 텍스트 교정기인데, 소리가 비슷해 헷갈리는 건 발음 차이로 인한 혼동이니 무의미한 교정이다." (§25와 같은 사유로 2026-07-21에야 실제 코드에 재적용됨 — 위 §25 "주의" 참고.)

- **기존 결정과의 관계**: §21에서 이미 "느리다/늘이다/늘리다", "주리다/줄이다" 두 쌍만 "의미가 뚜렷이 달라 혼동될 일이 없다"는 개별 판단으로 제외한 적이 있음. 이번엔 그 논리를 제57항 전체로 확장하자는 요청이라, "반드시/반듯이", "시키다/식히다"처럼 실제 타이핑 과정에서도 흔히 오타로 나는 쌍들은 성격이 다르지 않냐고 먼저 반문함(듣고 헷갈리는 것과 별개로 쓰다가도 흔히 틀리는 쌍이라는 근거).
- **사용자 최종 판단**: "오타, 즉 타이핑은 따로 설명할 필요가 없고, 느리다/늘리다/늘이다 역시 개념상 혼동될 일이 없기 때문에 따로 설명할 필요가 없다" — 소리 혼동이든 단순 타이핑 실수든 이 도구가 별도로 설명(플래그)할 필요가 없다는 결론. 이에 따라 제57항 검사 자체를 전부 제거하기로 확정.
- **구현(2026-07-21 재적용 완료)**: `common_errors.py`의 `CONFUSABLE_PAIRS`(20개 쌍)와 관련 docstring 삭제, `engine.py`의 `check_confusable_words()`와 그 전용 헬퍼(`_CONFUSABLE_LOOKUP`, `_MACHIDA_OBJECTS`/`_MAJIDA_OBJECTS`/`_MACHIDA_MAJIDA_PAIR`/`_machida_majida_resolved()` — "마치다/맞히다" 목적어 기반 예외 처리 전용) 전부 삭제, `correct_entries()`의 플래그 목록에서도 제거, `dictionary.py`의 `usage_examples()` 독스트링에 남아있던 "제57항 동음이의어" 언급도 정리. 죽은 코드를 남기지 않고 완전히 삭제함(관련 원칙: 확실히 안 쓰는 코드는 흔적 없이 지운다).
- **검증**: 관련 테스트(`TestMachidaMajidaObjectDisambiguation` 4건) 삭제, `test_integration.py`의 전체 파이프라인 기대값도 갱신(entry 8 "고개를 반듯이 들어라"가 더 이상 플래그되지 않음 — `flag_line_indices`를 `{4,8,9,10}`에서 `{4,9,10}`으로 수정). 전체 83개 테스트 통과(§20 안되다 사각지대 보강분 포함 87개에서 이번에 지운 4건을 뺀 수).
